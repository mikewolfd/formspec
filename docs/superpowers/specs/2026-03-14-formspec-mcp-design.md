# Formspec MCP Server — Design Spec

**Date:** 2026-03-14
**Status:** Draft
**Branch:** studiofixes

---

## Context

`formspec-studio-core` is a headless authoring runtime for Formspec form bundles. It exposes a command-dispatch protocol, 40+ query methods, and 4-pass diagnostics. Its own ADR (0001) explicitly names LLM/agent workflows as a target consumer and calls commands "a durable protocol that can be generated by LLMs."

This spec defines a Model Context Protocol (MCP) server that wraps `formspec-studio-core` and exposes it to LLM clients (Claude Desktop, Claude Code, and any MCP-compatible client) as a set of behavior-driven tools.

The `pages.*` command namespace (ADR-0039) is already implemented and available. All page management tools route to these commands directly.

---

## Goals

- Let an LLM build, edit, and inspect Formspec forms through natural language
- Hide the three-tier artifact architecture (definition/component/theme) from the tool surface entirely
- Tools describe what the form *does*, not what the schema contains
- Work with Claude Desktop and Claude Code with zero configuration
- Studio-core helpers are reusable by CLI tools, visual builders, and tests — not just MCP

---

## Non-Goals

- Server-driven LLM generation (future: `formspec-mcp-generation` package)
- Output mapping to external systems (CRM, ERP) — mapping document out of scope
- Customer fork/variant model — out of scope
- TypeScript type export as a file artifact — CLI concern, future
- Naming policy enforcement — future project config

---

## Architecture

Two deliverables with a clean boundary:

```
packages/formspec-studio-core/src/helpers.ts     ← new public export (distinct from
                                                    src/handlers/helpers.ts which is
                                                    an internal handler utility)
packages/formspec-mcp/                           ← new package
  src/
    server.ts       — MCP server entry, tool registration
    projects.ts     — in-memory project registry
    tools/
      lifecycle.ts  — create/open/save/list/publish
      structure.ts  — field/content/group/page/place/repeat/update/remove
      flow.ts       — flow/branch/move/rename
      behavior.ts   — show_when/readonly_when/require/calculate/validate
      presentation.ts — layout/style/choices/after_submit
      query.ts      — preview/audit/describe/trace
      dispatch.ts   — raw escape hatch
    index.ts
  package.json
  tsconfig.json
```

**Studio-core** gains `src/helpers.ts` — a new public export at the package root level. This is distinct from the existing internal `src/handlers/helpers.ts` utility. The new file exports pure functions that compose studio-core commands. No new state, no new classes.

**`formspec-mcp`** is a new package. It depends on `formspec-studio-core`. It owns no form semantics — all logic delegates to helpers. The server maintains a `Map<string, Project>` keyed by `project_id`.

**Transport:** stdio. Works with Claude Desktop, Claude Code, and any MCP-compatible client.

**Client-driven generation:** When a client (e.g. Claude) needs to seed a project from a description, it generates the definition JSON itself — using `create`'s `inputSchema` (which references `schemas/definition.json`) as the generation constraint. The server validates on receipt. No second API call, no credentials required.

## Session Model

```typescript
// projects.ts
const registry = new Map<string, Project>();

function createProject(seed?: Partial<ProjectState>, registries?: object[]): string  // returns project_id (uuid)
function getProject(project_id: string): Project     // throws if not found
function closeProject(project_id: string): void
```

**Note on `create` tool seeding:** `ProjectOptions.seed` is typed as `Partial<ProjectState>`, not a bare definition document. When an LLM generates a definition JSON and passes it as `seed`, the MCP tool layer must wrap it: `{ definition: <generated JSON> }` before passing to `createProject`.

Each project is a fully independent `Project` instance. Projects persist in memory for the lifetime of the server process. `save` writes to disk; `open` loads from disk and registers.

## Studio-Core Helpers (`src/helpers.ts`)

A new public export. Each helper maps one behavior-driven concept to one or more studio-core commands, dispatched via `project.batch()`. Helpers intentionally use `batch()` rather than individual `dispatch()` calls — this means middleware is bypassed. This is acceptable because helpers are internal authoring operations, not external commands requiring policy enforcement.

**Note on FEL reference cascading:** `definition.renameItem` already rewrites all FEL expression references internally via `rewriteAllPathReferences()` — the helper only needs to dispatch that single command. For `removeItem`, `definition.deleteItem` removes binds and shapes belonging to the deleted field but does NOT clean up FEL expressions in other fields that reference it. The `removeItem` helper must call `project.fieldDependents(path)` first, then dispatch a batch of `definition.setBind` / `definition.setShapeProperty` to delete (not nullify) the affected expression properties before dispatching `definition.deleteItem`.

```typescript
// Data collection field
// Dispatches: definition.addItem + optional definition.setBind (if behavior props present)
//             + component.setFieldWidget (finds bound node by key, sets node.component type)
export function addField(
  project: Project,
  path: string,
  label: string,
  type: string,              // maps to FormspecItem dataType values
  props?: FieldProps
): CommandResult[]

// Display content — non-data element
// Dispatches: definition.addItem (type: 'display', presentationType matches ContentType)
export function addContent(
  project: Project,
  path: string,
  body: string,
  contentType?: 'heading' | 'instructions' | 'alert' | 'image' | 'divider'
): CommandResult[]

// Group/section container
// Dispatches: definition.addItem (type: 'group') + component.setGroupDisplayMode if displayMode set
export function addGroup(
  project: Project,
  path: string,
  label: string,
  props?: GroupProps
): CommandResult[]

// Conditional visibility
// Dispatches: definition.setBind { relevant: condition }
export function showWhen(project: Project, target: string, condition: string): CommandResult[]

// Readonly condition
// Dispatches: definition.setBind { readonly: condition }
export function readonlyWhen(project: Project, target: string, condition: string): CommandResult[]

// Required rule
// Dispatches: definition.setBind { required: condition ?? 'true()' }
export function require(project: Project, target: string, condition?: string): CommandResult[]

// Calculated value
// Dispatches: definition.setBind { calculate: expression }
export function calculate(project: Project, target: string, expression: string): CommandResult[]

// Cross-field validation
// Dispatches: definition.addShape
export function addValidation(
  project: Project,
  target: string,
  rule: string,
  message: string,
  options?: ValidationOptions
): CommandResult[]

// Branching — show different fields based on an answer
// Dispatches: multiple definition.setBind { relevant } calls, one per path in each branch
export function branch(project: Project, on: string, paths: BranchPath[]): CommandResult[]

// Spatial layout — wraps targets in a layout container
// Dispatches: component.addNode (layout container) + component.moveNode for each target
export function applyLayout(
  project: Project,
  targets: string | string[],
  arrangement: LayoutArrangement
): CommandResult[]

// Presentation style — routes based on target type
// target is a field path → theme.setItemOverride
// target is { type } or { dataType } → theme.addSelector
// target is 'form' → theme.setDefaults
export function applyStyle(
  project: Project,
  target: StyleTarget,
  properties: Record<string, unknown>
): CommandResult[]

// Shared choices
// Dispatches: definition.setOptionSet
export function defineChoices(
  project: Project,
  name: string,
  options: ChoiceOption[]
): CommandResult[]

// Repeat group
// Dispatches: definition.setItemProperty (repeatable, minRepeat, maxRepeat)
//             + component.setGroupRepeatable
export function makeRepeatable(project: Project, target: string, props?: RepeatProps): CommandResult[]

// Add page to flow
// Dispatches: pages.addPage
export function addPage(project: Project, title: string, description?: string): CommandResult[]

// Assign field/group to a page
// Dispatches: pages.assignItem (or pages.addRegion if grid placement specified)
export function placeOnPage(
  project: Project,
  target: string,
  pageId: string,
  options?: PlacementOptions
): CommandResult[]

// Set flow mode
// Dispatches: pages.setMode
export function setFlow(project: Project, mode: 'single' | 'wizard' | 'tabs', props?: FlowProps): CommandResult[]

// Move item in definition + component trees
// Dispatches: definition.moveItem
export function moveItem(project: Project, path: string, targetParentPath?: string, targetIndex?: number): CommandResult[]

// Rename item — cascades FEL refs at helper layer
// Dispatches: definition.renameItem only — FEL rewriting is handled entirely inside
//             the handler via rewriteAllPathReferences(). No additional batch needed.
export function renameItem(project: Project, path: string, newKey: string): CommandResult[]

// Remove item — helper layer cleans up dangling FEL references first
// Dispatches: project.fieldDependents(path) to find referencing expressions, then
//             batch of definition.setBind / definition.setShapeProperty to DELETE
//             (not nullify) affected expression properties + definition.deleteItem
export function removeItem(project: Project, path: string): CommandResult[]

// Update any property of an existing item — fan-out helper
// Routing rules:
//   label, hint, description, dataType, options, optionSet → definition.setItemProperty / definition.setFieldDataType / definition.setFieldOptions
//   required, constraint, calculate, relevant, readonly    → definition.setBind
//   widget                                                 → component.setFieldWidget (via component.setNodeType)
//   style                                                  → theme.setItemOverride
//   page                                                   → pages.assignItem
export function updateItem(project: Project, path: string, changes: ItemChanges): CommandResult[]

// After-submit behavior — confirmation, redirect, notifications
// Dispatches: definition.setDefinitionProperty for form-level properties
export function setAfterSubmit(project: Project, behavior: AfterSubmitBehavior): CommandResult[]
```

## Tool Catalog (32 tools)

### Project lifecycle (5)
| Tool | Description |
|------|-------------|
| `create(seed?, registries?)` | Start a new form project. `seed` accepts a partial definition JSON — the MCP layer wraps it into `{ definition: seed }` before passing to `createProject`. `inputSchema` references `schemas/definition.json` so an LLM client can generate correctly shaped seed data. Returns `project_id`. |
| `open(path)` | Load a form bundle from disk. Returns `project_id`. |
| `save(project_id, path?)` | Write the current bundle to disk. Defaults to the path it was opened from. |
| `list()` | List all active in-memory projects with title, field count, and dirty flag. |
| `publish(project_id, version, summary?)` | Publish a versioned release. The MCP tool layer calls `project.diagnose()` first and returns a structured error with the full `Diagnostics` object if any errors are found. Only calls `project.dispatch({ type: 'project.publish', ... })` when diagnose is clean. |

### Structure (8)
| Tool | Description |
|------|-------------|
| `field(project_id, path, label, type, props?)` | Add a data-collecting element. `type` maps to Formspec `dataType` values: text \| number \| date \| choice \| multichoice \| file \| boolean \| currency \| signature. `props`: placeholder, help, hint, ariaLabel, choices, choices_from, format (email\|phone\|url\|date\|…), widget override, page. |
| `content(project_id, path, body, type?)` | Add a display element. `type`: heading \| instructions \| alert \| image \| divider. |
| `group(project_id, path, label, props?)` | Create a named container for related fields. `props`: collapsible, display (card\|accordion\|table\|stack). |
| `repeat(project_id, target, props?)` | Allow multiple instances of a group. `props`: min, max, add_label, remove_label. |
| `page(project_id, title, description?)` | Add a step to a wizard or tabs layout. Routes to `pages.addPage`. |
| `place(project_id, target, page_id)` | Assign a field or group to a specific page. Routes to `pages.assignItem`. |
| `update(project_id, path, changes)` | Change any property of an existing element. The `updateItem` helper fans out each key to the correct artifact (see helper routing table above). |
| `remove(project_id, path)` | Delete a field, group, or content element. Helper layer queries `fieldDependents` and nullifies referencing expressions before deleting. |

### Flow and form shape (4)
| Tool | Description |
|------|-------------|
| `flow(project_id, mode, props?)` | Set how the form is navigated. `mode`: single \| wizard \| tabs. Maps directly to `pages.setMode` — `single` means no pages (one scrolling form), `wizard` is a step-by-step flow, `tabs` is tab-based. `props`: showProgress, allowBack, allowSkip. |
| `branch(project_id, on, paths)` | Show different fields or groups depending on an answer. `on`: field path or FEL expression. `paths`: array of `{ when, show }`. Dispatches multiple `definition.setBind { relevant }` calls. |
| `move(project_id, path, targetPath?, index?)` | Reorder or reparent a field or group. Routes to `definition.moveItem`. |
| `rename(project_id, path, newKey)` | Rename a field key. Routes to a single `definition.renameItem` command — the handler rewrites all FEL references internally. |

### Behavior (5)
| Tool | Description |
|------|-------------|
| `show_when(project_id, target, condition)` | Make a field, group, or page visible only when the FEL condition is true. Routes to `definition.setBind { relevant }`. |
| `readonly_when(project_id, target, condition)` | Lock a field when condition is true. Routes to `definition.setBind { readonly }`. |
| `require(project_id, target, condition?)` | Mark as required always, or only when the condition is true. Routes to `definition.setBind { required }`. |
| `calculate(project_id, target, expression)` | Derive a field's value from a FEL expression. Routes to `definition.setBind { calculate }`. |
| `validate(project_id, target, rule, message, options?)` | Enforce a data quality rule. `rule`: named format or FEL expression. `options`: timing (live\|submit\|both), blocking (server-side gate flag). Target supports wildcards (`items[*].cost`). Routes to `definition.addShape`. |

### Presentation (4)
| Tool | Description |
|------|-------------|
| `layout(project_id, target, arrangement)` | Arrange fields spatially. `arrangement`: columns (2\|3\|4), card, sidebar, inline. Routes to `component.addNode` + `component.moveNode`. |
| `style(project_id, target, properties)` | Set visual presentation. `target`: field path \| `{ type }` \| `{ dataType }` \| `'form'`. `properties`: density, labelPosition, visualWeight, size — not raw CSS. Routes to `theme.setItemOverride`, `theme.addSelector`, or `theme.setDefaults` based on target type. |
| `choices(project_id, name, options[])` | Define a reusable named option list. Routes to `definition.setOptionSet`. |
| `after_submit(project_id, behavior)` | Configure completion: confirmationMessage, redirectUrl, emailRespondent, notifyTeam. Routes to `definition.setDefinitionProperty`. |

### Understanding (4)
| Tool | Description |
|------|-------------|
| `preview(project_id, scenario?)` | Simulate a respondent's experience. `scenario`: `{ fieldPath: value }`. Returns a structured object — `{ visibleFields: string[], activePage: string, validationState: Record<string, string> }` — which the MCP server formats as readable text before returning to the client. |
| `audit(project_id)` | Runs `project.diagnose()` and additionally checks for: unreachable branches, unconstrained free-text fields, conditional dead ends. Returns structured findings with severity (error \| warning \| info). Exportable as JSON. |
| `describe(project_id, target?)` | Without `target`: returns `project.statistics()` + `project.fieldPaths()` formatted as readable text. With `target`: returns `project.itemAt(target)` + `project.bindFor(target)` + `project.componentFor(target)`. No LLM call required. |
| `trace(project_id, expression_or_field)` | Calls `project.expressionDependencies(expr)` or `project.fieldDependents(path)`. Returns the dependency graph for the target as readable text. |
| `validate_response(project_id, response)` | Feed a response document in, get a `ValidationReport` back. Creates a `FormEngine` from the project's current definition, replays response values via `setValue()`, calls `getValidationReport({ mode: 'submit' })`. Returns structured results with path, severity, message, and constraint kind per field. |

### Escape hatch (1)
| Tool | Description |
|------|-------------|
| `dispatch(project_id, command)` | Send a raw `{ type, payload }` command directly to the Project. Full command protocol access. Logged, undoable. Subject to studio-core's middleware if configured. |

## Error Handling

All tools return structured errors — never generic strings:

```typescript
interface ToolError {
  code: string;          // e.g. 'INVALID_FEL', 'PATH_NOT_FOUND', 'PUBLISH_BLOCKED'
  message: string;       // human-readable
  detail?: {
    path?: string;       // which field was involved
    expression?: string; // which FEL expression failed
    parseError?: { line: number; column: number; message: string };
    // For PUBLISH_BLOCKED: the full Diagnostics object from project.diagnose()
    // Shape: { structural, expressions, extensions, consistency, counts }
    diagnostics?: Diagnostics;
  }
}
```

`publish` calls `project.diagnose()` at the tool layer. If `diagnostics.counts.error > 0`, it returns a `PUBLISH_BLOCKED` error with the full `Diagnostics` object in the detail.

## Future Work

**`formspec-mcp-generation` (separate package)**
Server-driven generation: an optional `GenerationProvider` interface (Anthropic, OpenAI, or any structured-output LLM) that the MCP server can call to bootstrap artifacts. Useful for non-LLM-client contexts (automated pipelines, CLI batch generation). Intentionally excluded from this package to keep `formspec-mcp` dependency-free.

**`validate_response` tool — now in scope**
Moved into the Understanding tool group. Implemented via `FormEngine.getValidationReport()` from `formspec-engine`. No future work needed.

**Naming policy enforcement**
Project-level field key convention config. Validated on `field()` and surfaced in `audit()`.

**`duplicate` tool**
Copy a field or group. Straightforward addition once core tools are stable.
