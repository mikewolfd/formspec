## Implementation Isolation

All new code and tests for this spec are written in **new files, separate from existing studio-core source and tests**. Do not modify existing files until the new implementation is proven correct.

```
packages/formspec-studio-core/src/
  raw-project.ts          ← NEW — rename/copy of existing project.ts + new primitives
  project.ts              ← NEW — Project class wrapping RawProject
  evaluation-helpers.ts   ← NEW

packages/formspec-studio-core/tests/
  project-methods.test.ts ← NEW — all authoring method tests
  evaluation-helpers.test.ts ← NEW
  (existing tests stay untouched until cutover)
```

Once the new `Project` class and all its tests pass, the existing `project.ts` (and its tests) are deleted and replaced. The cutover is a single swap — not a gradual migration.

## Development Pattern

Strict red/green TDD, iterated. For each method:

1. **Red** — write a failing test for the behavior. Run it, confirm it fails for the right reason.
2. **Green** — write the minimum implementation to pass.
3. **Expand** — add tests for edge cases, error codes, warning conditions, FEL validation. Each new test goes red first.
4. **Repeat** — work through the full behavior, not just the happy path, before moving on.

**Tests must assert behavior, not structure.** A good test sets up a `Project`, calls a method, and checks what the form now does — visible fields, valid responses, FEL evaluation results, what `project.state` reflects. A bad test asserts which internal commands were dispatched or checks private state. If a test would only break because you changed the implementation rather than the behavior, it's testing the wrong thing.

Examples:
- ✓ `addField('email', 'Email', 'email')` → `project.fieldPaths()` includes `'email'`; `project.state.definition` has a field with `dataType: 'string'` and a constraint bind
- ✓ `removeItem('address')` on a form where another field's `relevant` references `address` → that bind is gone from state
- ✗ asserting that `this.raw.dispatch` was called with a specific payload (structural, not behavioral)
- ✗ asserting that `batchWithRebuild` was called (mechanism, not outcome)

## Purpose

`formspec-studio-core` exposes a command-dispatch protocol, 40+ query methods, and 4-pass diagnostics. Its ADR-0001 explicitly names LLM/agent workflows as a target consumer and calls commands "a durable protocol that can be generated by LLMs." But the raw command surface speaks the language of the spec (items, binds, shapes, component tree nodes) — not the language of form authors (fields, rules, pages, conditions, validations).

This adds a **behavior-driven interface layer** directly to the `Project` class: methods that compose one or more commands into a single meaningful authoring operation. They live on the object every consumer already holds — no separate import, no passing `project` as a first argument.

One separate module lives in `packages/formspec-studio-core/src/`:

```
src/evaluation-helpers.ts    — previewForm / validateResponse; imports FormEngine (separate module)
```

The evaluation helpers are kept out of `Project` because they pull `formspec-engine` as a dependency. Everything else — all mutation helpers — is added directly to `Project` in `project.ts`.

These methods are **reusable by any consumer**: CLI tools, visual builders, test harnesses, automation scripts, and LLM agents (via the MCP server). The MCP server is one consumer — not the point of the project. Building the right abstraction once means every consumer gets it.

## Architecture

```
packages/formspec-studio-core/src/
  raw-project.ts          ← RawProject class: dispatch, batchWithRebuild, clearRedo,
                            query methods, FEL tools, history, middleware, export
  project.ts              ← Project class: wraps RawProject, exposes it as .raw,
                            adds all authoring methods
  evaluation-helpers.ts   ← previewForm / validateResponse
                            (imports FormEngine; separate module)
  handlers/helpers.ts     ← resolveItemLocation; re-exported from index.ts for explicit access
```

**`RawProject`** (renamed from the current `Project` class) is the command-dispatch core. `dispatch()` accepts a single command or an array of commands; an array is executed atomically with a single undo entry and middleware runs once on the compound operation. It is exported for consumers that need raw access. It is the mechanical layer — it speaks the language of the spec.

**`Project`** wraps `RawProject` via composition and exposes it as `project.raw`. It adds all authoring methods (`addField`, `removeItem`, `branch`, etc.) as instance methods that return `HelperResult`. It speaks the language of form authors. `Project` is the primary export; `createProject()` returns a `Project` instance. Query methods, history (`undo`, `redo`, `canUndo`, `canRedo`), `onChange`, and `export()` are proxied directly on `Project` for convenience — callers do not need to reach through `.raw` for everyday read operations.

```typescript
// project.ts — shape overview
export class Project {
  readonly raw: RawProject     // escape hatch to the raw dispatch layer

  // proxied from raw (read / subscribe / export)
  get state(): Readonly<ProjectState>
  fieldPaths(): string[]
  diagnose(): Diagnostics
  undo(): boolean
  redo(): boolean
  get canUndo(): boolean
  get canRedo(): boolean
  onChange(listener: ChangeListener): () => void
  export(): ProjectBundle
  // ... all other query and FEL methods

  // authoring methods (all return HelperResult)
  addField(path, label, type, props?): HelperResult
  removeItem(path): HelperResult
  // ... etc.
}
export function createProject(options?: ProjectOptions): Project
```

**`evaluation-helpers.ts`** is a separate module that imports `FormEngine` from `formspec-engine`. It contains `previewForm` and `validateResponse` as standalone functions that take a `Project` instance. Keeping these out of `Project` makes the `formspec-engine` dependency explicit and opt-in — consumers that only use mutation methods do not pull in the evaluation stack.

**`handlers/helpers.ts`** exports `resolveItemLocation` for internal handler use. It is also re-exported from `index.ts` so consumers that need it can import it explicitly without reaching into the `handlers/` directory.

**Query formatters** (`audit`, `describe`, `trace`, `search`, `changelog`, `preview`, `validate_response`, `fel_context`, `fel_functions`, `fel_check`) belong in the MCP tool layer. These are thin wrappers that call a single method and format the result for LLM consumption. They have no reuse value outside MCP and are not part of this module.

## Design Decisions

> **Dispatch notation used throughout this spec:**
> `this.raw.dispatch('definition.setBind', { path, properties: {...} })` is shorthand for
> `this.raw.dispatch({ type: 'definition.setBind', payload: { path, properties: {...} } })`.
> `this.raw.dispatch()` takes either a single `AnyCommand` or an `AnyCommand[]`.
> An array argument is executed atomically: single undo entry, middleware runs once.
> All method implementations are inside the `Project` class; `this.raw` is the `RawProject`
> instance. There is no `batch()` — array dispatch replaces it.

**1. `dispatch()` is the single primitive; an array argument makes it atomic.**
`project.raw.dispatch(cmd)` — single command, runs middleware, one undo entry.
`project.raw.dispatch([cmd1, cmd2, ...])` — multiple commands, atomic: runs middleware once on the compound operation, single undo entry, rolls back entirely if any command throws. This replaces `batch()`, which is removed.

Methods call `this.raw.dispatch(cmd)` for single-command operations and `this.raw.dispatch([...])` when multiple commands must be atomic. There is no middleware bypass — dispatch always runs middleware regardless of how many commands are passed.

Consumers that want raw dispatch call `project.raw.dispatch()` directly (or via the MCP `dispatch` tool). The `project.raw` property is the explicit escape hatch; everyday callers use the authoring methods on `Project`.

**Two-phase methods** (`addField`, `addGroup` with `props.display`, `wrapItemsInGroup`, `copyItem(deep: true)`) require the component tree to rebuild between command phases. `definition.addItem` returns `{ rebuildComponentTree: true }`, so any component-tier follow-up command (e.g. `component.setFieldWidget`, `component.setGroupDisplayMode`) that depends on the rebuilt tree cannot be in the same `dispatch([...])`. These use `this.raw.batchWithRebuild()` (see Design Decision #5).

**2. `previewForm` and `validateResponse` are standalone functions in `evaluation-helpers.ts`, not `Project` methods.**
These two operations require instantiating a `FormEngine` with the current definition and replaying state to read reactive signals. Adding them as `Project` methods would pull the full evaluation runtime into every consumer of studio-core. `evaluation-helpers.ts` is a separate export that explicitly opts into the `formspec-engine` dependency. Consumers call `previewForm(project, scenario)` and `validateResponse(project, response)` — the functions read the definition via `project.export().definition`.

**3. Query formatting belongs in the consumer, not here.**
`audit`, `describe`, `trace`, `search`, `changelog`, and the FEL tools are single-method `Project` wrappers that add output formatting. They do not compose commands and have no reuse value outside MCP. Query tool implementations live in the MCP package (`packages/formspec-mcp/src/tools/query.ts` and `fel.ts`), not on `Project`.

**4. Error protocol: helpers pre-validate, the consumer translates.**
Studio-core handlers throw bare `Error` objects with human-readable messages and no structured codes — this is confirmed behavior. `dispatch()` propagates exceptions uncaught (atomicity: the state clone is discarded if a handler throws). The helper layer is therefore responsible for:
- **Pre-validating** inputs that have domain-specific error codes (`PATH_NOT_FOUND`, `PAGE_NOT_FOUND`, `INVALID_TYPE`, etc.) before dispatching — e.g., checking that a path exists via `this.fieldPaths()` before dispatching `definition.addItem` to its parent.
- **Throwing** a lightweight `HelperError` with a `code` string when pre-validation fails.

**Note on `definition.setBind` dispatch payload:** The handler takes `{ path: string, properties: Record<string, unknown> }`, not `{ path, property, value }`. Methods dispatching bind changes must use the properties-map form:
```typescript
this.raw.dispatch({ type: 'definition.setBind', payload: { path, properties: { [property]: null } } })
// NOT: { path, property, value: null }
```
Null values in `properties` trigger deletion semantics (the property is removed from the bind entry).

Consumers (e.g., the MCP tool layer) wrap every method call in try/catch: catches `HelperError` and maps `code` + detail to the consumer's error wire format; catches any other `Error` and maps to `COMMAND_FAILED` with `detail.handlerMessage`.

**5. `batchWithRebuild()` handles two-phase command sequences.**
Several methods dispatch a `phase1` command that triggers a component tree rebuild (`rebuildComponentTree: true`), then need to dispatch `phase2` commands against the rebuilt tree in the same atomic operation. `batchWithRebuild` is a first-class primitive on `RawProject`:
- Executes `phase1` commands on a state clone
- Triggers a synchronous component tree rebuild on the clone
- Executes `phase2` commands on the rebuilt clone
- Produces a single undo entry (pre-call snapshot); runs middleware once

```typescript
// RawProject in packages/formspec-studio-core/src/raw-project.ts:
batchWithRebuild(phase1: AnyCommand[], phase2: AnyCommand[]): CommandResult[]
clearRedo(): void  // { this._redoStack.length = 0; }
```

Methods that use `this.raw.batchWithRebuild()`: `addField`, `addGroup` (with `props.display`), `wrapItemsInGroup`, `copyItem(deep: true)`.

## Type Definitions

```typescript
// Return type for all helpers
// affectedPaths convention:
//   - Field/group/content/group add: affectedPaths[0] = inserted path (dot-path in definition)
//   - rename: affectedPaths[0] = new full path (e.g. 'parent.newKey')
//     Consumers must NOT recompute this — use affectedPaths[0] directly.
//   - addPage: affectedPaths[0] = new page ID (post-hoc read from state.theme.pages last entry)
//   - addValidation: affectedPaths[0] = new shape ID (needed for removeValidation / updateValidation)
//   - addSubmitButton: affectedPaths[0] = component node ID (post-hoc read from state)
//   - wrapInLayoutComponent: affectedPaths[0] = wrapper component node ID
//   - Form-level operations (setFlow, setMetadata, setScreener, makeRepeatable): [] (empty)
// createdId: the primary addressable ID created by this operation, when the operation
//   creates something that must be referenced by name/id (not by path).
//   Populated by: addPage (page ID), addValidation (shape ID), addSubmitButton (node ID),
//                 wrapInLayoutComponent (node ID), addWizardPage (group path).
//   Not populated by addField/addGroup/addContent — their identity is their path (affectedPaths[0]).
// warnings: structured warnings populated by helpers when an operation silently omits related data.
//   Example: copy(deep: false) emits { code: 'BINDS_NOT_COPIED', detail: { count: 2, properties: [...] } }
interface HelperResult {
  summary: string;              // "Added field 'email' (text) to section 'contact'"
  action: {                     // structured record of what the helper did — for programmatic consumers
    helper: string;             // e.g. 'addField', 'branch', 'removeItem'
    params: Record<string, unknown>; // echo of significant input params
  };
  affectedPaths: string[];      // paths created, modified, or deleted
  createdId?: string;           // ID of the primary entity created (page ID, shape ID, node ID).
                                //   Undefined for operations that do not create an addressable entity.
  warnings?: HelperWarning[];   // non-blocking issues the caller should know about
}

// Structured warning — prefer over prose strings for programmatic consumers
interface HelperWarning {
  code: string;           // e.g. 'BINDS_NOT_COPIED', 'SHAPES_NOT_COPIED'
  message: string;        // human-readable explanation
  detail?: object;        // structured detail, e.g. { count: 2, properties: ['required', 'calculate'] }
}

// Thrown by helpers when pre-validation fails.
// Consumers catch HelperError and map to their own error wire format.
// Must be implemented as a class (not just an interface) so instanceof checks work in catch blocks.
class HelperError extends Error {
  constructor(
    public readonly code: string,    // e.g. 'PATH_NOT_FOUND', 'INVALID_TYPE', 'PAGE_NOT_FOUND'
    message: string,
    public readonly detail?: object
  ) { super(message); this.name = 'HelperError'; }
}

// Field properties for addField / addScreenField
interface FieldProps {
  placeholder?: string;
  hint?: string;
  description?: string;
  ariaLabel?: string;
  choices?: ChoiceOption[];          // inline options (for choice/multichoice)
                                     // Mutually exclusive with choicesFrom — throws INVALID_PROPS if both set.
  choicesFrom?: string;              // named option set reference (takes precedence if both set — prefer not setting both)
  // Note: 'email', 'phone', and other semantic text formats are NOT supported via a bind property.
  // Use type aliases 'email' / 'phone' (see Field Type Alias Table) which map to 'string' dataType
  // and add a constraint expression. Example: addField(p, 'email', 'Email', 'email')
  // dispatches definition.setBind { constraint: "matches(email, '.*@.*')" } after addItem.
  widget?: string;                   // override default widget; accepts alias or component name (see Widget Alias Table)
  page?: string;                     // assign to page at creation time (pageId)
  required?: boolean;                // true → dispatches definition.setBind { required: 'true()' }
                                     // For conditional required, chain with require(project, path, condition)
  readonly?: boolean;                // true → dispatches definition.setBind { readonly: 'true()' }
                                     // For conditional readonly, chain with readonlyWhen(project, path, condition)
  initialValue?: unknown;            // → definition.setItemProperty { property: 'initialValue', value }
  insertIndex?: number;              // position within the parent; omit to append at end
  parentPath?: string;               // explicit parent path override (see addField path parameter doc)
}

// Choice option for inline options or defineChoices
// Note: the definition schema supports only value + label on option objects.
interface ChoiceOption {
  value: string;
  label: string;
}

// Group properties
interface GroupProps {
  collapsible?: boolean;
  display?: 'stack' | 'dataTable';
}

// Repeat group configuration
interface RepeatProps {
  min?: number;          // → definition.setItemProperty { property: 'minRepeat', value }
  max?: number;          // → definition.setItemProperty { property: 'maxRepeat', value }
  addLabel?: string;     // → component.setNodeProperty { node: { bind: groupKey }, property: 'addLabel', value }
  removeLabel?: string;  // → component.setNodeProperty { node: { bind: groupKey }, property: 'removeLabel', value }
                         // Note: renderer must read addLabel/removeLabel from the component node,
                         // falling back to default "Add/Remove {item.label}" if not set.
}

// Branch path — one arm of a conditional branch
// `when` is a value compared against the `on` field's current value.
// `mode` controls FEL expression construction:
//   'equals'   (default): string → `on = 'value'`, number → `on = 42`, boolean → `on = true()/false()`
//   'contains': → `selected(on, when)` — use for multiChoice fields where value is an array.
//               selected() handles both array membership (Array.isArray check) and scalar equality.
//               Do NOT use contains() for multiChoice — it is a string function that works on arrays
//               only by accident via duck-typed Array.prototype.includes.
// The helper auto-detects mode: if the `on` field's dataType is 'multiChoice', defaults to 'contains'.
// `show` is one or more field/group paths to make visible when `when` matches.
interface BranchPath {
  when: string | number | boolean;
  show: string | string[];
  mode?: 'equals' | 'contains';  // default: auto-detected from on-field dataType
}

// Layout arrangement for applyLayout
type LayoutArrangement = 'columns-2' | 'columns-3' | 'columns-4' | 'card' | 'sidebar' | 'inline';

// Placement options for placeOnPage
interface PlacementOptions {
  span?: number;    // column span in a grid page layout
}

// Flow configuration
interface FlowProps {
  showProgress?: boolean;
  allowSkip?: boolean;
  // allowBack is NOT a valid Wizard schema property and is not supported
}

// Validation options for addValidation / add_rule tool
interface ValidationOptions {
  timing?: 'continuous' | 'submit' | 'demand';  // default: 'continuous'
  severity?: 'error' | 'warning' | 'info';       // default: 'error'
  // severity: 'error'   → blocks submission
  // severity: 'warning' → advisory, does not block submission
  // severity: 'info'    → informational only
  code?: string;        // machine-readable error code (e.g. 'BUDGET_MISMATCH')
  activeWhen?: string;  // FEL expression: shape is skipped when this evaluates to false
}

// Named external data source (secondary instance)
interface InstanceProps {
  source?: string;
  data?: unknown;
  schema?: object;
  static?: boolean;
  readonly?: boolean;    // default true
  description?: string;
}

// Changes for updateItem — each key routes to a different artifact handler.
// Unknown keys produce INVALID_KEY with detail.validKeys.
// Routing table:
//   label, hint, description, placeholder, ariaLabel, options, choicesFrom,
//   currency, precision, initialValue                → definition.setItemProperty
//   prePopulate                                      → definition.setItemProperty { property: 'prePopulate' }
//                                                      NOTE: prePopulate is NOT a bind property.
//                                                      It routes to setItemProperty, not setBind.
//   dataType                                         → definition.setFieldDataType
//   required, constraint, constraintMessage,
//   calculate, relevant, readonly, default           → definition.setBind (null = delete)
//     required/readonly: boolean true → 'true()', false → null-deletion, string → FEL passthrough
//   repeatable, minRepeat, maxRepeat                 → definition.setItemProperty
//   widget                                           → see widget routing note below
//   style                                            → theme.setItemOverride
//   page                                             → pages.assignItem
interface ItemChanges {
  // → definition.setItemProperty
  label?: string;
  hint?: string;
  description?: string;
  placeholder?: string;
  ariaLabel?: string;
  options?: ChoiceOption[];
  choicesFrom?: string;         // → definition.setItemProperty { property: 'optionSet' }
  currency?: string;            // ISO 4217 code for money fields
  precision?: number;           // decimal places for decimal/money fields
  initialValue?: unknown;
  prePopulate?: unknown;        // → definition.setItemProperty { property: 'prePopulate' }
                                //   NOT a bind property — routes to setItemProperty, not setBind.
                                //   Used by BindsSection, FieldConfigSection, SelectedItemProperties in studio.
  // → definition.setFieldDataType (accepts both type aliases and schema dataType values)
  dataType?: string;
  // → definition.setBind (null value in properties map triggers deletion)
  required?: boolean | string;  // true → 'true()'; false/null → delete; string → FEL expression
                                 // Accepts boolean for consistency with FieldProps.required.
  constraint?: string;
  constraintMessage?: string;   // paired with constraint
  calculate?: string;
  relevant?: string;
  readonly?: boolean | string;  // true → 'true()'; false/null → delete; string → FEL expression
  default?: string;             // FEL expression: value applied when field transitions non-relevant→relevant
                                 // Distinct from initialValue (which is one-shot at load)
  // Note: 'format' is NOT a valid bind property and is not supported in updateItem.
  // Use constraint expressions directly: updateItem(p, path, { constraint: "matches(path, '.*@.*')" })
  // → definition.setItemProperty (repeatable group controls)
  repeatable?: boolean;         // → definition.setItemProperty { property: 'repeatable' }
  minRepeat?: number;           // → definition.setItemProperty { property: 'minRepeat' }
  maxRepeat?: number;           // → definition.setItemProperty { property: 'maxRepeat' }
  // → widget routing (TWO dispatches — must both fire for correct round-trip behavior):
  //   1. component.setFieldWidget { fieldKey: leafKey, widget: resolvedComponent }
  //      If component tree node absent: emit HelperWarning { code: 'COMPONENT_NODE_NOT_FOUND' }, continue.
  //   2. definition.setItemProperty { path, property: 'presentation.widgetHint', value: widgetHintFor(widget) }
  //      This second dispatch is REQUIRED. The widgetHint on the definition item is what survives
  //      form export and reload. Omitting it causes fields to silently lose their widget on round-trip.
  widget?: string;
  // → theme.setItemOverride (uses leaf key, not full path — see applyStyle LIMITATION note)
  style?: Record<string, unknown>;
  // → pages.assignItem
  page?: string;
}
```

## Widget Alias Table

The `widget` property in `FieldProps` and `ItemChanges` accepts both aliases and raw component names. Unknown values produce `INVALID_WIDGET` with `detail.validWidgets`.

| Widget alias | Component name |
|---|---|
| Widget alias | Component name | `presentation.widgetHint` |
|---|---|---|
| `radio` | `RadioGroup` | `'radio'` |
| `checkbox` | `CheckboxGroup` | `'checkbox'` |
| `toggle` | `Toggle` | `'toggle'` |
| `select` | `Select` | `'select'` |
| `slider` | `Slider` | `'slider'` |
| `rating` | `Rating` | `'rating'` |
| `textarea` | `TextInput` | `'textarea'` (requires TWO dispatches: `component.setFieldWidget { widget: 'TextInput' }` then `component.setNodeProperty { node: { bind: key }, property: 'widgetHint', value: 'textarea' }`) |
| `file` | `FileUpload` | `'file'` |
| `signature` | `Signature` | `'signature'` |
| `date` | `DatePicker` | `'date'` |
| `money` | `MoneyInput` | `'money'` |
| `number` | `NumberInput` | `'number'` |
| `text` | `TextInput` | `undefined` (no widgetHint — default TextInput rendering) |

**`widgetHintFor(alias)` rule:** The `presentation.widgetHint` value equals the alias string. Exception: `'text'` has no widgetHint (undefined). When a raw component name is passed directly (e.g. `"RadioGroup"`), convert to lowercase for the widgetHint (e.g. `'radio'`), unless the component has no widgetHint semantics (e.g. raw `"TextInput"` with no hint = plain text input).

Raw component names (e.g. `"RadioGroup"`) are also accepted directly.

## Project Methods (`project.ts` — extends `RawProject`)

```typescript
// raw-project.ts — current Project class, renamed
export class RawProject { ... }  // dispatch, batch, query, history, middleware

// project.ts — new behavior-driven surface
export class Project extends RawProject {
  // all authoring methods below
}
export function createProject(options?: ProjectOptions): Project
```

All methods return `HelperResult`. Methods are mutation-only — pure command composition. Query operations and output formatting are not here.

Handlers throw bare `Error` on failure. Methods pre-validate domain-specific invariants (path existence, type validity, etc.) and throw a `HelperError { code: string, message: string, detail?: object }` before dispatching when pre-validation fails.

**FEL expression validation:** All methods that accept a FEL expression string (`showWhen`, `readonlyWhen`, `require`, `calculate`, `addValidation`, `branch`, `updateItem` bind properties) parse the expression via `this.parseFEL()` before dispatching. Invalid expressions throw `INVALID_FEL` with `detail.expression`, `detail.parseError.message`. Auto-generated FEL expressions from semantic type aliases (`email`, `phone`) are hardcoded and known-valid; they are not re-parsed.

**Note on path parameter:** For methods that take `path` as a dot-path (e.g. `addField`, `addGroup`, `addContent`): the last segment is the item key; preceding segments are the `parentPath`. Root-level items use a bare key (e.g. `'email'`). Items inside a group use `'groupKey.fieldKey'`. The method splits on `.` to derive `parentPath` and `key` before dispatching. `FieldProps.parentPath` overrides this if provided; `FieldProps.insertIndex` controls position within the parent.

**Note on FEL reference cascading:** `definition.renameItem` rewrites all FEL expression references internally via `rewriteAllPathReferences()`. For `removeItem`, see the full algorithm below.

```typescript
// Data collection field
// Resolves type alias → { dataType, defaultWidget } via the Field Type Alias Table
// widget in props resolved via the Widget Alias Table before dispatch
// Pre-validation (before any dispatch):
//   - If props.page is set: verify pageId exists in state.theme.pages → PAGE_NOT_FOUND if not
//   - If props.widget is set: resolve alias → INVALID_WIDGET if unknown
//   - Resolve type alias → INVALID_TYPE if unknown
//
// Dispatch sequence:
//   this.raw.batchWithRebuild(

//     phase2: [

// Remove item — full reference cleanup before delete
//
// Algorithm (collect ALL dependents BEFORE any mutations, then single atomic batch):
//
// Step 1: Collect the full dependent set upfront.
//   Call this.fieldDependents(path) for the target item.
//   If target is a group, recursively call this.fieldDependents() for every descendant.
//   Build the complete cleanup list before dispatching anything — mutations change the state
//   that fieldDependents reads.
//
// Step 2: Classify dependents requiring cleanup on OTHER items' FEL expressions.
//   (The deleteItem handler already removes binds/shapes that TARGET the deleted item.
//    The helper's job is to clean up references TO the deleted item FROM other items.)
//   - Bind properties on other items (calculate, relevant, required, readonly, constraint, default)
//     where the FEL expression references the deleted path:
//     → definition.setBind { path: otherPath, properties: { [property]: null } }
//       (null in properties map triggers deletion semantics)
//   - Shape rules with broken constraint expression → definition.deleteShape
//   - Shape rules with broken activeWhen expression → definition.deleteShape
//     (delete whole shape, not just the guard — a shape with an unresolvable guard is meaningless)
//   - Shape rules with broken context entries → definition.deleteShape
//   - Variables with broken expressions → definition.deleteVariable
//   - Mapping rules: fieldDependents() already scans mapping.rules (sourcePath, expression,
//     condition). For each mapping rule referencing the deleted path:
//     → mapping.deleteRule { index }  (confirm handler name from mapping handler registry)
//     Note: delete the whole rule — a mapping rule whose source or expression is broken
//     has no valid transformation to perform.
//     IMPORTANT: mapping.deleteRule splices by index. If multiple rules match, sort their
//     indices in DESCENDING order and delete highest-index first. This prevents index shifts
//     from causing wrong-rule deletions.
//   - Screener route conditions: fieldDependents() scans screener.routes[].condition;
//     for each route referencing the deleted path → definition.deleteRoute { index }
//     (sort indices descending — deleteRoute splices by index)
//   - Inline item FEL expressions: fieldDependents() scans item.relevant, item.required,
//     item.readonly, item.calculate, item.constraint on each item;
//     for each inline expression referencing the deleted path → definition.setBind { path, properties: { [prop]: null } }
//
// Step 3: Dispatch ALL cleanups AND definition.deleteItem as a single this.raw.dispatch([...]).
//   Cross-ref cleanups (setBind null-deletions, deleteShape, deleteVariable, deleteRoute) and
//   deleteItem are combined into one atomic dispatch. They operate on different entries — no conflict.
//   The deleteItem handler removes binds/shapes that TARGET the deleted path; the cleanup
//   commands remove references TO the deleted path FROM other items. Single dispatch([...]) =
//   single undo entry and full atomicity: if deleteItem throws, the entire operation is discarded.
//
removeItem(path: string): HelperResult

//   key (suffixed for uniqueness), label, type, dataType, hint, description, placeholder,
//   options (inline choices), optionSet reference, presentation widgetHints, extensions,
//   i18n labels, repeatable/minRepeat/maxRepeat, initialValue, currency, precision,
//   and children for groups (recursively, with keys suffixed '_copy').
//
// What IS NOT copied:
//   - definition.binds[] entries (required, calculate, relevant, readonly, constraint,
//     constraintMessage, format) — these live in a separate array keyed by path
//   - definition.shapes[] validation rules — separate array targeting paths
//   - Theme overrides (theme.items[key]) — separate structure

//   { code: 'BINDS_NOT_COPIED', detail: { count: N, properties: ['required', 'calculate', ...] } }

//   FEL rewriting: all expressions that reference originalPath are rewritten to newPath.
//   For groups: child paths are also rewritten (originalPath.child → newPath.child).

//     phase1: [definition.duplicateItem { path }],
//     phase2: [
//       // For each bind entry where bind.path starts with originalPath:

//   Note: shape IDs for cloned shapes are auto-generated by the addShape handler.

// Note: definition.setRouteProperty does NOT have null-deletion semantics —
//   it assigns the value directly. Setting to null stores null.
updateScreenRoute(
  routeIndex: number,
  changes: { condition?: string; target?: string; label?: string }
): HelperResult

// Screener: reorder a routing rule (swap with neighbor)
// Dispatches: definition.reorderRoute { index: routeIndex, direction }
// direction 'up' swaps with the previous route; 'down' swaps with the next route.
reorderScreenRoute(
  routeIndex: number,
  direction: 'up' | 'down'
): HelperResult

// Screener: remove routing rule by zero-based index
// Dispatches: definition.deleteRoute with payload { index: routeIndex }
// Cannot delete the last remaining route — returns ROUTE_MIN_COUNT.
removeScreenRoute(routeIndex: number): HelperResult

// Named external data source
// Dispatches: definition.addInstance
addInstance(name: string, props: InstanceProps): HelperResult

// Update properties on an existing instance — changes bag, consistent with other update* helpers
// Dispatches: definition.setInstance { name, property, value } per key in changes
// Writable properties: source, data, schema, static, readonly, description
updateInstance(name: string, changes: Partial<InstanceProps>): HelperResult

// Rename an instance — rewrites all @instance('name') FEL references
// Dispatches: definition.renameInstance
renameInstance(name: string, newName: string): HelperResult

// Remove an instance
// Does NOT clean up @instance('name') FEL references — leaves them as dangling references.
// Emits HelperWarning { code: 'DANGLING_REFERENCES', detail: { referenceCount: N, paths: [...] } }
//   listing each expression that references the deleted instance.
// Run audit() afterward to confirm cleanup.
// Dispatches: definition.deleteInstance { name }
removeInstance(name: string): HelperResult
```

## Error Codes

`Project` methods throw `HelperError` with the following codes when pre-validation fails. The consumer maps these to its own wire format.

| Code | When |
|------|------|
| `PATH_NOT_FOUND` | Item doesn't exist at the given path. `detail.similarPaths` lists fuzzy-matched existing paths. |
| `INVALID_PATH` | Path syntax is malformed |
| `PAGE_NOT_FOUND` | `page_id` does not exist |
| `VARIABLE_NOT_FOUND` | Variable name does not exist |
| `INSTANCE_NOT_FOUND` | Instance name does not exist. `detail.validInstances` lists existing instance names. |
| `ROUTE_OUT_OF_BOUNDS` | `route_index` is out of bounds |
| `ROUTE_MIN_COUNT` | Attempted to delete the last remaining screener route. `detail.currentRouteCount` and `detail.routes` provided. |
| `INVALID_TYPE` | Unknown `type` alias. `detail.validTypes` lists the alias table. |
| `INVALID_KEY` | Key in `updateItem()` is not in `ItemChanges`. `detail.validKeys` lists all accepted keys. |
| `INVALID_WIDGET` | Unknown widget alias or component name. `detail.validWidgets` lists the Widget Alias Table. |
| `INVALID_FORMAT` | Unknown `format` value. `detail.validFormats` lists valid values. |
| `DUPLICATE_KEY` | An item with that key already exists at the target path |
| `INVALID_FEL` | FEL expression parse error. `detail.expression`, `detail.parseError.message`, `detail.parseError.offset`, `detail.parseError.errorType` provided. |
| `INVALID_TARGET_TYPE` | Operation requires a group item but the target path is a `field` or `display` item. `detail.actualType` provides the actual item type. |

**Warning codes** (non-blocking, returned in `HelperResult.warnings`):

| Code | When |
|------|------|
| `BINDS_NOT_COPIED` | `copyItem(deep: false)` — bind entries omitted. `detail.count`, `detail.properties`. |
| `SHAPES_NOT_COPIED` | `copyItem(deep: false)` — shape rules omitted. `detail.count`. |
| `AMBIGUOUS_ITEM_KEY` | `applyStyle` / `updateItem({ style })` — leaf key shared by multiple items. `detail.leafKey`, `detail.conflictingPaths`. |
| `COMPONENT_NODE_NOT_FOUND` | `updateItem({ widget })` — component tree node absent for field; widgetHint updated on definition but component tree unchanged. |
| `RELEVANT_OVERWRITTEN` | `branch()` — a target path already had a `relevant` bind that was replaced. `detail.path`, `detail.previousExpression`. |
| `DANGLING_REFERENCES` | `removeVariable()` / `removeInstance()` — FEL expressions referencing the deleted entity were left in place. `detail.referenceCount`, `detail.paths`. |

## Future Work

**Implement `definition.renameVariable` handler in studio-core** ← unblocks `renameVariable`
Requires a `definition.renameVariable` handler that rewrites all `$name → $newName` FEL variable references throughout binds, shapes, and expressions — analogous to `definition.renameItem` calling `rewriteAllPathReferences()`. Until this handler exists, `renameVariable` is not implemented.

**`fieldDependents()` must cover screener routes and inline item FEL**
`fieldDependents()` in `RawProject` must scan:
1. **Screener route conditions** — walk `state.definition.screener.routes[].condition` in the dependency scan so `removeItem` can clean up broken routes.
2. **Inline item FEL expressions** — walk `item.relevant`, `item.required`, `item.readonly`, `item.calculate`, `item.constraint` on each item so `removeItem` can clean up broken inline expressions.

**`kind: 'image'` in `addContent`**
No `Image` component exists in the component schema (v1.0 defines 33 built-ins, none named Image). To support this: add `Image` to the component schema and a renderer plugin. Until then, `'image'` is excluded from the `addContent` kind union.

**`audit()` extended checks**
Unreachable branches, unconstrained free-text fields, conditional dead ends. Query-layer concerns (live in the MCP package), but analysis logic may eventually move to studio-core.

**Naming policy enforcement**
Project-level field key convention config. Validated on `addField()` and surfaced in `audit()`.

**`project.allBindsNormalized()` query** ← eliminates 3 incompatible `normalizeBinds` implementations in studio
`project.bindFor(path)` exists for single-path access but there is no bulk accessor. Studio has three separate `normalizeBinds()` implementations with different semantics (StudioApp, CommandPalette, LogicTab) that all work around this gap. Adding `project.allBindsNormalized(): Array<{ path: string; entries: Record<string, string> }>` would consolidate all three.

**`project.defLookup()` query** ← eliminates `buildDefLookup()` called 4 times with `as any` casts
`buildDefLookup(items as any)` is called independently in 4 studio files to build a path→item map. A `project.defLookup(): Map<string, Item>` query would replace all 4 call sites and eliminate the `as any` casts.

**`generateUniqueKey(prefix, project)` shared utility** ← fixes key collision risk in studio
Two files (`StructureTree.tsx`, `EditorCanvas.tsx`) each define a module-scoped `nextItemId` counter starting at 1. Both generate the same keys independently (e.g., `string1`, `group2`). A single `generateUniqueKey(prefix, project)` that scans existing definition keys for the next available suffix — part of the helpers module or studio-core — would fix the collision risk and is the natural implementation of the key generation contract for `addField` and `addGroup`.
