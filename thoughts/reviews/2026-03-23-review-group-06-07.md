# Groups 6–7 Review — Tailwind Adapter & Demo Integration

## Summary

Group 6 delivers a Tailwind CSS render adapter (15 component renderers) with an accompanying `tailwind-formspec-core.css` file for the non-adapter-rendered slots (Card, SubmitButton, ValidationSummary). The CSS decoupling approach is architecturally sound — adapters emit semantic HTML with Tailwind utility classes and inject no global CSS themselves; the `.css` file covers only the structural components that the webcomponent renders itself. Group 7 adds a standalone Tailwind demo app, wires it into the references browser and site hosting pipeline, then removes the standalone app in the final commit, landing in a clean end state where the tailwind adapter is published as a package but no separate demo is shipped.

Most of the work is correct. There are several medium-priority issues worth addressing: inconsistent error-styling patterns across widgets, hard-coded Tailwind color tokens that bypass the CSS variable system in several renderers, a dead `applyErrorStyling` parameter, one misleading test assertion, and a leftover `site/public/tailwind-demo/` directory that is untracked and ungitignored.

## Findings

### Medium: Hard-coded color tokens in Slider, Rating, MoneyInput, and FileUpload bypass the CSS variable system

**File(s):**
- `packages/formspec-adapters/src/tailwind/slider.ts` (lines 15, 44, 56)
- `packages/formspec-adapters/src/tailwind/rating.ts` (line 59, 82)
- `packages/formspec-adapters/src/tailwind/money-input.ts` (lines 15, 42)
- `packages/formspec-adapters/src/tailwind/file-upload.ts` (lines 13, 29, 33, 35, 42, 61-71, 88-89)

**Details:**
The `TW` object in `shared.ts` uses only CSS variable references (`var(--formspec-tw-*)`) for theming, which is the right model. Several widget files bypass this and embed raw Tailwind color tokens directly in class strings:

- `slider.ts` uses `bg-zinc-700`, `accent-teal-500`, `bg-teal-900/40`, `text-teal-200`, `ring-teal-500/20`, `accent-rose-500`
- `rating.ts` uses `text-zinc-600`, `hover:text-teal-400`, `ring-red-500`
- `money-input.ts` uses `border-zinc-700`, `bg-zinc-800`, `text-zinc-400`, `border-zinc-700`, `bg-zinc-900/80`, `text-zinc-100`, `focus:border-teal-500`, `focus:ring-teal-500/15`, `text-zinc-100`
- `file-upload.ts` uses `border-gray-300`, `text-gray-400`, `text-gray-600`, `text-blue-600`, `text-blue-500`, `text-gray-500`, `border-blue-400`, `bg-blue-50`, `border-red-500`

These widgets will not respond to theme token overrides via `--formspec-color-*` variables. Slider's teal styling is especially jarring — it looks like leftover dark-theme aesthetics from an earlier design pass (zinc/teal combination vs. the CSS-variable-driven indigo scheme in `tailwind-formspec-core.css`). FileUpload's static gray palette will look disconnected in any host app that customizes the formspec color tokens.

**Recommendation:**
Add `TW` entries for the affected widget visual elements (track, knob-accent, drop-zone border, drag-active, star-default, star-hover) using CSS variable references. The pattern is already established in `shared.ts`. This is a polish issue, not a functional bug, but it means the adapter is not actually themeable end-to-end.

### Medium: Wizard and Tabs renderers use hard-coded Tailwind color classes inconsistent with the adapter's own CSS variable system

**File(s):**
- `packages/formspec-adapters/src/tailwind/wizard.ts` (lines 42-43, 48, 57, 132, 134, 136)
- `packages/formspec-adapters/src/tailwind/tabs.ts` (lines 21, 58-59, 82-83)

**Details:**
Wizard's step progress indicator uses `bg-blue-600`, `border-gray-300`, `text-gray-500`, `bg-green-500` — direct Tailwind color tokens with no CSS variable indirection. Tabs use `border-gray-200`, `border-blue-500`, `text-blue-600`, `text-gray-500`, `hover:border-gray-300`, `hover:text-gray-700`.

Both of these widgets are structural/interactive components rather than form fields, and neither inherits from `createTailwindFieldDOM`, so they have no natural anchor to the CSS variable system. However, since `tailwind-formspec-core.css` defines `--formspec-tw-accent` for the blue accent color and `--formspec-tw-border` / `--formspec-tw-text` for borders and text, these widgets should use those variables to stay consistent.

The wizard connector line uses inline `style.left`/`style.right` (lines 59-60) which bypasses Tailwind entirely — a minor inconsistency.

**Recommendation:**
At minimum, replace the blue/green step indicator colors with `bg-[var(--formspec-tw-accent)]` and `bg-[var(--formspec-tw-success)]` (add a success variable if needed). The gray separators should use `border-[color:var(--formspec-tw-border)]`. This keeps wizard/tab color tokens participatory in the host theme.

### Medium: `applyErrorStyling` has dead parameter logic and produces dynamically-assembled class names

**File(s):**
- `packages/formspec-adapters/src/tailwind/shared.ts` (lines 143-148)

**Details:**
```ts
export function applyErrorStyling(el: HTMLElement, hasError: boolean, errorColor = 'rose'): void {
    const prefix = errorColor === 'rose' ? 'rose' : errorColor;
    el.classList.toggle('ring-2', hasError);
    el.classList.toggle(`ring-${prefix}-400/60`, hasError);
    el.classList.toggle('rounded-xl', hasError);
}
```

Two issues: (1) The parameter computation `const prefix = errorColor === 'rose' ? 'rose' : errorColor` is identity — it always returns `errorColor`. The entire conditional is dead logic. (2) The dynamic class assembly `ring-${prefix}-400/60` is a known Tailwind JIT concern. In a CDN or runtime context this doesn't matter because the class will be recognized at runtime, but in a build-time Tailwind scan context the class won't be included in the purged output unless the values appear as complete strings somewhere. Since the adapter is designed to work with CDN Tailwind (per the JSDoc in `index.ts`), this is lower risk, but the dead parameter logic is still confusing.

Additionally, `radio-group.ts` bypasses `applyErrorStyling` entirely and directly uses `ring-rose-400/60` (line 82), making the abstraction inconsistent — some callers use the helper, some inline the same class pattern.

**Recommendation:**
Remove the dead `errorColor` parameter. Align `radio-group.ts` to use `applyErrorStyling`. The function should either go through CSS variables (`ring-[color:var(--formspec-tw-danger)]`) or commit to the hardcoded rose color consistently. Pick one.

### Low: Test for prefix/suffix rendering queries a class that doesn't exist

**File(s):**
- `packages/formspec-adapters/tests/tailwind/structural.test.ts` (lines 78-86)

**Details:**
```ts
const prefixes = parent.querySelectorAll('.formspec-tw-input-addon');
// Prefix and suffix should both exist in the group
const texts = Array.from(parent.querySelectorAll('[class*="text-zinc"]')).map(e => e.textContent);
expect(texts).toContain('$');
expect(texts).toContain('.00');
```

The query on line 81 selects elements with class `formspec-tw-input-addon`, which does not exist in the implementation (`text-input.ts` uses `inline-flex items-center rounded-l-xl border border-r-0 border-zinc-700 bg-zinc-800 px-3 text-sm text-zinc-400`). The `prefixes` result is an empty NodeList. The test happens to pass because the actual assertions on lines 83-85 use a different selector (`[class*="text-zinc"]`). The dead query on line 81 is misleading — it implies a class that was either planned but never added, or was removed during the refactor.

**Recommendation:**
Either remove the dead `prefixes` query (it asserts nothing) or replace it with `parent.querySelectorAll('[class*="text-zinc"]')` to match the actual assertions. Also add an assertion that `prefixes.length === 0` is not what you want — this test should assert something structural about the prefix/suffix group, not just that the text content appears somewhere.

### Low: `tailwind-formspec-core.css` validation-summary section uses hard-coded hex colors, not CSS variables

**File(s):**
- `packages/formspec-adapters/src/tailwind/tailwind-formspec-core.css` (lines 101-157)

**Details:**
The `:root` block correctly defines `--formspec-tw-danger` as `var(--formspec-color-error, #dc2626)`. But the validation summary and shape error/warning/info blocks (lines 96-170) use raw hex codes (`#fecaca`, `#fef2f2`, `#7f1d1d`, `#ef4444`, `#450a0a`, `#f59e0b`, `#fffbeb`, `#451a03`, `#0ea5e9`, `#f0f9ff`, `#0c4a6e`) instead of the CSS variables defined in the same file.

This means the ValidationSummary and shape-message colors will not respond to theme overrides through `--formspec-color-error` or similar. The background/border/text tints for the severity states are a fixed red/amber/sky palette hardwired into the CSS, separate from the token system.

This is intentional hardcoding for accessibility-critical severity semantics (the colors do serve a semantic purpose), but it's inconsistent with the rest of the file's design. Worth a comment to explain the intent, and worth evaluating whether these should be additional CSS variables (e.g., `--formspec-tw-danger-bg`, `--formspec-tw-warning-bg`) that default to the current values.

**Recommendation:**
Either add CSS variable definitions for each severity background/border/text color with the current hex values as fallbacks, or add a comment that these severity colors are intentionally fixed for accessibility. The current state looks like an oversight rather than a decision.

### Low: `build-css.mjs` uses `findPackageJSON` from `node:module` — Node 22+ only

**File(s):**
- `packages/formspec-adapters/scripts/build-css.mjs` (line 7)

**Details:**
```js
import { findPackageJSON } from 'node:module';
```

`findPackageJSON` was added in Node.js 22.2.0. The repo runs Node 22.21.1 today, so this works. There is no engine constraint in `packages/formspec-adapters/package.json` to communicate this requirement. If CI or a contributor's machine is on Node 20 LTS, the USWDS CSS build step will silently fail with `SyntaxError: The requested module 'node:module' does not provide an export named 'findPackageJSON'`. Node 20 LTS is still in maintenance and commonly used.

**Recommendation:**
Add `"engines": { "node": ">=22.2.0" }` to `packages/formspec-adapters/package.json`, or replace the `findPackageJSON` call with `createRequire(import.meta.url).resolve('@uswds/uswds/package.json')` which works on all Node versions.

### Informational: CSS decoupling design is correct

**File(s):**
- `packages/formspec-adapters/src/tailwind/index.ts`
- `packages/formspec-adapters/src/tailwind/tailwind-formspec-core.css`

**Details:**
The decision NOT to set `integrationCSS` on the adapter (verified by test on line 421-424) is correct. Injecting global CSS from an adapter would conflict with the host's Tailwind build and create `!important` wars with theme overrides. The `tailwind-formspec-core.css` file correctly uses `@layer components` so host utilities can override the structural plugin styles. The `@exports` entry `"./tailwind-formspec-core.css": "./dist/tailwind-formspec-core.css"` is correctly scoped so consumers must opt in.

The CSS variable bridge design (`--formspec-tw-*` mapping to `--formspec-color-*` with hex fallbacks) is the right pattern — it decouples Tailwind's utility syntax from the formspec theme token layer. The `:root` block covers all the shared surface/text/border/accent/danger tokens used by the utility classes in `shared.ts`.

### Informational: Semantic HTML structure is correct across all 15 components

The widget implementations use the appropriate HTML elements:
- `TextInput`/`NumberInput`/`DatePicker`: `<input>` with `type`, `id`, `name`, `aria-describedby` set
- `RadioGroup`/`CheckboxGroup`: `<fieldset>` + `<legend>` pattern
- `Select`: `<select>` with placeholder `<option>` and clearable semantics
- `Toggle`: `<input role="switch">` with `aria-describedby`
- `Rating`: `role="slider"` with `aria-valuemin/max/now/text` and keyboard handlers (Arrow, Home, End)
- `Tabs`: `role="tablist"`, `role="tab"`, `role="tabpanel"` with `aria-selected`, `aria-controls`, `aria-labelledby`
- `Wizard`: `<nav aria-label="progress">` + `<ol>` for step indicator, `role="region"` panels
- `FileUpload`: `role="button"` drop zone with keyboard (Enter/Space) activation, `<input type="file" class="sr-only">`
- `Signature`: `role="img"` canvas with keyboard label

All adapters correctly call `behavior.bind()` with the required `FieldRefs` and register dispose via `actx.onDispose()`. Cascade obligations (`applyCascadeClasses`, `applyCascadeAccessibility`) are applied on the root element for all non-container widgets.

### Informational: SPA base URL routing is correct

The `ASSET_BASE` pattern (`import.meta.env.BASE_URL.replace(/\/$/, '')`) is the standard Vite idiom. All example `dir` and `registry` paths are correctly prefixed. Vite dev middleware strips the base prefix before mapping to repo-root paths. The Astro middleware uses `Response.redirect` rather than Astro's static redirect config, correctly avoiding the overwrite of `public/references/index.html`. The `/refrences` typo alias in middleware is intentional (preserves the historical folder name).

## Verdict

**Approve with follow-up work recommended.** The adapter architecture is sound, the semantic HTML is correct, the CSS decoupling design is well-executed, and the demo lifecycle cleanup is complete. The main issues to address are:

1. **Color token consistency** (Medium): Slider, Rating, MoneyInput, and FileUpload embed raw Tailwind color tokens that bypass the CSS variable theme system. These should use CSS variable references to make the adapter fully themeable.
2. **Wizard and Tabs color isolation** (Medium): Step indicator and tab bar colors use hardcoded blue/gray/green instead of the `--formspec-tw-accent` and `--formspec-tw-border` variables established in the CSS file.
3. **`applyErrorStyling` dead parameter** (Low): Remove the identity-expression `errorColor` parameter logic and align `radio-group.ts` to use the helper.
4. **Missing Tailwind demo in references** (Low): The tailwind adapter is undemonstrated in the live app. Either complete the wiring or track the gap.
5. **Empty `site/public/tailwind-demo/` artifact** (Low): Delete the empty directory that survived the demo removal.

None of these block merge. The CSS variable inconsistency is the highest-priority item for follow-up.
