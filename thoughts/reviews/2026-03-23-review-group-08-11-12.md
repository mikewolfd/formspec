### [Medium]: ValidationSummary mode "submit" gate — test gap for the negative case

**File(s):** `packages/formspec-webcomponent/src/components/display.ts:376-382`, `tests/e2e/browser/clinical-intake.spec.ts:786`

**Details:** The fix is architecturally correct. The root cause was that `touchedVersion` is incremented on every field `focusout` event (see `packages/formspec-webcomponent/src/behaviors/shared.ts:135`), which caused the ValidationSummary with `source: "live"` and `mode: "submit"` to appear after any field blur — before any submit attempt. The fix correctly gates on `latestSubmitDetailSignal !== null` for `mode: "submit"` and retains the original gate (`submitOccurred || wizardNavigated`) for `mode: "continuous"`.

The existing E2E test at clinical-intake.spec.ts:786 confirms the "shows after submit" path. What's missing is the negative-case assertion: "should NOT show after field blur when mode is submit". The clinical-intake fixture uses `mode: "submit"` (examples/clinical-intake/intake.component.json:194), so the fix is exercised in the right fixture, but nothing asserts the pre-fix bug is gone — only that the post-fix success case works.

**Recommendation:** Add an E2E assertion to the clinical-intake suite: navigate to Summary page, blur a required field, confirm `.formspec-validation-summary--visible` is absent before submit is clicked. This takes two lines and pins the regression gate.

### [Medium]: Mapping spec commit message overstates schema change — "array innerRules without transform" not reflected in schema

**File(s):** `schemas/mapping.schema.json:221`, `schemas/mapping.schema.json:531`, `specs/mapping/mapping-spec.md:773`

**Details:** The commit message says "array innerRules without transform" but the JSON Schema still has `"required": ["transform"]` on both `FieldRule` (line 221) and `InnerFieldRule` (line 531) in the schema. The actual schema change was limited to description text on `transform`, `expression`, and the `valueMap` shorthand. The spec prose added a SHOULD paragraph (spec line 773) that says runtimes MAY accept a missing parent `transform` when `innerRules` is present, treating it as `"preserve"` — but the schema validator will still reject that document as invalid.

This creates a spec–schema divergence: the spec says "runtimes MAY accept a missing `transform`" but the schema enforces `transform` as required. A document following the spec's lenient guidance will fail schema validation. The spec prose says "authors SHOULD not rely on that shorthand in documents that must validate everywhere" — but that caveat is easy to miss.

This was presumably intended to address runtime leniency but the schema was not actually relaxed. If the intent was to relax it, the fix belongs in the schema: either remove `"transform"` from `required` on `InnerFieldRule` and add an `anyOf` that implies `"preserve"` when `innerRules` is non-empty, or document the current state as intentionally strict (schema-valid = transform required, runtime leniency is engine-only). The current state of "spec permits, schema denies" is a silent trap.

**Recommendation:** Decide intent, then align:
- If the schema should stay strict: remove the "MAY accept a missing parent `transform`" sentence from the spec or downgrade it to "engines are permitted to be lenient for developer convenience but MUST NOT export such documents."
- If the schema should be relaxed: update `InnerFieldRule.required` to not include `"transform"` and add a conditional default. Regenerate types.

### [Low]: Project import stale-page logic is all-or-nothing — may over-clear in partial-overlap scenarios

**File(s):** `packages/formspec-core/src/handlers/project.ts:48-56`

**Details:** The new logic reads: if any region key across any page fails to match a key in the new definition, ALL theme pages are dropped. The old behavior dropped pages only when NO region matched. Both policies have edge cases. The new policy is more aggressive: a single stale key on page 2 of a 5-page layout causes pages 1-5 to all be discarded, including pages where every region is valid.

The comment acknowledges the old policy's false positive ("accidental key match e.g. shared field name 'name' must not preserve an entire stale page graph") but the new policy creates a false negative in the opposite direction: a mostly-valid layout with one stale region is wiped out entirely. The diagnostics system in `resolvePageStructure` already handles broken regions gracefully with `UNKNOWN_REGION_KEY` warnings — they show as `status: "broken"` in the PageItemView. That's the designed degradation path.

A more conservative fix would be: drop pages that are entirely composed of invalid regions; preserve pages with at least one valid region. Or: preserve all pages (the diagnostics handle it). The current behavior may surprise users who import a new definition that shares most fields with the old one but renamed one field — their page layout evaporates.

No tests cover this new behavior path.

**Recommendation:** Add tests for both the "all valid after import" case and the "one stale key drops everything" case so the policy is pinned explicitly. Consider whether the correct policy is "drop pages where EVERY region is invalid" rather than "drop ALL pages when ANY region is invalid." Add a comment naming the chosen policy explicitly.

### [Low]: ValidationSummary mode "continuous" naming is confusing — `touchedVersion > 0` means wizard navigation, not continuous validation

**File(s):** `packages/formspec-webcomponent/src/components/display.ts:371-378`

**Details:** The comment says "mode 'continuous': require submit or wizard navigation." But `touchedVersion > 0` also fires on regular field blur (not just wizard navigation). The variable is named `wizardNavigated` in the new code but really means "any touch event occurred." This is pre-existing confusion, not introduced by this commit, and is functionally correct — the intent is "show after any interaction" for continuous mode. But the naming in the fix (`wizardNavigated`) adds a misleading alias on top of an already ambiguous signal name.

**Recommendation:** Rename the local variable from `wizardNavigated` to `anyInteractionOccurred` or `touchOccurred` to match what `touchedVersion` actually tracks.

### [Low]: Firebase `define:vars` serializes config into built HTML — expected for public keys, document for team clarity

**File(s):** `site/src/components/FirebaseAnalytics.astro:10`, `site/src/lib/firebase-public-config.ts`

**Details:** `define:vars={{ firebaseConfig }}` in Astro serializes the config object as a JSON literal in the compiled HTML. Since the site is statically generated (no `output: 'server'` in `astro.config.mjs`) and all keys are `PUBLIC_FIREBASE_*` (meaning they are intentionally client-visible by Astro convention), this is the correct and expected pattern. Firebase web SDK config values are designed to be public — they identify the Firebase project but do not grant privileged access. The `apiKey` is an identifier, not a secret.

The `getFirebasePublicConfig` null-guard is solid: all seven required keys must be non-empty strings, or the component renders nothing. No Firebase SDK is loaded in builds without config. The `.gitignore` excludes `.env` and `.env.local`, and the `.env.example` contains no actual values.

No security concern. Documenting this finding for completeness given the review task asked to check for exposed keys.

### [Low]: Page-view resolution fix is correct — delegate to `resolvePageStructure` for placement map

**File(s):** `packages/formspec-core/src/queries/page-view-resolution.ts:149`, `packages/formspec-core/src/page-resolution.ts:82-96`

**Details:** The old code rebuilt `itemPageMap` from `page.items` directly, which missed the bidirectional propagation logic in `resolvePageStructure` (parent-to-child and child-to-parent group inheritance, lines 99-120 of `page-resolution.ts`). This caused nested fields to appear as "unassigned" in the palette even when their parent group was placed. The fix correctly delegates: `const itemPageMap = { ...resolved.itemPageMap }`.

The updated test now asserts that unknown/broken region keys are NOT in `itemPageMap`, which matches the behavior of `resolvePageStructure` (only `region.exists === true` keys are added at line 88). This is a semantically correct change — broken regions never had a valid page ID to map to.
