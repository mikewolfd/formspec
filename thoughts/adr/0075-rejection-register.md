# ADR 0075: WOS Rejection Register

**Status:** Proposed
**Date:** 2026-04-24
**Scope:** WOS — spec authoring and contributor governance
**Related:** [WOS README §"Why not extend BPMN?"](../../wos-spec/README.md#why-not-extend-bpmn); [WOS README §"Why JSON-native?"](../../wos-spec/README.md#why-json-native); [Standards Absorption Gap Analysis §"Do Not Integrate"](../../wos-spec/thoughts/2026-04-24-standards-absorption-gap-analysis.md#do-not-integrate); [FlowSpec Counter-Proposal Disposition §"Red flags"](../../wos-spec/counter-proposal-disposition.md#red-flags--stack-invariant-conflicts-not-absorbable-in-any-form); [FlowSpec Counter-Proposal Disposition §"Rejects requiring explicit spec documentation"](../../wos-spec/counter-proposal-disposition.md#rejects-requiring-explicit-spec-documentation); [platform decision register](../specs/2026-04-22-platform-decisioning-forks-and-options.md); [ADR 0073 (case initiation and intake handoff)](./0073-stack-case-initiation-and-intake-handoff.md)

## Context

WOS has accumulated an explicit list of ideas it rejects. The rationale is not a style preference; each rejection protects a load-bearing invariant — one signing pipeline, intake-handoff boundary, statechart-as-authoritative-lifecycle, FEL as the only expression language, Rust as the spec authority, no event-log-as-ledger.

Those rejections sit in three documents today:

- **WOS README** — `why-not-extend-bpmn` and `why-json-native` sections frame BPMN / XML-native alternatives.
- **Standards Absorption Gap Analysis** — `Do Not Integrate` section names FEEL, DMN-engine semantics, BPMN gateways, CMMN planning tables, SCXML executable/datamodel semantics, SCXML as wire format.
- **FlowSpec Counter-Proposal Disposition** — `Red flags` (stack-invariant conflicts) and `Rejects requiring explicit spec documentation` (onComplete, flat edge-graph, X31 explicit-edge, event-log-as-ledger), plus REJECT-bucket rows embedded in the disposition tables (node-local AI provider config, X33 heuristic, two-tier conformance, start-node `requiredPayload`, §3.3 `tools`).

New contributors read one, miss the others, re-raise rejected ideas. Each re-raise costs a review cycle, a thread, and a spec-authoring pushback. At WOS's current surface area (four streams, 18 specs, 18 schemas) the cost is load-bearing.

The rejections themselves are not the problem. The scatter is. A reader cannot derive "would X be rejected?" without reading three documents end-to-end.

## Decision

Consolidate every rejection into one canonical register at this ADR. The register states the rejection, the rationale, the invariant it protects, and the deepest prior analysis. README, gap analysis, disposition, and any future doc that wants to express "we considered this and rejected it" cites this ADR rather than duplicating the content.

### Invariants (the `why`)

The register below is not a list of opinions. Every row protects at least one of these invariants. A reader asking "would X be rejected?" can answer by checking X against this list.

| # | Invariant | What it means |
|---|-----------|---------------|
| I-1 | **Statechart is deliberate.** | Lifecycle is an SCXML/Harel-derived JSON statechart with nested states, parallel regions, history states, guard-composable transitions. Not a flat node/edge graph. Authoring sugar that compiles to statechart transitions is permitted; replacement topologies are not. |
| I-2 | **FEL is the only expression language.** | FEL drives guards, deontic constraints, equity expressions, condition events. No second expression language, no embedded decision engines. Decision-service evidence is captured as provenance (`DecisionServiceEvidence`), not by executing foreign grammars. |
| I-3 | **Spec-authoritative in Rust.** | `wos-core` owns spec semantics; `wos-runtime` is the in-memory conformance oracle. Spec behavior does not live in scripting layers, JSON processors, or adapter glue. |
| I-4 | **Intake-handoff boundary.** | Formspec owns intake session truth; WOS owns governed case identity and `case.created`; the seam is the named `IntakeHandoff` artifact (ADR 0073). Neither layer emits the other's boundary event. |
| I-5 | **One signing pipeline.** | All signature semantics route through the Signature Profile's `SignatureAffirmation` provenance and Trellis custody / export. Product shortcuts (click-to-sign, magic links) are workflow-lite paths over the same pipeline, not a second meaning of "signed." |
| I-6 | **No event log as audit ledger.** | Durable execution checkpoints state for crash recovery; Trellis anchors provenance for audit. WOS emits provenance records; Trellis appends to custody. A workflow engine's internal event log is not the stack's ledger. |
| I-7 | **AI non-determination is enforced, not guidance.** | L2 AI Integration §4 defines deontic constraints with four types (`permission` / `prohibition` / `obligation` / `right`) per OASIS LegalRuleML, with fixed enforcement ordering (§4.6), composition rules (§4.7), and impact-capped autonomy (§5.3). Rights-impacting workflows cannot downgrade this to best-effort guidance. |
| I-8 | **Four-stream rights-proportional conformance.** | Compliance claims reference a pair of stream versions (e.g. `wos-kernel@1.0 + wos-ai@0.5`). Two-tier core/extended models collapse the governance-vs-AI distinction that the four-stream model exists to preserve. |
| I-9 | **Governance attaches to transitions.** | Semantic `tags` on transitions (Kernel §4.12) are the attachment surface for review protocols, due-process gates, decision trace, and deontic enforcement. Node-centric or edge-annotation-only routing loses this attachment point. |
| I-10 | **Adapter concerns stay in adapters.** | Model identity, tool grants, and capability permissions are governed at L2 via `AgentDeclaration` (§3.1) and the `Capability` sub-shape (§3.3 Capability Declaration). Provider selection (chat-completion endpoint, temperature, prompt template) is adapter territory (extension E5), not workflow-node config. |
| I-11 | **Deterministic evaluation.** | Kernel §4.2 MUST: "Two conformant Kernel Processors given the same Kernel Document and the same sequence of events MUST produce the same sequence of state transitions." Load-bearing for rejecting embedded scripting (row 6), embedded DMN execution (row 4), and unconstrained CMMN runtime mutation (row 5). |
| I-12 | **Tiered provenance on every transition.** | Kernel §8 requires every transition to emit a Facts-tier provenance record; higher layers add `reasoning` / `counterfactual` / `narrative` tiers via the `provenanceLayer` seam (§10.3). Load-bearing for rejecting event-log-as-ledger (row 10) and embedded decision execution without trace (row 4). |

## Consequences

**Positive.**

- One canonical register. New contributors learn "what WOS rejects" from one document, not three.
- Invariant names replace ad-hoc rationales. "X violates I-1 (statechart is deliberate)" is shorter and more auditable than re-explaining why statechart topology matters.
- Future docs reject ideas by citation, not by duplication. README, gap analysis, and disposition update to cite this ADR instead of carrying their own rejection lists.
- A reader asking "would X be rejected?" can derive the answer from the invariant list without re-reading every row.

**Negative.**

- README, gap analysis, and disposition SHOULD cite this ADR rather than duplicate the rejection content. Each of those documents needs a single edit pass to replace its rejection list with a pointer. Until those edits land, the register is canonical but not yet the only copy.
- New rejections require an ADR amendment (or a superseding ADR under [ADR 0066 (amendment and supersession)](./0066-stack-amendment-and-supersession.md)). Scattering was cheaper to author; consolidation pays off at read time, not write time.

**Neutral.**

- The register is not a "things WOS will never do" list. "Reject as core" does not preclude authoring sugar that compiles to the accepted model (see row 2, boundary events). The distinction lives in the rationale column.
- Some rows (e.g. 13 — AI provider config) are partial rejections: the topology-level knob is rejected, the underlying governance concern (model identity) is absorbed elsewhere. The rationale column makes the distinction explicit; the register is not a substitute for reading it.

- **Land this ADR** with status Proposed. Owner: Michael Deeb.
- **Edit downstream docs** to cite this ADR rather than duplicate rejection content:
  - `wos-spec/README.md` — `why-not-extend-bpmn` and `why-json-native` sections link to this ADR's register for the itemized list; retain framing prose that explains the JSON-native / governance-semantics thesis.
  - `wos-spec/thoughts/2026-04-24-standards-absorption-gap-analysis.md` — `Do Not Integrate` section shortens to a pointer at this ADR, with the absorption-plan sections (gaps worth integrating, refactor target, priority) retained.
  - `wos-spec/counter-proposal-disposition.md` — `Red flags` and `Rejects requiring explicit spec documentation` sections shorten to pointers at this ADR's rows (cite row numbers); REJECT-bucket rows in the disposition tables retain their one-line rationale plus a row citation.
- **Conformance.** No schema or lint change. This ADR governs contributor discourse, not processor behavior. Future CI may add a lint rule that any new spec doc citing "we rejected" content without a pointer to this ADR fails the promotion gate, once the register is accepted.

## Open questions

1. **Boundary-event companion surface (row 2).** Authoring sugar that compiles to statechart transitions is permitted; the companion document does not yet exist. Open question is whether it lands as a Runtime Companion section or a separate authoring companion. Out of scope for this ADR; tracked in the gap analysis refactor target.
2. **Governed discretionary work (row 5).** The rejection stands for unconstrained planning-table mutation. The governed discretionary work model is a separate open design item — catalog shape, activation criteria, authorization model, provenance emission. Tracked in the gap analysis priority list.
3. **Register maintenance.** New rejections should land as ADR amendments or superseding ADRs. Threshold for "is this rejection register-worthy?" is: an idea has been raised (or is likely to be raised) by more than one contributor, crosses one of the ten invariants, and the rationale is non-obvious from the invariant alone.
