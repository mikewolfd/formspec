# GPT (JDFM) — Feature Requirements Matrix

## Summary

JDFM (JSON Declarative Form Model) proposes a JSON-native format specification organized around three layers: instance data (values), behavior (bindings with XForms-style MIPs), and constraints (SHACL-inspired composable shapes). It adopts FHIR R5's canonical identity and lifecycle model for versioning and explicitly separates form logic from presentation. The expression language (JDFM-EL) uses ODK-style `${...}` field references with a pure functional design intended for deterministic dependency tracking and incremental recomputation.

### Strengths

- **Exceptionally thorough coverage.** 33 of 35 requirements are fully addressed. The proposal clearly worked through the requirements list systematically.
- **Strong conceptual lineage.** The XForms MIP model, SHACL validation shapes, and FHIR R5 canonical identity are well-understood and coherently integrated rather than superficially referenced.
- **`whenExcluded` semantics are a standout original contribution.** Separating submission pruning from evaluation masking with configurable policies is more nuanced than any of the source standards.
- **Validation modes are well-defined.** The four-tier draft/save/submit/audit model with normative non-blocking guarantees directly addresses the real-world problem of partial saves.
- **Dual addressing (FieldRef + DataPointer)** is a pragmatic solution to the tension between stable definition-time references and concrete runtime paths, especially important for repeat rows.
- **Concrete example is comprehensive.** A single cohesive example covers most of the hard cases (budget summing, conditional sections, repeats, YoY warnings, screener routing, external validation).

### Weaknesses

- **FL-04 (cross-form dependencies) is only partially addressed.** The `instances` mechanism handles read-only pre-loaded data, but there is no protocol for live cross-form reactivity or for one form's response to dynamically reference another active form session. The proposal treats cross-form data as static inputs.
- **VS-04 (cross-form validation) has the same limitation.** Prior-year comparisons work via pre-loaded instances, but true cross-form validation (e.g., identifier consistency checks across multiple simultaneously active submissions) lacks a defined mechanism.
- **Expression language grammar is underspecified.** JDFM-EL is described via examples and a "normative subset" of operators, but no formal grammar (EBNF, PEG, or railroad diagram) is provided. Operator precedence, associativity, and tokenization edge cases are left to implementers.
- **No formal JSON Schema provided.** The structural contract is described in prose and examples, but no companion JSON Schema is included. For a standard that claims to be "schema-driven," the absence of a validatable schema is a notable gap.
- **Modular composition is sketched but not specified.** The subform/assembly mechanism is mentioned as borrowed from SDC but lacks detail — no JSON structure for subform placeholders, no assembly algorithm, no conflict resolution rules.
- **The attachment type's runtime representation is example-only.** The `Attachment` object shape (filename, contentType, url, size, hash) appears only in the example, not in the normative type definitions.
