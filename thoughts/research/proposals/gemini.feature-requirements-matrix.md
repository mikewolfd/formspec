# Gemini (UDFA) -- Feature Requirements Matrix

## Proposal Summary

The Universal Declarative Form Architecture (UDFA) proposes a JSON-native standard built on a strict MVC separation with four core abstractions: FormDefinition (immutable blueprint), DataInstance (runtime state), BindGraph (reactive MIP engine), and ValidationShape (SHACL-inspired constraint layer). The expression language is "JEXL-F," a purpose-built, non-Turing-complete language using `${field}` syntax with dot-notation paths, explicit coercion rules, and built-in aggregate functions. Versioning follows FHIR R5 semantics with canonical URLs, semver, status lifecycles, response pinning via `derivedFrom`, and modular composition via an `$assemble` pattern.

### Strengths

- **Comprehensive conceptual model.** The four-entity architecture (FormDefinition, DataInstance, BindGraph, ValidationShape) provides clean separation of concerns and maps well to the MVC pattern inherited from XForms.
- **Strong XForms lineage.** MIP semantics (calculate, constraint, relevant, required, readonly) are faithfully transposed to JSON with clear execution semantics and non-relevant data exclusion.
- **Robust validation architecture.** The SHACL-inspired shape-based validation model with three severity tiers, validation modes, external injection, and structured results addresses virtually all validation requirements.
- **Well-specified versioning.** FHIR R5-derived canonical URLs, semver, response pinning, lifecycle states, and modular assembly cover evolution requirements comprehensively.
- **Concrete examples.** Six hard-case examples (Sections 7.1-7.6) demonstrate real JSON structures for budget totals, conditional sections, year-over-year warnings, screener routing, external validation, and calculated fields in repeatable contexts.
- **Financial type as first-class citizen.** The `financial` composite type (amount + currency) is a thoughtful domain-specific design that avoids floating-point issues.

### Weaknesses

- **Expression language underspecified.** JEXL-F is described at a conceptual level but lacks a formal grammar, operator precedence table, complete built-in function catalog, and rigorous type coercion rules. The name suggests derivation from Apache JEXL or the npm `jexl` package but the exact relationship and deviations are not defined.
- **JSON examples are incomplete or malformed.** Several critical examples (Sections 7.1, 7.3, 7.5) have truncated `shapes` arrays or `validationReport` arrays shown as empty brackets, leaving the reader to infer the exact structure. Some JSON snippets contain escaped characters that obscure the actual syntax.
- **No formal schema definition.** The proposal describes concepts but does not provide a JSON Schema for FormDefinition, DataInstance, ValidationResult, or any other entity. A standards-body submission would require normative schemas.
- **Cross-form validation limited to snapshots.** VS-04 (cross-form validation) is addressed only via read-only secondary instances, which handles prior-year comparison but not real-time cross-submission consistency checks or identifier validation across simultaneously open forms.
- **Repeat cardinality unspecified.** While repeatable sections are well-addressed syntactically, min/max instance constraints (e.g., "at least 1 line item, no more than 50") are not explicitly discussed.
- **Bidirectional mapping briefly sketched.** Section 8.3 mentions CommonGrants-inspired MappingSchema but provides no concrete JSON structure or detailed semantics.
- **Narrative/long-text distinction absent.** FT-01 lists "narrative/long-text" as a distinct requirement; UDFA subsumes it under `string` without a semantic marker for multiline or rich-text content.
- **Heavy on prose, light on normative specification.** The document reads more as an architecture whitepaper than a standard specification. It lacks the normative language (MUST, SHOULD, MAY), conformance levels, and processing rules that a standards-body submission would require.
