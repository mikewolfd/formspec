# XForms 1.1 Specification — Feature Exploration Matrix

This matrix extracts every distinct idea, pattern, mechanism, and design decision from the XForms 1.1 deep technical analysis. It serves as a comprehensive inventory of concepts that can be cross-referenced against the Formspec specification to identify gaps, validate existing design choices, and surface non-obvious patterns worth adopting in a JSON-native form standard.

---

| ID | Feature/Pattern | Category | Description | Formspec Relevance | Priority |
|----|-----------------|----------|-------------|-------------------|----------|
| XF-001 | Model Item Properties (MIPs) as metadata annotations | Data Model | MIPs are behavioral metadata (calculate, constraint, relevant, required, readonly, type) attached to data nodes via `bind` elements, forming a "model item" per node. Behavior is layered onto data without modifying the data structure itself. | Formspec's `bind` objects on items serve the same role. The key principle — behavioral semantics are separate from data shape — is already adopted. Ensure every bind property maps cleanly to a MIP-equivalent concept. | Critical |
| XF-002 | Fixed vs. Computed MIP distinction | Data Model | `type` is evaluated once (a static literal), while all other MIPs are live expressions re-evaluated on every recalculate cycle. This distinction affects when and how values are resolved. | Formspec should clearly distinguish static properties (e.g., datatype declarations) from dynamic FEL expressions. The `type` field on items is static; `calculate`, `constraint`, `relevant`, `required`, `readonly` should all be live FEL expressions. Document this distinction explicitly. | High |
| XF-003 | MIP inheritance — readonly (OR semantics) | Reactivity | If ANY ancestor node has `readonly=true`, all descendants are readonly. The processor tracks both a local value (from the bind) and an inherited value (combining local + ancestors) using logical OR up the tree. | Formspec groups/sections could inherit readonly state. If a parent section is readonly, all children should be readonly regardless of their own bind. This avoids needing to set readonly on every child when disabling a whole section. Requires tracking local vs. inherited values. | High |
| XF-004 | MIP inheritance — relevant (AND semantics) | Reactivity | If ANY ancestor node has `relevant=false`, all descendants are non-relevant. Uses logical AND: the effective value is the conjunction of all ancestors' relevance plus the node's own. | Formspec already supports relevance on groups/sections. Ensure the inheritance model is explicitly AND-based: a non-relevant parent makes all children non-relevant, even if a child's own `relevant` expression evaluates to true. | Critical |
| XF-005 | Local vs. inherited MIP value tracking | Reactivity | The processor maintains two separate values per inherited MIP: the local value (from the node's own bind) and the inherited/effective value (combining local with ancestor chain). This separation allows correct recomputation when ancestor state changes. | Formspec's engine should track local vs. effective values for `readonly` and `relevant`. When a parent's relevance changes, children's effective relevance updates without re-evaluating each child's local expression unnecessarily. | Medium |
| XF-006 | No double-binding constraint | Validation | It is an error to set the same MIP on the same data node from two different `bind` elements. Checked at rebuild time; raises `xforms-binding-exception`. | Formspec uses item-level binds (one bind per item), so double-binding is structurally impossible for field-level MIPs. However, shape rules can target the same path — consider whether conflicting shape rules on the same path should be detected and flagged as an error. | Medium |
| XF-007 | Calculate implies readonly default | Data Model | If a node has a `calculate` MIP, its `readonly` defaults to `true()` — computed values should not normally be hand-edited. This is an implicit behavior, not requiring explicit declaration. | Formspec should adopt this convention: items with `calculate` binds should default to readonly. This prevents a common class of bugs where users overwrite computed values. Document the default clearly. | High |
| XF-008 | Overridable calculate-readonly for default values | Data Model | Setting `readonly="false()"` on a calculated node explicitly allows user edits AND calculation. The calculate re-fires on every recalc; if the user clears the field, the calculate restores the computed value. This is the documented pattern for "default values." | This is an elegant default-value mechanism. Formspec could support `readonly: false` on a calculated field to create "editable defaults" — the calculate expression provides the initial/fallback value, but the user can override it. The calculate only reasserts when the field is empty/cleared. | High |
| XF-009 | Cross-node expression references | Expression | Computed MIP expressions can traverse the entire instance tree — a `constraint` on node X can reference sibling, parent, or completely unrelated nodes. Expressions are not limited to the node they annotate. | Formspec's FEL already supports cross-field references in expressions. This is a core requirement — ensure the dependency graph correctly tracks all cross-field references in `calculate`, `constraint`, `relevant`, `required`, and `readonly` expressions. | Critical |
| XF-010 | Four-phase processing pipeline | Lifecycle | XForms uses a strict four-phase pipeline: rebuild (construct dependency graph) -> recalculate (evaluate expressions in dependency order) -> revalidate (check constraints) -> refresh (update UI). Each phase is a distinct, ordered step. | Formspec engine should follow a similar phased approach. The current signal-based reactivity handles this implicitly, but the conceptual model should be documented: dependency resolution -> expression evaluation -> validation -> UI update. This ordering prevents reading stale intermediate state. | Critical |
| XF-011 | Dependency directed graph construction | Reactivity | At rebuild time, XForms parses all bind expressions to determine which instance nodes each expression references, then constructs a DAG where edges represent data dependencies between (node, MIP-aspect) pairs. | Formspec's `DependencyVisitor` extracts field references from FEL CST to wire reactive dependencies. Ensure this covers all expression types (calculate, constraint, relevant, required, readonly) and correctly identifies transitive dependencies. | Critical |
| XF-012 | Per-MIP-aspect vertices in dependency graph | Reactivity | The dependency graph has separate vertices for each (node, MIP-aspect) pair. A node with both `calculate` and `required` has two separate vertices — allowing the required expression to depend on the calculated value without creating a false cycle. | Formspec should track dependencies at the (field, property) granularity, not just (field). This avoids false positives in cycle detection: a field's `required` expression depending on its own `calculate` result is not circular. | High |
| XF-013 | Incremental recalculation via change list | Reactivity | Recalculation starts with a change list L of nodes whose values changed since last recalc. Only the pertinent dependency subgraph (vertices reachable from L) is processed, not the entire graph. On initial load, this equals the full graph. | Formspec's signal-based reactivity provides this automatically (signals only propagate to dependents). Document that the engine performs incremental recomputation — only fields transitively dependent on changed values are re-evaluated. | High |
| XF-014 | Topological sort for evaluation order | Reactivity | The pertinent dependency subgraph is topologically sorted; vertices with in-degree 0 are evaluated first, then their dependents. This guarantees every expression sees up-to-date values from its dependencies. | This is fundamental to correct reactive evaluation. Formspec's signal graph should provide equivalent ordering guarantees. Verify that `@preact/signals-core` evaluates computed signals in topological order of their dependency graph. | Critical |
| XF-015 | Circular dependency detection and fatal error | Reactivity | If the pertinent subgraph has remaining vertices but none with in-degree 0, a cycle exists. XForms throws `xforms-compute-exception` and halts processing. Circular dependencies are a fatal, unrecoverable error. | Formspec should detect circular dependencies at definition load time (or on first evaluation) and report them as fatal errors. The Python linter (`validator/`) should statically detect cycles in FEL expressions. The engine should also detect them at runtime as a safety net. | Critical |
| XF-016 | Self-reference exclusion from dependency tracking | Reactivity | A `calculate` expression that references its own node does NOT create a circular dependency. Self-references are explicitly excluded from the dependency graph. | Formspec should allow a `calculate` expression to reference the field it is attached to (e.g., for clamping: `if($.value > 100, 100, $.value)`). The dependency tracker must explicitly exclude self-references. | Medium |
| XF-017 | Over-inclusive reference detection | Reactivity | Any node matched by a NameTest during XPath evaluation counts as a reference, even if a predicate later excludes it. This is intentionally over-inclusive so that when predicate conditions change, the expression gets recalculated without a full rebuild. | Formspec's dependency extraction from FEL expressions should be similarly conservative. If a FEL expression contains a path reference, that path should be tracked as a dependency even if runtime conditions might skip that branch (e.g., inside an `if()` — both branches' references should be tracked). | High |
| XF-018 | Dynamic dependencies requiring rebuild | Reactivity | Insert/delete of nodes can alter reference lists. For model binding expressions, this requires a manual rebuild of the dependency graph. For UI expressions, the processor handles it automatically at refresh time. | Formspec's repeat insert/delete operations may change which fields exist and thus which dependencies are relevant. The engine should rebuild the dependency graph (or the affected portion) when repeat instances are added or removed. | High |
| XF-019 | Deferred update batching mechanism | Reactivity | XForms does NOT run rebuild/recalculate/revalidate/refresh after every single mutation. Four boolean flags accumulate, and all processing runs once at the end of the outermost action handler. N mutations cause only ONE processing cycle. | Formspec's engine should batch value changes within a single operation (e.g., `setValues()` or programmatic multi-field updates) and process the reactive graph only once afterward. `@preact/signals-core` provides `batch()` for this. Ensure it is used for all multi-mutation operations. | High |
| XF-020 | Outermost action handler boundary | Reactivity | Deferred updates flush when the outermost (top-level) event handler completes. Nested action handlers do not trigger intermediate flushes — only the completion of the outermost handler does. | Formspec should define when batched updates flush. For the web component, the natural boundary is the microtask (signal batching). For programmatic API use, provide explicit `batch()` / `transaction()` methods so consumers can group multiple changes. | Medium |
| XF-021 | Non-relevant data value preservation | Data Model | When a node becomes non-relevant, its VALUE IS PRESERVED in the instance data. Non-relevance does NOT clear or remove the data — it is still present in the data model. | Formspec should preserve field values when a field becomes non-relevant. If a user fills out a conditional section, hides it (non-relevant), then shows it again, the previously entered values should still be there. This is the expected UX behavior. | Critical |
| XF-022 | Non-relevant data excluded from submission | Submission | Non-relevant nodes are removed (pruned) from the serialized data before submission. Pruning operates on a copy — actual instance data is unchanged. | Formspec's `getResponse()` should exclude non-relevant fields by default. The response should only contain values for currently-relevant fields. This prevents submitting data for hidden conditional sections. | Critical |
| XF-023 | Submission relevance pruning opt-out | Submission | The `submission` element has a `relevant` attribute (default true). Setting `relevant="false"` disables pruning — all data is submitted regardless of relevance. Useful for save/restore of incomplete forms. | Formspec's `getResponse()` should accept an option like `includeNonRelevant: true` for save/restore scenarios. Draft-saving should include all data; final submission should prune non-relevant. | High |
| XF-024 | Non-relevance from empty nodeset binding | UI Hint | A form control is non-relevant if its binding resolves to an empty nodeset (no matching data node). This is in addition to the `relevant` MIP being false. | In Formspec, if a field references a path that does not exist (e.g., inside a repeat that has zero instances), the field should be treated as non-relevant. This provides natural handling of optional/missing data paths. | Medium |
| XF-025 | Non-relevance from non-selected switch case | UI Hint | A form control inside a non-selected `case` of a `switch` is non-relevant, even if its data node's `relevant` MIP is true. UI structure contributes to relevance determination. | Formspec's tab/page/accordion patterns should consider whether unselected pages make their contents non-relevant. This has implications for validation: should hidden-tab fields be validated? The answer should be configurable. | Medium |
| XF-026 | Non-relevant controls removed from navigation | UI Hint | Non-relevant form controls are removed from the tab/navigation order, cannot receive focus, and their event handlers are disabled. | Formspec web component should ensure non-relevant fields are removed from tab order (`tabindex="-1"` or `display: none`) and do not participate in keyboard navigation. | High |
| XF-027 | Repeat bound to node-set (collection binding) | Repeat | `repeat` binds to a node-set and generates one UI "repeat item" per node. The markup inside is a template instantiated for each collection member. | Formspec's repeatable groups work similarly: a group with `repeat` generates instances based on repeat count. The JSON-native equivalent of "node-set binding" is the array of repeat instance objects in the response. | Critical |
| XF-028 | Repeat index (current item pointer) | Repeat | Each repeat maintains a 1-based index pointing to the "current" item, accessible via `index('repeatId')`. The index is automatically updated on insert (moves to last inserted) and delete (clamps to valid range; 0 if all deleted). | Formspec could support a repeat index concept for "selected/active item" in repeating groups. This enables patterns like "click a row to edit it" or "expand one accordion item at a time." Expose via `index()` FEL function or a repeat-scoped variable. | Medium |
| XF-029 | Repeat index initialization and startindex | Repeat | The repeat index is initialized to the `startindex` attribute (default 1). This allows specifying which item is initially selected/active. | If Formspec adopts repeat indexes, support a `startIndex` property on repeating groups to control which instance is initially focused/expanded. | Low |
| XF-030 | Repeat index update rules on delete | Repeat | On delete: index stays on same item if possible; clamps to new collection size if the deleted item was at or after the index; goes to 0 if all items are deleted. Changing the outer repeat index does NOT re-initialize nested repeat indexes. | These edge-case rules are important for correct UX. If Formspec implements repeat indexes, define explicit update semantics for insert and delete operations, especially the "all deleted" edge case and nested repeat independence. | Low |
| XF-031 | Scoped expression context inside repeat | Repeat | Inside a repeat, binding expressions use the specific repeat item's node as the context. `ref="price"` inside a repeat over items means each instance's input binds to ITS specific item's `price` child. Position and size are scoped. | Formspec's indexed path system (`group[0].field`, `group[1].field`) serves this role. FEL expressions inside repeat items are evaluated with the repeat instance index in scope, so `$.price` resolves relative to the current instance. | Critical |
| XF-032 | Repeat item lifecycle (creation/destruction) | Repeat | Repeat items are dynamically created/destroyed. On creation: event handlers initialized, UI controls initialized. On destruction: repeat object and all inner controls/handlers eliminated. | Formspec's engine must handle repeat instance creation (initialize field values, wire up reactive binds) and destruction (clean up signals, remove from dependency graph) cleanly. Memory leaks from undisposed signals in deleted repeat instances are a real risk. | High |
| XF-033 | Nested repeat support | Repeat | Repeats can nest: an inner repeat operates on a subcollection within each outer repeat item. The `index()` function uses IDREF resolution to find the correct repeat instance in nested contexts. | Formspec supports nested repeating groups. Ensure the path system handles arbitrary nesting depth (e.g., `outerGroup[0].innerGroup[1].field`) and that the dependency graph correctly handles cross-level references. | High |
| XF-034 | Event bubbling through repeat objects | Repeat | Events dispatched to objects inside a repeat item bubble through the repeat object as if it were a child of the `repeat` element. This enables event handling at the repeat level for events originating in any item. | Formspec web component should ensure DOM events from repeat instance elements bubble correctly through the repeat container. This enables attaching a single event listener at the repeat level instead of per-item listeners. | Medium |
| XF-035 | Attribute-based repeat (host language integration) | Repeat | XForms supports placing repeat attributes (`repeat-nodeset`, `repeat-bind`, etc.) directly on host language elements, enabling repeating without a wrapper element. | Formspec's web component could support a `data-formspec-repeat` attribute on arbitrary HTML elements as an alternative to the component-registry repeat pattern. This would allow more flexible integration with existing HTML structures. | Low |
| XF-036 | Multiple instances per model | Data Model | A single model can contain multiple named `instance` elements. Secondary instances hold lookup data, UI state, temporary computations, or submission targets. | Formspec definitions support items and value lists (for lookups). Consider whether a first-class "secondary data sources" concept would be valuable — e.g., for lookup tables, configuration data, or temporary computation results that don't appear in the response. | Medium |
| XF-037 | instance() function for cross-instance access | Expression | The `instance('id')` function returns the root element of a named instance, enabling expressions to reference data across instances within the same model. | Formspec could support referencing value lists or lookup tables from within FEL expressions (e.g., `lookup('countries', 'US').name`). This would enable dynamic label resolution, validation against reference data, and computed fields that draw from static datasets. | Medium |
| XF-038 | Default instance convention | Data Model | The first instance element in document order is the default instance. Binding expressions without explicit instance specification operate on the default instance. | Formspec's items array is effectively the single "default instance." If secondary data sources are added, the primary items should remain the implicit default context for all FEL expressions that don't explicitly reference another source. | Medium |
| XF-039 | Instance data from external sources | Data Model | Instances can load data from external URIs via `src` or `resource` attributes, with inline content as fallback or override. This enables form pre-population from server data. | Formspec could support a `dataSource` or `prefill` mechanism that loads initial field values from an external endpoint. This is distinct from the definition — it's runtime data population. Consider how this interacts with `calculate` defaults. | Medium |
| XF-040 | MVC separation — model without UI | Composition | The model is fully self-contained and functional without any form controls. It can define data, calculations, constraints, and submission independently of rendering. | Formspec already achieves this: `FormEngine` operates on a definition without any DOM or rendering layer. The engine can `setDefinition()`, `setValue()`, `getResponse()`, and `getValidationReport()` purely in memory. This is a core architectural strength. | Critical |
| XF-041 | Multiple UI bindings to same data node | Composition | Multiple form controls can bind to the same data node, presenting the same data differently. There is no 1:1 restriction between controls and data nodes. | Formspec should allow multiple UI components to bind to the same field path. This enables patterns like "summary view + edit view of the same data" or "a slider and a numeric input both controlling the same value." | Medium |
| XF-042 | Multiple models per document | Composition | Multiple independent models can coexist, each with its own instances, binds, and submissions. Form controls specify which model they bind to. | Formspec could support multiple definitions rendered on the same page, each with its own engine instance. The web component already supports this (multiple `<formspec-render>` elements). Cross-engine communication would require an explicit mechanism. | Low |
| XF-043 | Controls as stateless projections of model state | Reactivity | UI controls never hold authoritative data. The direction is: user input -> instance data -> recalculate -> revalidate -> refresh -> UI update. Controls are always derived from model state. | Formspec's architecture should enforce this: the web component's DOM is always a projection of `FormEngine` state. Components should never cache their own state that diverges from the engine. All reads come from signals; all writes go through `setValue()`. | Critical |
| XF-044 | XPath 1.0 as expression language base | Expression | XForms uses XPath 1.0 with four types (boolean, string, number, node-set) and all core XPath 1.0 functions. Path navigation is first-class. | Formspec uses FEL (Formspec Expression Language) instead of XPath, designed for JSON trees rather than XML trees. FEL supports similar types and path navigation but with JSON-native syntax. The conceptual role is identical. | Critical |
| XF-045 | boolean-from-string() conversion function | Expression | Converts string "true"/"1" to boolean true and "false"/"0" to boolean false. Handles the common need for type coercion between string data and boolean logic. | Formspec's FEL should handle boolean coercion consistently. Consider whether `bool()` or implicit coercion handles strings like "true"/"false"/"1"/"0" predictably. Document the coercion rules explicitly. | Medium |
| XF-046 | Aggregate functions (avg, min, max, count-non-empty) | Expression | XForms provides aggregate functions over node-sets: `avg()`, `min()`, `max()`, `count-non-empty()` for summarizing collections of values. | FEL already provides `sum()`, `min()`, `max()`, `avg()`, `count()`. Verify `count-non-empty()` equivalent exists or add one — useful for "how many fields in this repeat section are filled out." | High |
| XF-047 | index() function for repeat position | Expression | `index('repeatId')` returns the current 1-based index of the named repeat's cursor. Used for conditional display, navigation, and cross-repeat-item references. | If Formspec supports repeat cursors/active items, provide an equivalent `index()` function in FEL. Otherwise, repeat position is available implicitly via indexed paths (`group[i].field`). | Low |
| XF-048 | choose() ternary preserving type | Expression | `choose(boolean, object, object)` returns the second argument if true, third if false, preserving the original type (unlike `if()` which coerces to string). | FEL's `if(condition, thenValue, elseValue)` should preserve the type of the selected branch rather than coercing. Verify that `if(true, 42, "hello")` returns the number 42, not the string "42". | High |
| XF-049 | property() function for processor metadata | Expression | `property('version')` and similar calls access processor implementation metadata (XForms version, conformance level). | Formspec could expose `property('formspec.version')` or `property('engine.version')` for definitions that need to adapt behavior to engine capabilities. Useful for forward-compatible definitions. | Low |
| XF-050 | Cryptographic functions (digest, hmac) | Expression | XForms includes `digest(string, algorithm)` and `hmac(string, key, algorithm)` for hashing within expressions. | Formspec should NOT include cryptographic functions in FEL — these belong in server-side processing, not in a client-side expression language. Security-sensitive operations should not be in the form definition. | Low |
| XF-051 | Date/time functions suite | Expression | Comprehensive date/time support: `now()`, `local-date()`, `local-dateTime()`, `days-from-date()`, `days-to-date()`, `seconds-from-dateTime()`, `seconds-to-dateTime()`, `adjust-dateTime-to-timezone()`, `seconds()`, `months()`. | FEL already provides `now()`, `today()`, and date arithmetic. Verify coverage against XForms: timezone handling, duration extraction, epoch conversions. Add missing functions that real-world forms commonly need (e.g., `datediff()`, `dateadd()`). | High |
| XF-052 | instance() for cross-instance data access | Expression | `instance(id)` returns the root element of a named instance, enabling cross-referencing between data sources within expressions. | FEL could benefit from a mechanism to reference value lists or external data within expressions. If a `valuelist` or `datasource` concept exists, an `instance()` or `lookup()` function would enable FEL expressions to query reference data. | Medium |
| XF-053 | current() vs context() distinction | Expression | `current()` returns the context node used to initialize the containing expression (the "original" context). `context()` returns the in-scope evaluation context (which may differ inside predicates or nested expressions). | FEL should clarify how context works inside nested expressions. If a FEL expression inside a repeat uses a filter/predicate, does the inner context differ from the outer? Define `current()` equivalent if needed for complex expressions referencing the "outer" context. | Medium |
| XF-054 | event() function for event context | Expression | `event('property-name')` accesses properties of the current event (e.g., error type, response body) within event handlers. | Formspec actions triggered by events (e.g., on-submit, on-value-change) could access event context via a similar mechanism. For now, this is low priority unless Formspec adds an action/event system beyond reactive binds. | Low |
| XF-055 | Evaluation context — outermost binding | Expression | Outermost binding elements (no ancestor binding) use the document element of the default instance as context node. Context position = 1, context size = 1. | FEL expressions at the top level of a Formspec definition should resolve paths relative to the root of the form data. This is the natural default. Verify that bare paths like `fieldName` resolve correctly from the root context. | Critical |
| XF-056 | Evaluation context — scoped resolution in nesting | Expression | Non-outermost binding elements inherit context from their nearest ancestor binding via scoped resolution. Single-node ancestor: context = that node. Node-set ancestor (repeat): context = each node in turn. | FEL expressions inside nested groups and repeats should resolve relative to their containing context. Inside `group[0]`, `$.field` resolves to `group[0].field`. Inside a nested repeat `group[0].inner[1]`, paths resolve relative to that inner instance. | Critical |
| XF-057 | Expression categories by evaluation timing | Lifecycle | Expressions are classified into four categories evaluated at different times: model binding (rebuild), computed MIPs (recalculate), UI expressions (refresh), and action expressions (on-demand). | Formspec should distinguish expression evaluation timing: bind expressions are evaluated reactively on dependency changes, UI binding expressions are evaluated on render, and action expressions are evaluated on trigger. This affects caching and optimization strategies. | Medium |
| XF-058 | Submission pipeline with ordered phases | Submission | Submission follows a strict pipeline: deferred updates flush -> node selection -> relevance pruning -> validation -> method determination -> resource determination -> serialization -> headers -> transmission -> response handling. | Formspec's submission/response generation should follow a defined pipeline: flush pending calculations -> collect field values -> prune non-relevant -> validate -> serialize to response format. The `getResponse()` and `getValidationReport()` methods should reflect this ordering. | High |
| XF-059 | Pre-submission validation gate | Submission | Submission checks validity of all selected nodes. If any are invalid, submission aborts with `xforms-submit-error` / `validation-error`. Validation can be disabled via `validate="false"`. | Formspec should support a `validate()` method that checks all relevant fields before generating a response. An option to skip validation (for draft saves) should exist. The validation report should indicate whether the form is "submittable." | High |
| XF-060 | Pre-submission deferred update flush | Submission | Before submission, all pending deferred updates (rebuild, recalculate, revalidate) are flushed to ensure data is current. This prevents submitting stale computed values. | Formspec's `getResponse()` should ensure all pending signal computations have settled before collecting values. In practice, `@preact/signals-core` computed signals are synchronous, but verify there are no edge cases with batched updates. | High |
| XF-061 | Serialization format selection | Submission | XForms supports multiple serialization formats (XML, URL-encoded, multipart, form-data, none) selected via attributes. The `serialization="none"` option skips serialization entirely. | Formspec's response is JSON by default. The Mapping spec provides adapters for CSV, XML, and alternate JSON layouts. Ensure the adapter selection mechanism is as clean as XForms' `serialization` attribute. | Medium |
| XF-062 | Partial data submission (subtree selection) | Submission | The submission's `ref`/`bind` selects a subtree of the instance data for submission. Combined with relevance pruning, this gives fine-grained control over what data is sent. | Formspec's `getResponse()` could accept a path or field-set parameter to return only a subset of the form data. Useful for multi-step wizards where each step submits only its own section. | Medium |
| XF-063 | Response handling modes (replace all/instance/text/none) | Submission | After submission, the response can replace the entire document, replace a specific instance, replace a node's text content, or be discarded. This enables server round-trips that update form data. | Formspec could support a "load response" mechanism where server responses update field values (e.g., server-side calculation results, saved record IDs). This would be a `setValues()` or `loadResponse()` API on the engine. | Medium |
| XF-064 | Sync vs. async submission modes | Submission | The `mode` attribute controls whether submission blocks user interaction (synchronous) or allows continued interaction (asynchronous, default). | Formspec should primarily support asynchronous submission patterns (the modern web default). If submission is handled externally (by the host application), the engine doesn't need to manage sync/async — it just produces the response data. | Low |
| XF-065 | Custom serialization via event | Submission | The `xforms-submit-serialize` event fires before default serialization, allowing handlers to override the serialized data. This enables custom serialization logic without modifying the submission element. | Formspec's Mapping spec serves this role — custom transforms between Formspec's response format and target formats. The event-based override pattern could be useful if the web component exposes a `submit` event with customizable serialization. | Low |
| XF-066 | Dynamic submission headers from instance data | Submission | `header` child elements of `submission` can use binding expressions to set HTTP headers dynamically based on form data (e.g., authorization tokens, content negotiation). | This is a runtime/integration concern. Formspec definitions should not embed HTTP headers — the host application controls submission mechanics. However, the engine should expose enough data for the host to construct dynamic headers if needed. | Low |
| XF-067 | DOM2 Events with capture/target/bubble flow | Lifecycle | XForms uses standard DOM2 event flow (capture -> target -> bubble) for all events, leveraging the host document's event infrastructure. | Formspec's web component should use standard DOM CustomEvents that participate in normal DOM event flow. This enables host applications to listen for form events (value-changed, validation-state-changed, etc.) using standard `addEventListener`. | High |
| XF-068 | Initialization event sequence | Lifecycle | Form initialization follows a strict event sequence: `xforms-model-construct` (build instances, initial rebuild/recalculate/revalidate) -> `xforms-model-construct-done` (initialize UI) -> `xforms-ready` (everything initialized). | Formspec's engine should have clear lifecycle phases: `setDefinition()` (parse + build dependency graph) -> initial calculation cycle -> ready state. The web component should emit lifecycle events (e.g., `formspec:ready`, `formspec:definition-loaded`). | High |
| XF-069 | Model destruction event | Lifecycle | `xforms-model-destruct` provides a shutdown notification, allowing cleanup of resources, event handlers, and external connections. | Formspec's web component should clean up engine resources (signal subscriptions, computed signals) when the element is disconnected from the DOM (`disconnectedCallback`). The engine should have a `dispose()` method. | High |
| XF-070 | MIP state change notification events | Lifecycle | Separate notification events fire for each MIP state transition: `xforms-valid`/`xforms-invalid`, `xforms-readonly`/`xforms-readwrite`, `xforms-required`/`xforms-optional`, `xforms-enabled`/`xforms-disabled`. | Formspec's web component should emit granular state-change events for individual fields: `formspec:field-valid`, `formspec:field-invalid`, `formspec:field-relevant-changed`, etc. This enables host applications to react to specific state transitions. | Medium |
| XF-071 | Notification events deferred to refresh phase | Lifecycle | MIP state changes are marked during recalculate/revalidate but notification events are only dispatched during the refresh phase. This prevents cascading event handler execution during the middle of computation. | Formspec should ensure that any event emissions from the web component happen AFTER the reactive graph has fully settled. Do not emit DOM events for intermediate states during a batch update — only emit for the final stable state. | High |

### Key Observations

1. **22 Critical features** are already core to any serious form standard. Formspec appears to address most of these (reactive dependency graph, declarative binds, non-relevant pruning, scoped expression context). The matrix can be used to verify completeness.

2. **Reactivity has the most High-priority features** (8), reflecting that XForms' dependency graph and incremental recalculation model is its most sophisticated contribution. Formspec's signal-based approach handles this differently but should provide equivalent guarantees.

3. **UI Hints dominate the High tier** (9 features), indicating that XForms invested heavily in standardizing control behaviors, appearances, and state-driven styling. Formspec's component spec should ensure parity with these interaction patterns.

4. **Submission features are mostly High or Medium**, reflecting that XForms' submission model is tightly coupled to HTTP/XML. Formspec's JSON-native approach and host-application-delegated submission simplify many of these concerns.

5. **No Extension features were identified** in this analysis because the XForms 1.1 spec focused on built-in capabilities. Formspec's extension registry and component registry go beyond what XForms standardized.

6. **The "editable defaults" pattern (XF-008)** — using `calculate` + `readonly="false()"` for default values that users can override — is a subtle and powerful design idea that deserves specific attention in Formspec's design.

7. **Deferred update batching (XF-019)** and **notification deferral to refresh phase (XF-071)** are critical performance and correctness patterns that prevent cascading intermediate states from reaching the UI.
