# W3C SHACL — Feature Exploration Matrix

This matrix inventories every distinct concept, mechanism, and design pattern identified in the W3C SHACL (Shapes Constraint Language) specification analysis. Each feature is evaluated for its relevance to Formspec — a JSON-native declarative form specification — with the goal of enabling systematic cross-referencing against the synthesized Formspec proposal. SHACL's graph-oriented validation model offers significant structural parallels to form-level constraint systems, particularly in its composable shapes, structured validation reports, and extensible constraint components.

| ID | Feature/Pattern | Category | Description | Formspec Relevance | Priority |
|---|---|---|---|---|---|
| SH-001 | Shape as Composable Validation Unit | Validation | A shape is a named, referenceable entity that bundles a set of constraints. Shapes can reference other shapes, creating a graph of validation logic. | Directly analogous to Formspec's "shape rules" — cross-field/form-level constraints that can be composed and reused. Formspec should ensure shapes are first-class referenceable objects, not just inline constraint lists. | Critical |
| SH-002 | NodeShape vs PropertyShape Distinction | Validation | NodeShape validates a focus node as a whole (no path). PropertyShape validates value nodes reachable via a property path. The two types are disjoint. | Maps to Formspec's distinction between bind constraints (field-level, path-scoped) and shape rules (form-level, entity-scoped). Formalizing this separation improves clarity about what is being validated. | High |
| SH-003 | Shape–Data Separation | Data Model | Shapes exist in a shapes graph completely decoupled from the data graph. They are two independent inputs to validation. | Formspec already separates definition (the "shape") from response (the "data"). This pattern validates that design and argues for keeping validation rules portable and data-agnostic. | Critical |
| SH-004 | Target Declarations | Data Model | Shapes declare which data nodes to validate via target types (targetNode, targetClass, targetSubjectsOf, targetObjectsOf). The target of a shape is the union of all its declared targets. | Analogous to Formspec's path-based field targeting (e.g., `items[*].field` wildcards). SHACL's richer targeting vocabulary suggests Formspec could benefit from more expressive targeting patterns beyond simple path wildcards. | Medium |
| SH-005 | Implicit Class Target | Data Model | If a shape is also declared as an rdfs:Class, all instances of that class become targets automatically. | Formspec could adopt a convention where a named item group implicitly targets all its member fields for group-level constraints, reducing boilerplate in shape rule declarations. | Low |
| SH-006 | Target as Discovery, Not Constraint | Data Model | Targets say "validate THESE nodes" — they do not require those nodes to exist. Target nodes need not be present in the data graph. | Relevant to Formspec's handling of optional fields and conditional relevance. A validation target should not imply the field must have a value; cardinality constraints handle that separately. | Medium |
| SH-007 | Three Severity Levels | Validation | SHACL defines Violation, Warning, and Info severity levels, with Violation as the default. Severity is declared on the shape, not on individual constraints. | Formspec already has error/warning/info severity in ValidationReport. SHACL confirms this is the right granularity. Consider whether severity should be declarable per-shape-rule (as SHACL does) rather than per-individual-constraint. | Critical |
| SH-008 | Severity as Metadata, Not Logic | Validation | Severity has NO impact on validation logic or the conforms boolean. It is purely a categorization mechanism for UI/reporting tools. | Formspec should decide whether severity affects the "valid" boolean on submission. SHACL's approach (all results = non-conforming) is strict; Formspec may want warnings/info to NOT block submission, diverging from SHACL intentionally. | High |
| SH-009 | Custom Severity IRIs | Extension | Any IRI can be used as a custom severity — SHACL is not limited to the three built-in levels. | Formspec's extension registry could allow custom severity levels for domain-specific triage (e.g., "audit", "compliance", "suggestion"). | Low |
| SH-010 | Structured ValidationResult | Validation | Each result is a structured record with focusNode, resultSeverity, sourceConstraintComponent, optional resultPath, value, sourceShape, resultMessage, and detail. | Directly maps to Formspec's ValidationReport results. SHACL's `sourceConstraintComponent` (the type of check that failed) and `value` (the specific offending value) are especially useful — Formspec should ensure its results include constraint-kind and offending-value fields. | Critical |
| SH-011 | Mandatory Result Fields | Validation | focusNode, resultSeverity, and sourceConstraintComponent are MANDATORY on every validation result. | Formspec should mandate equivalent fields: target field path, severity, and constraint kind. This enables programmatic handling and automated remediation. | Critical |
| SH-012 | Result Path for Locating Violations | Validation | `sh:resultPath` identifies the property path being validated, enabling tools to navigate directly to the problematic property. | Formspec already uses path-based targeting. Ensuring every validation result includes the resolved field path (including repeat indices) is essential for UI error placement. | Critical |
| SH-013 | Offending Value in Results | Validation | `sh:value` pinpoints the specific value that caused the violation, not just "this field is wrong." | Formspec results should include the actual value that failed validation. This aids debugging, audit trails, and programmatic error correction. | High |
| SH-014 | Source Shape in Results | Validation | `sh:sourceShape` links back to the shape that declared the constraint, enabling traceability from result to rule. | Formspec results should reference the originating bind constraint or shape rule ID. Critical for audit and for UIs that need to explain why a rule exists. | High |
| SH-015 | Multi-language Result Messages | Validation | `sh:resultMessage` supports multiple values with different language tags, inherited from `sh:message` on the shape or auto-generated by the processor. | Formspec should support i18n for validation messages. SHACL's approach of allowing both author-specified and processor-generated messages is a good pattern. | Medium |
| SH-016 | Detail Nesting in Results | Validation | `sh:detail` links a result to child AbstractResult nodes providing deeper causal explanation (e.g., why an sh:node check failed). | Valuable for complex cross-field validations. Formspec shape rules that compose multiple sub-constraints could produce nested detail explaining which sub-constraint actually failed. | Medium |
| SH-017 | Fresh Result Nodes | Validation | Results are always fresh — even if the same node is validated against the same shape via different paths, distinct result nodes are created. | Formspec should guarantee result uniqueness. If the same field is validated by multiple rules, each produces a separate result. No deduplication by default. | Medium |
| SH-018 | ValidationReport Top-Level Structure | Validation | A report contains exactly one conforms boolean and zero or more ValidationResult nodes. Results are a flat union of all violations found. | Formspec's ValidationReport already follows this pattern. SHACL confirms the flat-collection-with-optional-nesting approach as the right design. | Critical |
| SH-019 | Strict Conformance Boolean | Validation | `sh:conforms` is true if and only if zero validation results are produced, regardless of severity. | Formspec should define clearly whether its "conforms/valid" flag treats warnings and info the same as errors. SHACL's strict approach is notable but may be too strict for form UIs where warnings should not block submission. | High |
| SH-020 | Shapes Graph Well-Formedness | Validation | The report can include `sh:shapesGraphWellFormed` indicating whether the shapes graph itself passed syntax checking. | Formspec's static validator/linter (Python backend) already does this. Including a "definition well-formed" flag in validation output is useful for tooling and CI. | Medium |
| SH-021 | sh:and (Conjunction) | Composition | A SHACL list of shapes; each value node must conform to ALL shapes. Enables shape specialization/inheritance. | Maps to Formspec's ability to compose multiple constraints on a single field (all bind constraints are implicitly conjunctive). Could be made explicit for shape-rule composition: "this entity must satisfy rule A AND rule B." | High |
| SH-022 | sh:or (Disjunction) | Composition | A SHACL list of shapes; each value node must conform to at least one. Enables union types. | Formspec could use this for fields with alternative valid shapes (e.g., "provide either email OR phone"). FEL's `or()` function handles expression-level disjunction, but schema-level disjunction for constraint composition is distinct. | High |
| SH-023 | sh:not (Negation) | Composition | A single shape; each value node must NOT conform. Enables exclusion constraints. | Useful for Formspec exclusion rules: "if field A has value X, field B must NOT match pattern Y." FEL can express this, but a declarative negation operator at the constraint level is more analyzable. | Medium |
| SH-024 | sh:xone (Exclusive Or) | Composition | A SHACL list of shapes; each value node must conform to exactly one. Enables mutually exclusive alternatives. | Directly relevant to Formspec's common "provide fullName XOR (firstName AND lastName)" pattern. Currently expressible in FEL but a declarative xone constraint would be more discoverable and statically analyzable. | High |
| SH-025 | Nested Shape Conformance Checking | Composition | Logical operators validate referenced shapes via conformance checking (boolean). Sub-validation results are separated from the main report to prevent result explosion. | Critical for Formspec shape rules that compose sub-rules. If a shape rule references another rule, the inner results should not automatically pollute the top-level report — only the outer "this composite rule failed" result should appear, with optional detail linking. | High |
| SH-026 | Result Separation for Nested Validation | Composition | Sub-results from sh:or, sh:not, sh:node conformance checks do NOT bubble up to the main report. They may be linked via sh:detail. | Prevents result flooding in Formspec's deeply nested or composed validation rules. The UI should show one message for a composite constraint failure, not N messages for each sub-constraint. | High |
| SH-027 | Cardinality Constraints (minCount/maxCount) | Validation | Property shapes can declare minimum and maximum counts for value nodes. | Directly maps to Formspec's `required` (minCount >= 1) and repeatable group min/max cardinality. SHACL's numeric cardinality is more general — Formspec could benefit from `minCount`/`maxCount` on multi-value fields. | Critical |
| SH-028 | Value Type Constraints (class, datatype, nodeKind) | Validation | Constraints that check the type/class/datatype of value nodes. | Maps to Formspec's field type system. SHACL's datatype constraint is analogous to Formspec's type declarations. The `nodeKind` concept (IRI vs Literal vs BlankNode) has no direct parallel but the pattern of type-kind validation is relevant. | High |
| SH-029 | Value Range Constraints (min/maxInclusive/Exclusive) | Validation | Constraints on numeric/date ordering: >, >=, <, <=. | Formspec's `constraint` expressions in FEL can express these, but dedicated declarative range constraints (as SHACL provides) are more statically analyzable and map cleanly to HTML input attributes (min, max). | High |
| SH-030 | String Length Constraints (minLength/maxLength) | Validation | Constraints on string representation length. | Directly relevant to Formspec text inputs. Currently handled via FEL expressions; dedicated declarative constraints would improve static analysis and HTML attribute mapping. | High |
| SH-031 | Pattern Constraint (regex) | Validation | String representation must match a regex pattern, with optional flags parameter. | Formspec's FEL `matches()` function provides this. SHACL's pattern-as-declarative-constraint is more statically analyzable. Consider supporting a `pattern` property on bind constraints alongside FEL expressions. | High |
| SH-032 | Language Tag Constraints | Validation | Constraints on BCP47 language tags and uniqueness of language tags across values. | Limited direct relevance to form fields, but relevant if Formspec supports multilingual field values or i18n content entry. | Low |
| SH-033 | Property Pair Constraints (equals, disjoint, lessThan, lessThanOrEquals) | Validation | Cross-property comparisons: value sets of two properties must be equal, disjoint, or ordered relative to each other. | Directly maps to Formspec's cross-field validation needs. "Password must equal confirmPassword" is sh:equals. "Start date must be less than end date" is sh:lessThan. These are currently FEL expressions; declarative pair constraints would be powerful. | High |
| SH-034 | Enumeration Constraint (sh:in) | Validation | Each value node must be a member of a specified list. | Maps to Formspec's select/radio/checkbox components with predefined option lists. A declarative `in` constraint could validate that a field value is one of the allowed options. | Medium |
| SH-035 | HasValue Constraint | Validation | A specified value must be among the value nodes (existence check, not equality). | Relevant for Formspec fields that must include a specific value among their selections (e.g., "terms and conditions must be checked"). | Medium |
| SH-036 | Closed Shape Constraint | Validation | If sh:closed is true, data nodes may only have properties explicitly listed via sh:property paths (plus ignored properties). | Analogous to Formspec rejecting unexpected fields in a response. Could be useful for strict form definitions that should not accept extraneous data — important for security and data integrity. | Medium |
| SH-037 | Qualified Cardinality Constraints | Validation | At least/at most N value nodes must conform to a specific sub-shape. Supports disjointness with sibling shapes. | Relevant to Formspec repeating sections: "at least 2 of the repeated items must have status=complete." This goes beyond simple min/max count to conditional counting. | Medium |
| SH-038 | Property Paths — Predicate (Direct) | Data Model | Direct property traversal: the simplest path type, accessing an immediate property. | Formspec's dot-notation paths (e.g., `group.field`) are equivalent. Already fully supported. | Critical |
| SH-039 | Property Paths — Sequence | Data Model | Follow one property then another: `ex:parent/ex:name`. Enables traversal through nested structures. | Maps to Formspec's nested path resolution (e.g., `address.city`). Already supported for bind expressions and repeat indexing. | High |
| SH-040 | Property Paths — Alternative | Data Model | Either of two properties: `ex:father|ex:mother`. | Could be useful in Formspec for constraints that apply to "either fieldA or fieldB" without duplicating the constraint. Currently not supported but expressible via FEL. | Low |
| SH-041 | Property Paths — Inverse | Data Model | Follow a property backwards. | Limited relevance for tree-structured form data, but could be useful for referencing a parent field's value from within a repeating group. Formspec's `../` relative path syntax partially covers this. | Low |
| SH-042 | Property Paths — Transitive Closure | Data Model | Zero-or-more / one-or-more path steps. Enables validation of arbitrary-depth hierarchies. | Limited direct relevance since Formspec data is tree-structured with known depth. Could be relevant for deeply nested repeating groups, but likely over-engineering for form scenarios. | Low |
| SH-043 | Custom Constraint Components | Extension | New constraint types are declared with parameters and validators, becoming first-class citizens indistinguishable from built-in constraints. | Directly relevant to Formspec's extension registry. Custom validation components should follow the same parameter+validator pattern as built-in ones. This ensures uniform discovery, tooling, and composability. | Critical |
| SH-044 | Constraint Component Parameter Declarations | Extension | Each constraint component declares mandatory and optional parameters, with optional constraints on parameter values. | Formspec's extension registry should require parameter declarations for custom constraints. This enables schema validation of the constraints themselves, not just the data they validate. | High |
| SH-045 | SPARQL-based Inline Constraints | Expression | Shapes can embed SPARQL SELECT queries that return violation rows. Pre-bound variables provide context (focus node, path, shapes graph). | Analogous to Formspec's FEL expressions embedded in bind constraints. FEL serves the same role as SPARQL in SHACL — a query/expression language for constraint logic. The pre-bound variable pattern ($this, $PATH) maps to FEL's path references. | High |
| SH-046 | SPARQL-based Reusable Constraint Components | Extension | Reusable constraint types with SPARQL validators (ASK or SELECT). Once declared, parameters work on any shape like built-in ones. | Pattern for Formspec's custom FEL functions or custom constraint types in the extension registry. A custom constraint should be declarable with FEL-based validators and reusable across definitions. | High |
| SH-047 | ASK vs SELECT Validator Distinction | Extension | ASK validators return boolean per value node (simpler). SELECT validators return violation rows (more flexible). Separate validators for node shapes vs property shapes. | Formspec could distinguish between simple boolean constraints (like FEL expressions returning true/false) and rich constraints that produce structured violation details. Most form constraints are boolean; complex cross-field rules may need richer output. | Medium |
| SH-048 | Label Templates for Constraints | Extension | Constraint components can declare `sh:labelTemplate` for human-readable descriptions with parameter interpolation. | Formspec's validation messages could support template syntax with field-value interpolation (e.g., "Value {value} exceeds maximum of {max}"). | Medium |
| SH-049 | Prefix/Namespace Management | Expression | SHACL provides a prefix declaration mechanism so SPARQL queries can use short prefixed names. | Formspec's FEL already uses dot-notation paths which serve a similar purpose. Less relevant since FEL is a domain-specific language, not embedded SPARQL. | Low |
| SH-050 | Pre-bound Context Variables | Expression | SPARQL constraints receive pre-bound variables ($this, $shapesGraph, $currentShape, $value) providing execution context. | Maps to FEL's implicit context: the current field path, the engine state, repeat indices. Formalizing what context is available to FEL expressions (and documenting it) follows SHACL's pattern. | Medium |
| SH-051 | Shape Deactivation | Lifecycle | `sh:deactivated true` disables a shape without removing it. Enables selective override when importing shared shapes. | Useful for Formspec definition inheritance/composition: import a base definition but deactivate certain constraints for a variant form. Currently not supported but relevant for the extension/registry system. | Medium |
| SH-052 | Shape Import/Composition (owl:imports) | Composition | Shapes graphs can import other shapes graphs, enabling modular composition of validation rule sets. | Directly relevant to Formspec's extension registry and definition reuse. Forms should be composable — importing shared field groups, validation rules, or theme fragments from published packages. | High |
| SH-053 | Shape Description Properties | UI Hint | Shapes carry `sh:name`, `sh:description`, `sh:order`, `sh:group`, `sh:defaultValue` for documentation and UI generation. | Directly maps to Formspec's item labels, descriptions, display order, groups, and default values. SHACL confirms that validation schemas should also carry presentation metadata. | High |
| SH-054 | sh:order for Display Ordering | UI Hint | Numeric ordering hint for presenting shape properties in UIs. | Formspec uses array ordering (items appear in definition order). SHACL's explicit `sh:order` is useful when shapes are composed from multiple sources and need deterministic ordering. | Medium |
| SH-055 | sh:group for Logical Grouping | UI Hint | Groups related property shapes together for UI presentation, with each group having its own label and order. | Maps to Formspec's group items and page/section layout. SHACL's group-as-a-separate-entity (rather than inline nesting) enables more flexible UI composition. | Medium |
| SH-056 | sh:defaultValue | Data Model | Declares a default value for a property within a shape context. | Formspec already supports default values via `calculate` expressions and initial values. SHACL's declarative `sh:defaultValue` is simpler for static defaults. | Medium |
| SH-057 | Immutability During Validation | Lifecycle | Both shapes graph and data graph MUST remain immutable during validation. Processing is idempotent. | Formspec's validation should be a pure function of current state — no side effects, no mutation during the validation pass. This is implicit in the current design but worth making explicit in the spec. | High |
| SH-058 | Data Graph Suggests Shapes Graph | Lifecycle | Data graphs can hint at which shapes graph to use via `sh:shapesGraph`. This is a suggestion, not a requirement. | Formspec responses could optionally reference their originating definition, enabling validation tools to discover the correct schema. Useful for long-lived stored responses. | Low |
| SH-059 | Flat Result Collection with Optional Nesting | Validation | Results are a flat union of all violations, not hierarchically grouped. Optional nesting via sh:detail for sub-validations. | Formspec's ValidationReport should follow this pattern: flat by default (easy to iterate, display, and count), with optional nesting for complex composed constraints. | High |
| SH-060 | Recursion Explicitly Undefined | Validation | SHACL deliberately leaves recursive shape validation undefined, allowing implementations to choose their strategy. | Formspec should document whether circular FEL references or recursive shape rules are supported, rejected, or undefined. Currently the dependency visitor detects cycles; making this an explicit spec decision follows SHACL's precedent. | Medium |
| SH-061 | Dual-Use: Validation AND Description | Data Model | Shapes serve both as validation rules (checking conformance) and as descriptions (documenting expected structure for UI/code generation). | Core to Formspec's design: the definition IS both the validation schema and the UI description. SHACL validates this dual-use pattern as architecturally sound. | Critical |
| SH-062 | Uniform Constraint Model | Extension | Every constraint — built-in or custom — follows the same parameter+validator pattern. No special cases for built-in constraints. | Formspec's constraint system should ensure custom constraints (from the extension registry) are indistinguishable from built-in ones. The same rendering, the same validation result structure, the same tooling support. | Critical |
| SH-063 | Conformance Checking as Internal Building Block | Composition | Boolean conformance checking (does node X conform to shape Y?) is used internally by logical operators. Sub-results are separated from the main report. | Formspec's composed shape rules should use an internal boolean check for sub-rules, surfacing only the outer result. This prevents N*M result explosion in deeply composed validations. | High |
| SH-064 | Multiple Constraints per Shape (Conjunctive) | Validation | Multiple single-parameter constraint components on the same shape are implicitly conjunctive — all must be satisfied. | Matches Formspec's model where multiple bind constraints on a field are all enforced. Worth making the conjunctive semantics explicit in the spec. | High |
| SH-065 | Shape as Reusable Validation Module | Composition | A shapes graph is a portable, publishable, importable validation module that can be applied to any data. | Formspec definitions should be publishable validation modules via the extension registry. A published definition fragment (e.g., "address validation rules") should be importable into any form. | High |
| SH-066 | Explicit Invocation vs Target-based Discovery | Data Model | Shapes can be applied either via target declarations (automatic discovery) or via explicit invocation (processor told to validate node X against shape Y). Targets are ignored during explicit invocation. | Formspec supports both: bind constraints are "auto-targeted" to their field, while shape rules can target arbitrary paths. The distinction between automatic and explicit constraint application is worth formalizing. | Medium |
| SH-067 | Value Nodes Abstraction | Data Model | The concept of "value nodes" bridges shapes and data: for node shapes, the value node IS the focus node; for property shapes, value nodes are reachable via the path. Same constraint logic works uniformly. | Formspec could formalize a similar abstraction: bind constraints operate on the field's value (value node), shape rules operate on the form/entity as a whole (focus node). Uniform constraint evaluation regardless of scope. | High |
| SH-068 | Multiple Target Types per Shape | Data Model | A single shape can have multiple target declarations of different types; the effective target is their union. | Formspec shape rules already support path-based targeting with wildcards. Supporting multiple target paths per rule (union semantics) would increase expressiveness. | Medium |
| SH-069 | Unique Per-Result Constraint Component Identification | Validation | Every result identifies its `sourceConstraintComponent` — the specific type of check that failed (datatype, cardinality, pattern, etc.). | Formspec's constraint kinds in ValidationReport serve this purpose. Ensuring every built-in and custom constraint has a unique, stable identifier enables programmatic result handling. | High |
| SH-070 | Qualified Value Shape Disjointness | Validation | `sh:qualifiedValueShapesDisjoint` ensures that values counted for one qualified cardinality constraint are not also counted for a sibling constraint. | Relevant to complex Formspec repeating sections where different items must satisfy different criteria without overlap. Niche but important for advanced form scenarios. | Low |

*Note: Five features are double-categorized in the table but counted once each in the category totals. The total unique features is 70. SHACL's scope does not cover Reactivity, Repeat mechanics, or Submission — those are form-specific concerns absent from a pure constraint/validation language. This is itself a finding: Formspec's value proposition extends significantly beyond what SHACL addresses.*

### Key Takeaways

1. **Validation report structure is SHACL's strongest contribution** (27 features). Formspec's ValidationReport design should ensure parity with SHACL's structured results: mandatory constraint-kind identification, offending-value inclusion, path-based localization, and severity classification.

2. **Constraint composition operators (and/or/not/xone) are highly relevant** and currently underserved in Formspec's declarative layer. FEL can express these, but declarative composition enables static analysis, tooling discovery, and cleaner error reporting.

3. **Extension uniformity is a critical pattern**. SHACL's principle that custom constraints are first-class citizens (same parameter model, same result structure, same tooling) should be the north star for Formspec's extension registry.

4. **SHACL explicitly does NOT address** reactivity, repeat mechanics, submission workflows, or UI rendering — confirming that Formspec's multi-tier architecture (Core + Theme + Component) covers substantially more ground than a pure validation spec.

5. **The severity-does-not-affect-conformance decision** is SHACL's most controversial design point for form contexts. Formspec should intentionally diverge here: warnings and info messages should not block form submission, while errors should.
