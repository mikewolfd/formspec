# FHIR R5 Questionnaire & SDC — Feature Exploration Matrix

This matrix inventories every distinct design pattern, mechanism, and concept identified in the FHIR R5 Questionnaire base resource and the Structured Data Capture (SDC) implementation guide. Each entry is assessed for its transferability to a JSON-native declarative form specification and prioritized for later cross-referencing against the synthesized Formspec proposal.

---

| ID | Feature/Pattern | Category | Description | Formspec Relevance | Priority |
|---|---|---|---|---|---|
| FH-001 | Canonical URI identity (`url`) | Data Model | Every Questionnaire has a globally unique canonical URI that serves as its persistent identity, independent of version or hosting location. | Formspec definitions should carry a canonical `$id` or `url` field so that forms can be referenced unambiguously across systems and registries. | Critical |
| FH-002 | Business version string (`version`) | Data Model | A separate `version` field tracks the business-meaningful version of the form, decoupled from the canonical URI. | Formspec already has a version field in its definition schema; aligning it with explicit semver or date-based semantics would strengthen interop. | Critical |
| FH-003 | Version algorithm declaration (`versionAlgorithm`) | Data Model | Questionnaires explicitly declare how their version strings should be compared (semver, date-based, integer, etc.), removing ambiguity. | Useful for registries and tooling that must resolve "latest" or determine upgrade compatibility. A `versionAlgorithm` field in the registry spec would enable this. | Medium |
| FH-004 | Status lifecycle (`draft → active → retired`) | Lifecycle | Mandatory `status` field with a fixed enum governing the form's lifecycle. `unknown` is also available as a fallback. | Formspec should adopt a similar lifecycle enum. Critical for registry publishing, deprecation signaling, and tooling that needs to distinguish editable drafts from locked releases. | Critical |
| FH-005 | Version-pinned canonical references | Composition | Canonical references can include a version suffix (`url|2.0`) so that consumers lock to a specific version of a dependency. | Essential for modular composition and sub-form references. Formspec's `$ref`-style inclusion should support version pinning to prevent silent breakage. | High |
| FH-006 | Lineage tracking (`derivedFrom`) | Composition | A Questionnaire can declare that it is derived from one or more parent Questionnaires, enabling provenance and lineage graphs. | Valuable for form variant tracking (e.g., localized versions, org-specific customizations). A `derivedFrom` array of canonical URIs would support this pattern. | Medium |
| FH-007 | Definition / Response separation | Data Model | Hard split between the form schema (Questionnaire, rules, options) and the instance data (QuestionnaireResponse, answers only). The response carries no rules or option lists. | Formspec already implements this via `definition.json` and `response.json`. The principle is foundational — confirm the response schema remains a thin, data-only document. | Critical |
| FH-008 | `linkId` as join key | Data Model | Every item has a `linkId` that is unique within the Questionnaire and serves as the sole bridge between definition items and response items. | Formspec uses `bind` identifiers as the equivalent join key. Ensure uniqueness is enforced and that the linkage model is clear in the spec. | Critical |
| FH-009 | Item type taxonomy (group / display / question) | Data Model | Three fundamental item roles: `group` (structural container, no answer), `display` (instructional content, no children), and `question` (captures a typed answer). | Formspec's component spec already distinguishes layout, display, and input roles. Confirm the taxonomy is clean and that every component maps to exactly one role. | Critical |
| FH-010 | Typed question answer types | Data Model | Question items declare their answer data type from a fixed set: `boolean`, `decimal`, `integer`, `date`, `dateTime`, `time`, `string`, `text`, `url`, `coding`, `quantity`, `reference`, `attachment`. | Formspec should define a canonical set of answer types in the core spec. The current approach conflates UI component type with data type; separating them (as FHIR does) would improve mapping clarity. | High |
| FH-011 | Arbitrary item nesting | Data Model | Groups contain questions/groups; questions can also contain child questions/groups, enabling recursive "panel" structures. | Formspec supports nested items. Ensure the spec explicitly allows question items to have children (sub-questions conditionally revealed by the parent answer). | High |
| FH-012 | `repeats` flag (boolean) | Repeat | A boolean on each item indicating whether it can appear multiple times in the response. Simple, declarative. | Formspec uses `repeatMin`/`repeatMax` which is more expressive. The boolean shorthand could be a convenience alias but the min/max model is strictly superior. | High |
| FH-013 | `required` flag | Validation | Boolean declaring that at least one answer must be provided for a question or that a group must have at least one populated descendant. | Formspec supports this via `bind.required` (can be a boolean or FEL expression). Already adopted. | Critical |
| FH-014 | `readOnly` flag | Reactivity | Boolean preventing human editing of an item. Cascades from groups to all descendants. | Formspec supports this via `bind.readonly` (can be a boolean or FEL expression). Cascading behavior should be explicitly specified. | High |
| FH-015 | `maxLength` constraint | Validation | Integer limiting the maximum number of characters for string/text/url answer types. | Formspec supports this via `bind.constraint` expressions, but a first-class `maxLength` on string types would be more ergonomic and schema-enforceable. | Medium |
| FH-016 | `enableWhen` — simple conditional logic | Reactivity | Items are conditionally enabled based on comparing another item's answer to a value using operators (`exists`, `=`, `!=`, `>`, `<`, `>=`, `<=`). | Formspec uses FEL expressions for relevance (`bind.relevant`), which is strictly more powerful. However, a simplified declarative shorthand (like enableWhen) could lower the barrier for simple cases. | Medium |
| FH-017 | `enableBehavior` (all / any) | Reactivity | When multiple `enableWhen` conditions exist, `enableBehavior` specifies whether ALL (AND) or ANY (OR) must be true. | Formspec's FEL expressions handle this natively with `&&` and `||`. No separate combiner field is needed when you have a real expression language. | Low |
| FH-018 | `disabledDisplay` (hidden / protected) | UI Hint | Controls whether a disabled item is completely hidden or shown in a greyed-out/protected state. | Formspec should specify this behavior. Currently relevance (`bind.relevant = false`) implies hidden. A `disabledDisplay` hint or theme-level control would add the "protected" mode. | High |
| FH-019 | Group cascading (disabled state) | Reactivity | If a group is disabled, ALL descendants are disabled regardless of their own enableWhen/relevant conditions. | Formspec should explicitly specify this cascading rule. It is implicit in the tree evaluation model but should be normative in the processing model section. | Critical |
| FH-020 | Group cascading (readOnly state) | Reactivity | If a group is readOnly, ALL descendants are readOnly regardless of their own readOnly binding. | Same as FH-019 — specify cascading for readonly. Formspec's processing model should state that parent readonly overrides children. | High |
| FH-021 | `answerOption` — inline permitted answers | Data Model | An inline array of permitted answer values attached directly to the question item. Self-contained, no external resolution needed. | Formspec's component spec handles this via `options` arrays on select/radio/checkbox components. Confirm the core spec also supports answer constraints at the data model level (not just UI). | High |
| FH-022 | `answerValueSet` — external option sets | Data Model | A reference to an externally maintained set of permitted answers (e.g., a terminology value set). Supports reuse across forms. | Formspec's registry spec supports external option set references. Ensure the resolution mechanism is clearly specified and that offline/fallback behavior is defined. | Medium |
| FH-023 | `answerConstraint` (optionsOnly / optionsOrType / optionsOrString) | Validation | Controls whether users must pick from the list, can also enter a typed value, or can enter free text. Elegantly handles open vs. closed choice. | Formspec should adopt a similar constraint mode. Currently there's no explicit mechanism to declare "pick from list OR type your own." This is a gap. | High |
| FH-024 | `answerOption.initialSelected` | Data Model | Pre-selects specific options in the answer list as default selections. | Formspec supports `bind.calculate` for initial values and component-level `defaultValue`. Ensure initialSelected semantics work for multi-select scenarios. | Medium |
| FH-025 | `initial.value[x]` — fixed initial values | Data Model | One or more typed initial values for a question, set at form load time. Cannot coexist with `answerOption.initialSelected`. | Formspec's `bind.calculate` with `readOnly: false` approximates this. A dedicated `initial` property would be more explicit and avoid confusion with continuously calculated values. | High |
| FH-026 | `item.definition` — external schema reference | Data Model | Links a question to an ElementDefinition, allowing property inheritance (code, text, required, repeats, maxLength, answerValueSet, initial). Inline values override inherited ones. | Formspec's mapping spec partially addresses this. A first-class `definition` reference enabling property inheritance from external schemas would strengthen interop with healthcare/government data models. | Medium |
| FH-027 | Expression data type (5-element structure) | Expression | Expressions are structured objects with `description`, `name`, `language`, `expression`, and `reference` fields, not bare strings. | Formspec's FEL expressions are currently bare strings in bind properties. Wrapping them in a structured object (at least optionally) would enable descriptions and named outputs. | Medium |
| FH-028 | `variable` extension — named computed values | Expression | Compute a named intermediate value available to the defining item and all its descendants. Enables expression decomposition and reuse. | Formspec supports this via `bind.calculate` with a named bind. Ensure lexical scoping (ancestor-to-descendant visibility) is explicitly specified in the processing model. | Critical |
| FH-029 | Variable lexical scoping (ancestor → descendant) | Expression | Variables defined on an item are visible to that item and all descendants. Sibling items have disjoint scopes. Name collisions on the same node are errors; overriding on a descendant is permitted. | Formspec's bind resolution should follow these same scoping rules. Document the scoping model explicitly: same-node collision = error, descendant override = OK, sibling reuse = OK. | Critical |
| FH-030 | `initialExpression` — one-time computed initial value | Expression | An expression evaluated once (on item creation or first enable) to set the initial answer. User can edit afterward. Primarily for pre-population. | Formspec could distinguish `initial` (evaluate-once) from `calculate` (continuous). Currently `bind.calculate` is always continuous; adding an `initialExpression` mode would cover pre-population cleanly. | High |
| FH-031 | `calculatedExpression` — continuously recomputed value | Expression | An expression that re-evaluates whenever its dependencies change, continuously updating the answer. Usually paired with `readOnly`. If user edits, auto-update stops. | Formspec's `bind.calculate` is this pattern. Ensure the "user edit stops auto-update" behavior is specified and that the reload semantics (stored value vs. calculated value comparison) are documented. | Critical |
| FH-032 | Calculated field user-edit semantics | Reactivity | If a user edits a calculated field, it stops auto-updating. On reload, if the stored value matches the recalculated value, it remains calculated; if different, it is treated as user-edited. | This is a subtle but important UX rule. Formspec should specify these exact semantics in the processing model to ensure cross-implementation consistency. | High |
| FH-033 | `enableWhenExpression` — complex conditional logic | Reactivity | An expression (FHIRPath/CQL) that replaces simple `enableWhen` for cases needing nested AND/OR/NOT, cross-question comparisons, or external data. Must evaluate to boolean. | Formspec's `bind.relevant` already accepts FEL expressions, which serve this exact purpose. Already adopted — this is the primary conditional mechanism. | Critical |
| FH-034 | `answerExpression` — dynamic option lists | Expression | An expression that dynamically computes the list of allowed answers, evaluated on render or interaction. | Formspec does not currently support dynamically computed option lists via expressions. This would enable cascading dropdowns and context-sensitive choices. | High |
| FH-035 | `candidateExpression` — suggested answers from external data | Expression | An expression that retrieves candidate answers from external data for user selection. Distinct from answerExpression (candidates are suggestions, not constraints). | Maps to a population/pre-fill pattern. Formspec could support this via a `suggestedValues` expression that populates a pick-list without constraining the final answer. | Medium |
| FH-036 | `contextExpression` — related data display | Expression | An expression that retrieves related data shown alongside the question to help the user decide, without constraining the answer. | This is a UI/population pattern. Formspec could model it as a `context` display binding that shows computed reference data without affecting the answer model. | Low |
| FH-037 | `answerOptionToggleExpression` — dynamic option visibility | Expression | An expression on individual answer options that controls whether each option is shown or hidden based on current form state. | Useful for conditional option filtering (e.g., hide "N/A" after a certain answer). Formspec could support per-option `relevant` expressions on answer option arrays. | Medium |
| FH-038 | `itemPopulationContext` — group context from query | Repeat | Establishes a context resource for a group, creating one group repetition per query result. Drives dynamic repeating sections from external data. | Powerful pattern for data-driven repeating groups. Formspec's repeating sections are currently count-based; adding an expression-driven population context would enable "one row per record" patterns. | High |
| FH-039 | `targetConstraint` — custom validation invariants | Validation | A custom expression-based validation rule (beyond required/maxLength) that can reference multiple items and produce structured error messages. | Formspec's `bind.constraint` and shape-level validation rules serve this purpose. Ensure the validation model supports both field-level and cross-field constraint expressions with custom messages. | Critical |
| FH-040 | Three expression languages (FHIRPath, CQL, x-fhir-query) | Expression | SDC supports multiple expression languages, declared via the `language` field, with FHIRPath as the default. | Formspec uses FEL as its single expression language. The pluggable language concept is interesting but likely unnecessary for a JSON-native standard — one well-designed language is simpler. | Low |
| FH-041 | Expression description field | Expression | Expressions carry a human-readable `description` explaining their intent, separate from the expression text itself. | Valuable for maintainability and debugging. Formspec could optionally support `description` annotations on bind expressions, surfaced in tooling and validation reports. | Medium |
| FH-042 | Named expression results (`%name` variables) | Expression | An expression's `name` field makes its result available as a `%name` variable to descendant expressions, enabling composition. | Formspec's bind names serve a similar purpose. Ensure the variable reference syntax and resolution order are clearly specified. | High |
| FH-043 | Reserved context variables (`%resource`, `%context`, `%questionnaire`, `%qitem`) | Expression | SDC defines reserved variable names that always resolve to specific context objects (the response root, the current item context, the definition, etc.). | Formspec's FEL should define equivalent reserved names (e.g., `$response`, `$definition`, `$currentItem`). Some of these may already exist; ensure they are documented. | High |
| FH-044 | Expressions defined in definition, evaluated against response | Expression | Expression text lives in the Questionnaire (definition) but evaluates against QuestionnaireResponse (instance) data. Clean separation of logic from data. | Formspec follows this same model — FEL expressions are in the definition and evaluate against runtime values. Confirm this is explicit in the spec. | Critical |
| FH-045 | Dependency-ordered evaluation | Reactivity | Expressions are evaluated in document order (depth-first), with iterative re-evaluation until stable. Circular dependencies must be detected and handled gracefully. | Formspec's engine uses signal-based reactivity which handles this automatically. The spec should still define the normative evaluation order and circular dependency behavior for non-signal implementations. | Critical |
| FH-046 | Circular dependency detection | Reactivity | Systems must detect circular dependencies and infinite loops in expressions, failing gracefully rather than diverging. | Formspec's DependencyVisitor already extracts dependency graphs. The spec should mandate that implementations detect and report cycles rather than silently producing wrong results or hanging. | High |
| FH-047 | `subQuestionnaire` — section-level modular composition | Composition | A display item with a `subQuestionnaire` extension referencing another Questionnaire's canonical URL. During assembly, the display item is replaced by the referenced form's root items. | Formspec should support a composition mechanism where a placeholder item is replaced by the contents of a referenced sub-form. This is the primary modular reuse pattern. | Critical |
| FH-048 | `$assemble` operation — two-phase publish workflow | Composition | Modular forms are authored with references, then assembled into a self-contained form at publish time. Fill-time systems only see the assembled result. | Formspec should define an assembly step (CLI tool or build operation) that resolves all sub-form references into a single self-contained definition. Keeps the runtime simple. | High |
| FH-049 | Assembly expectation metadata | Composition | An extension declares whether a Questionnaire needs assembly before use, can be used as a sub-module, or both. Codes: `assemble-root`, `assemble-child`, etc. | Formspec should include metadata indicating whether a definition is a root form, a reusable fragment, or both. Helps tooling and registries classify and validate forms. | Medium |
| FH-050 | `linkIdPrefix` — namespace isolation for composed forms | Composition | A variable prepended to all linkIds from a sub-questionnaire during assembly to prevent ID collisions. Also applied to expression references and enableWhen targets. | Essential for safe composition. Formspec's assembly mechanism must perform equivalent ID prefixing to prevent collisions when merging sub-forms. | Critical |
| FH-051 | Questionnaire Library Referencing (question-level reuse) | Composition | `item.definition` using format `[Questionnaire URL]#[linkId]` to reference a specific item and its descendants from another Questionnaire. | Formspec could support fragment references (`url#itemId`) for fine-grained reuse of individual questions or small groups from a library form. | Medium |
| FH-052 | Extension propagation rules during assembly | Composition | Different extension types have different propagation behaviors during assembly: some propagate to root, some to the containing item, some are dropped. | Formspec's assembly process needs clear rules for how metadata, variables, and validation rules from sub-forms are merged into the parent. Document propagation semantics. | High |
| FH-053 | Contained resource merging during assembly | Composition | Resources contained within sub-questionnaires are merged into the parent's contained resources during assembly. | If Formspec definitions can embed resources (e.g., option sets, external data), assembly must merge these correctly with deduplication. | Medium |
| FH-054 | Metadata alignment requirements for assembly | Composition | Security labels, language, and endpoint metadata must match between parent and child questionnaires during assembly. | Formspec assembly should validate that locale, access control, and other metadata are compatible between composed forms. | Medium |
| FH-055 | Recursive sub-questionnaire resolution | Composition | Sub-questionnaires can reference further sub-questionnaires, resolved recursively during assembly. | Formspec's assembly must handle recursive resolution with cycle detection to prevent infinite loops. | High |
| FH-056 | Graceful degradation for unresolved sub-forms | Composition | The display item used as a sub-questionnaire placeholder should carry meaningful fallback text in case assembly fails. | Formspec placeholder items should have fallback content so that unresolved references degrade gracefully in tooling and preview scenarios. | Medium |
| FH-057 | `$populate` operation — server-side pre-population | Submission | A server operation that generates a pre-filled QuestionnaireResponse from existing data before the user sees the form. | Formspec could define a population API contract where a server returns a partial response pre-filled from external data sources. Separates population logic from the form definition. | Medium |
| FH-058 | Continuous/interactive population | Submission | The form filler handles population internally, using answers to earlier questions to drive population of later questions during the session. | Formspec's `bind.calculate` and `initialExpression` patterns already support this. Ensure the spec distinguishes server-side pre-population from client-side interactive population. | High |
| FH-059 | Three population modes (automated / choice / context) | Submission | Automated: system provides exact answer. Choice: system provides candidates, user picks. Context: system shows related data to inform the user's decision. | These three modes capture real population patterns. Formspec could formalize them as population strategy types in the mapping or population spec. | Medium |
| FH-060 | Observation-based population (code + time window) | Submission | Items with a `code` (e.g., LOINC) and a time window extension are auto-populated from the most recent matching Observation. Bidirectional with extraction. | Domain-specific to healthcare, but the pattern (lookup by code within a time window) generalizes to any system with coded data and temporal relevance. Low priority for a generic form standard. | Low |
| FH-061 | Expression-based population | Submission | Uses `launchContext`, `variable`, `initialExpression`, `candidateExpression`, and `contextExpression` to populate from external data via expressions. Most flexible approach. | Formspec's expression-based bindings already support this. The key insight is formalizing `launchContext` as declared runtime inputs. | High |
| FH-062 | `launchContext` — declared runtime inputs | Expression | A named set of external context objects (patient, user, encounter, etc.) available to expressions at runtime. Serves as an interface contract between the form and its hosting environment. | Formspec should define a `launchContext` or `externalContext` mechanism declaring what runtime data the form expects. Makes the form's external dependencies explicit and testable. | High |
| FH-063 | StructureMap-based population (full transform) | Submission | Uses FHIR Mapping Language to transform query results into a complete pre-filled response. Most powerful but most complex; all-or-nothing execution. | The pattern of a declarative transform from source data to form response is powerful. Formspec's mapping spec already addresses bidirectional transforms; ensure it can serve as a population mechanism. | Low |
| FH-064 | Re-population semantics (don't clobber user edits) | Lifecycle | Re-population only occurs at user request, never silently reverts edits, highlights changes, and flags changed fields for review. | Critical UX rule. Formspec's processing model should specify that re-population respects user edits, with explicit change-detection and user-review requirements. | High |
| FH-065 | Population of disabled items | Reactivity | Disabled items should still be populated (their data may be needed if enabled later), but disabled answers must not be displayed to the user. | Formspec should specify that irrelevant items retain their values in the data model even when hidden from the UI. Aligns with current engine behavior but needs normative spec text. | High |
| FH-066 | Capture context into hidden answers for stability | Submission | External dependencies (launch context, query results) may change between sessions; best practice is to capture needed context into hidden answer fields for stable validation. | Formspec could recommend this pattern: forms that depend on ephemeral context should snapshot it into hidden fields so that saved responses remain self-contained and validatable. | Medium |
| FH-067 | Layered complexity model (base → expression → transform) | Data Model | FHIR structures complexity in layers: base Questionnaire has simple rules; SDC expressions are an opt-in advanced layer; StructureMap transforms are expert-level. Most implementations need only the base. | Formspec already follows this with Core (data & logic) → Theme (presentation) → Component (interaction). Validate that each tier is independently useful without requiring the next. | Critical |
| FH-068 | Expressions as extensions (not core fields) | Data Model | SDC adds expression capabilities via extensions rather than core fields, acknowledging that most systems don't need them and keeping the base resource simple. | Formspec integrates expressions (FEL) into the core spec. This is a deliberate design choice — FEL is lightweight enough to be core. However, advanced features (population, transforms) should remain in separate spec tiers. | Medium |
| FH-069 | Separation of data flow from rendering | Expression | Expressions define data flow and computed values; the rendering layer just reads and displays those values. No rendering logic in expressions. | Formspec should maintain this separation. FEL expressions compute values and conditions; component renderers consume the results. No UI logic in FEL, no data logic in components. | Critical |
| FH-070 | `linkId` uniqueness across assembled content | Composition | After assembly, all linkIds must be unique across the entire assembled Questionnaire. Enforced by linkIdPrefix namespacing. | Formspec must enforce bind ID uniqueness in assembled forms. The assembly tool should validate this and fail with clear errors on collision. | Critical |
| FH-071 | Answer type vs. UI component separation | Data Model | FHIR separates the data type of an answer (integer, date, coding) from the UI widget used to capture it. The same data type can be rendered by different widgets. | Formspec's architecture should cleanly separate data types (core spec) from components (component spec). A `date` field could render as a date-picker, text input, or calendar — the data type doesn't dictate the widget. | High |
| FH-072 | Multiple initial values for repeating items | Repeat | If an item repeats, `initial` can contain multiple values, each becoming a separate initial answer instance. | Formspec should support array-valued initial values for repeating items, with each element seeding one repetition. | Medium |
| FH-073 | Mutual exclusivity constraints (initial vs initialSelected) | Data Model | `initial.value[x]` and `answerOption.initialSelected` cannot coexist on the same item, preventing ambiguity about default values. | Formspec should define similar mutual exclusivity rules where overlapping features could create ambiguity. Schema-level constraints (`oneOf`) can enforce this. | Medium |
| FH-074 | Inline override of inherited properties | Data Model | When `item.definition` references an external schema, inline Questionnaire values override the inherited values. Explicit > inherited. | Formspec should follow the same "explicit overrides inherited" principle for any property inheritance mechanism. | Medium |
| FH-075 | External expression references | Expression | The expression data type includes a `reference` field for externally maintained expressions, enabling shared expression libraries. | Formspec could support expression library references for organizations maintaining shared business rules across many forms. Low priority initially. | Low |
| FH-076 | Event-based listener approach for expression evaluation | Reactivity | SDC recommends event-based listeners for efficient expression re-evaluation rather than polling or full re-evaluation on every change. | Formspec's engine already uses Preact Signals (reactive primitives). The spec should remain implementation-agnostic but recommend reactive/event-driven approaches for efficiency. | Medium |
| FH-077 | Access control for population data | Submission | Population must not expose data the user wouldn't normally be authorized to see. Population mechanisms must respect access control boundaries. | If Formspec supports server-side population, the spec or guidance should address authorization — populated data must respect the user's access level. | Medium |
| FH-078 | Hierarchical observation period inheritance | Submission | An `observationLinkPeriod` on a parent group applies to all coded descendants unless overridden, reducing repetitive configuration. | The pattern of hierarchical property inheritance (parent → descendants with override) is useful for any configuration that tends to be uniform within groups. | Low |
| FH-079 | Bidirectional population/extraction symmetry | Submission | The same metadata (codes, expressions) that drives population also drives extraction — answering "where does this data come from?" automatically answers "where does it go?" | Formspec's mapping spec supports bidirectional transforms. Ensure the population and extraction paths use the same metadata so they stay in sync. | High |
| FH-080 | `candidateExpression` on repeating groups | Repeat | When applied to a repeating group, a candidate expression creates one group instance per user-selected candidate, enabling "select which records to include" patterns. | This combines population with dynamic repetition. Formspec could support expression-driven repetition where the number of instances is determined by a query result or user selection. | Medium |

### Key Observations

1. **Data Model dominance**: Nearly a quarter of all features relate to the core data model (identity, types, answer constraints, definition/response separation). This is the foundation — getting it right unlocks everything else.

2. **Reactivity and Expression are tightly coupled**: Together they account for 24 features. The interplay between named variables, scoping rules, evaluation order, and cascading behavior is the most specification-dense area.

3. **Composition is underserved in most form standards**: FHIR/SDC's modular forms machinery (12 features) represents hard-won lessons about reuse at scale. Formspec should invest here early — retrofitting composition is painful.

4. **Submission/Population is a separate concern**: 10 features relate to how forms are pre-filled and how responses are extracted. This is correctly a separate spec tier (mapping), but the interfaces between population and the core expression model must be clean.

5. **The "already adopted" set is large**: Many Critical features (definition/response split, named variables, calculated expressions, expression-based relevance, dependency tracking) are already implemented in Formspec. The matrix highlights gaps rather than greenfield work.
