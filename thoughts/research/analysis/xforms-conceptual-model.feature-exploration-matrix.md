# XForms 1.1 Conceptual Model — Feature Exploration Matrix

This matrix is a comprehensive inventory of every distinct idea, mechanism, design pattern, inheritance rule, default value, and edge case described in the XForms 1.1 Conceptual Model synthesis. Each entry is tagged by category and assessed for relevance to Formspec, a JSON-native declarative form specification. The goal is to enable systematic cross-referencing when evaluating what Formspec has already adopted, what it should adopt, and what it should consciously reject.

| ID | Feature/Pattern | Category | Description | Formspec Relevance | Priority |
|----|-----------------|----------|-------------|-------------------|----------|
| XCM-001 | Model Item Properties (MIPs) as bind metadata | Data Model | MIPs are metadata (type, readonly, required, relevant, calculate, constraint, p3ptype) attached to instance data nodes via `<bind>` elements, not embedded in UI markup. Each bind selects a node-set and applies properties uniformly. | Formspec already adopts this: `bind` objects in the definition attach `readonly`, `required`, `relevant`, `calculate`, and `constraint` to data paths. The separation of data-level properties from UI is a core Formspec tenet. | Critical |
| XCM-002 | `type` MIP — schema datatype association | Data Model | The `type` MIP associates an XML Schema datatype (e.g., `xsd:decimal`) with a node's string value. It is a literal (non-computed) property that contributes to validation. Default is `xsd:string`. | Formspec uses `datatype` on items (e.g., `"string"`, `"number"`, `"date"`, `"boolean"`) to associate type constraints. JSON-native typing is simpler than XSD QNames but serves the same purpose. Already adopted. | Critical |
| XCM-003 | `readonly` MIP with inheritance and calculate default | Reactivity | `readonly` is a computed boolean expression. Defaults to `false()`, but automatically becomes `true()` when `calculate` is set. Inherits via OR up the ancestor chain — if any ancestor is readonly, descendants are too. | Formspec supports `readonly` as a FEL expression in binds. The auto-readonly-when-calculated behavior is worth evaluating — currently Formspec does not document an explicit default when `calculate` is present. Ancestor OR-inheritance is not currently specified in Formspec. | High |
| XCM-004 | `required` MIP — non-empty enforcement | Validation | `required` is a computed boolean expression (default `false()`). When true, the node's value must be non-empty (string-length > 0) for submission to succeed. Does not inherit. | Formspec already supports `required` as a FEL expression in binds. The semantics match: a required field must have a non-empty value. No inheritance, consistent with XForms. | Critical |
| XCM-005 | `relevant` MIP with AND-inheritance | Reactivity | `relevant` is a computed boolean (default `true()`). When false, the node and all descendants (via AND-inheritance) are non-relevant. Affects both UI visibility and data submission. | Formspec supports `relevant` as a FEL expression. AND-inheritance down the tree is a key subtlety — if a parent group is non-relevant, all children should be too. Formspec should ensure this cascading behavior is normatively specified. | Critical |
| XCM-006 | `calculate` MIP — derived/computed values | Reactivity | `calculate` is any XPath expression whose result is converted to string and stored as the node's value. Creates derived data that updates reactively. | Formspec's `calculate` in binds directly mirrors this. FEL expressions replace XPath but serve the same role. Already adopted. | Critical |
| XCM-007 | `constraint` MIP — custom validity predicate | Validation | `constraint` is a computed boolean expression (default `true()`). It provides validation beyond type checking, and can reference other nodes (e.g., cross-field date comparisons). | Formspec's `constraint` in binds is the direct equivalent. FEL expressions can reference other fields for cross-field validation. Already adopted. | Critical |
| XCM-008 | `p3ptype` MIP — privacy policy metadata | Data Model | A literal string associating a P3P privacy data category with a node. Purely decorative/policy metadata with no computational effect. | Low relevance for Formspec's current scope. However, the pattern of attaching non-functional metadata to data nodes is interesting for future GDPR/privacy annotation needs. Could be addressed via extension metadata. | Low |
| XCM-009 | Duplicate MIP assignment as fatal error | Validation | Assigning the same MIP to the same node from two different `<bind>` elements raises a fatal `xforms-binding-exception`. This prevents ambiguity. | Formspec's bind model currently uses a single bind per path. If multiple binds can target overlapping paths (e.g., via wildcards), Formspec should define conflict resolution or error semantics. Worth specifying as a normative constraint. | High |
| XCM-010 | Bind processing in document order | Data Model | Binds are processed in document order within the model. This determines precedence and scoping. | In Formspec, binds are an array in the definition JSON. Array order is the natural equivalent of document order and should be normatively meaningful for evaluation sequencing. | Medium |
| XCM-011 | Nested bind scoping | Data Model | Nested `<bind>` elements scope their `nodeset` relative to the parent bind's node-set, enabling hierarchical targeting. | Formspec uses flat bind arrays with explicit path expressions rather than nesting. The flat approach is simpler for JSON but loses the implicit scoping. FEL path expressions compensate. Worth documenting the trade-off. | Medium |
| XCM-012 | Reactive dependency graph — vertices and edges | Reactivity | Each computed MIP expression bound to a specific instance node forms a vertex. Edges represent data references: if vertex C references node N, then N lists C as a dependent. | Formspec engine uses Preact signals to build a reactive graph. The conceptual model is the same — field values are nodes, computed expressions create edges. The signal-based approach handles this implicitly. | Critical |
| XCM-013 | Self-reference exclusion in calculate | Reactivity | Self-references in `calculate` expressions are explicitly excluded from creating dependency edges. This prevents trivial cycles and enables "default value" patterns where a calculate can read its own current value. | Formspec should evaluate whether self-referential calculate expressions are useful (e.g., "if empty, default to X"). If supported, the engine must exclude self-edges from the dependency graph. | Medium |
| XCM-014 | Circular dependency detection | Reactivity | Circular dependencies (A depends on B depends on A) raise `xforms-compute-exception`. Detected when topological sort finds no zero-in-degree vertex. | Formspec engine should detect and report circular dependencies. This is a critical correctness guarantee. The engine should surface clear error messages identifying the cycle. | Critical |
| XCM-015 | Rebuild phase — structural re-evaluation | Lifecycle | Rebuild re-evaluates all bind `nodeset` expressions, re-applies MIP declarations, reconstructs the dependency graph, and marks all computed nodes as dirty. Triggered by structural changes (insert/delete). | Formspec engine needs an equivalent when repeating group instances are added/removed. The engine must rebuild dependency tracking for new/removed paths. Signal-based reactivity may handle this implicitly, but the concept should be specified. | High |
| XCM-016 | Recalculate phase — topological evaluation | Lifecycle | Recalculate takes a change list, computes the pertinent dependency subgraph (only reachable vertices), topologically sorts it, and evaluates each vertex exactly once in dependency order. | Formspec engine's signal-based reactivity handles this. Preact signals propagate changes through the graph. The topological ordering is implicit in signal evaluation. The "pertinent subgraph" optimization (only recalculating what's affected) is important for performance. | Critical |
| XCM-017 | Revalidate phase — validity checking | Lifecycle | After recalculation, every instance node is checked: constraint must be true, required must be satisfied (non-empty if required), and type must match. Nodes whose validity state changed are marked for notification. | Formspec engine already performs validation after state changes. The key insight is that validation is a distinct phase that runs after all calculations settle, not interleaved with them. | Critical |
| XCM-018 | Refresh phase — UI synchronization | Lifecycle | Refresh re-evaluates all UI binding expressions, propagates values/validity/MIP states to form controls, dispatches notification events, and updates repeat structures. | Formspec web component's reactive rendering serves this role. Signal subscriptions in the DOM automatically trigger re-renders. The notification events (value-changed, valid/invalid, etc.) map to custom events on the web component. | High |
| XCM-019 | Four-phase cycle ordering | Lifecycle | The phases must execute in strict order: rebuild, recalculate, revalidate, refresh. Each phase has clear responsibilities and prerequisites. | Formspec should document its processing model with explicit phase ordering. Even with signal-based reactivity, the conceptual phases exist: structural update, value propagation, validation, DOM update. | High |
| XCM-020 | Deferred update processing | Lifecycle | During action processing, rebuild/recalculate/revalidate/refresh are deferred — flags are set but processing happens at the end of the action block or at submission time. | Formspec engine should batch updates when multiple values change in rapid succession (e.g., programmatic bulk setValue calls). Signal batching in Preact handles this, but the concept should be documented. | Medium |
| XCM-021 | Pertinent dependency subgraph optimization | Reactivity | Recalculate does not re-evaluate all expressions — only those reachable from the changed nodes. This is a critical performance optimization for large forms. | Preact signals naturally implement this — only signals that depend on changed values re-evaluate. Worth documenting as a performance guarantee in the Formspec spec. | High |
| XCM-022 | Non-relevant node UI hiding | UI Hint | When `relevant` is false, bound form controls are hidden, removed from tab order, cannot receive focus. Groups and containers become invisible with all children. | Formspec's web component should hide non-relevant fields and remove them from tab order. The `relevant` bind property already drives this, but the precise UI behavior (tab order, focus, ARIA states) should be specified. | High |
| XCM-023 | Non-relevant data pruning at submission | Submission | By default, non-relevant nodes are pruned from submission data. If pruning removes all data, a submission error occurs. | Formspec's `getResponse()` should exclude non-relevant fields by default. This is a key data integrity feature — forms should not submit hidden conditional data. Already partially adopted. | Critical |
| XCM-024 | Configurable relevance pruning on submission | Submission | The `relevant` attribute on `<submission>` controls whether pruning occurs. Default is `true` for serialized submissions. Can be disabled. | Formspec could allow the submission configuration to opt out of relevance pruning. Useful for audit/debugging scenarios where you want the complete data model. | Medium |
| XCM-025 | Non-relevant nodes remain programmatically accessible | Reactivity | Non-relevance only affects UI binding. Actions like `setvalue` and submission elements can still operate on non-relevant nodes. Calculate expressions continue to run. | Formspec should clarify that non-relevant fields still participate in calculations and can be set programmatically. Only UI visibility and default submission behavior are affected. | High |
| XCM-026 | Dynamic relevance toggling | Reactivity | Relevance expressions are recalculated reactively. A node can toggle between relevant and non-relevant as upstream data changes, causing UI sections to appear/disappear dynamically. | Formspec already supports this via reactive FEL expressions in the `relevant` bind. The web component re-renders accordingly. Core feature. | Critical |
| XCM-027 | Repeat bound to node-set (collection binding) | Repeat | `<repeat nodeset="...">` binds to a collection of nodes. For each node, a repeat item is created containing a cloned UI template. | Formspec's repeating groups bind to a named group with instances. The conceptual model is equivalent: a template is instantiated per collection item. | Critical |
| XCM-028 | Repeat index (1-based current item pointer) | Repeat | Each repeat maintains a 1-based index pointing to the "current" item. Accessed via `index('repeatId')` function. Manipulated via `setindex` action. Default `startindex` is 1. | Formspec could benefit from a current-item concept for repeating groups, enabling "selected row" patterns in table/list UIs. Not currently specified but useful for detail/master views. | Medium |
| XCM-029 | Index maintenance on insert | Repeat | After insert, the repeat index moves to the last newly inserted item. | If Formspec adopts a repeat index concept, this insertion behavior should be specified — it provides natural UX where the user's focus follows the new item. | Medium |
| XCM-030 | Index maintenance on delete | Repeat | On delete: if indexed item still exists, index tracks it; if deleted and collection non-empty, index = min(oldIndex, newSize); if collection empty, index = 0. | Precise index maintenance rules prevent dangling references and provide predictable behavior. Important if Formspec adopts repeat indexes. | Medium |
| XCM-031 | Automatic context scoping within repeats | Repeat | Each repeat item's inner elements get an evaluation context where the context node is the collection node for that item, position is the item's index, and size is the collection size. | Formspec already supports this: within a repeating group, FEL expressions resolve paths relative to the current instance (e.g., `items[0].price` scoped to the current row). | Critical |
| XCM-032 | Nested repeat context resolution | Repeat | Inner repeat nodesets are evaluated relative to each outer repeat item's context node. Context narrows progressively with nesting depth. | Formspec supports nested repeating groups. The path resolution must correctly scope inner paths relative to outer instances. Already addressed in engine path resolution. | High |
| XCM-033 | Focus-driven index update | Repeat | Focusing a control inside a repeat automatically updates that repeat's index, and recursively updates outer repeat indexes. | This is a UI-level behavior that Formspec's web component could implement. When a user clicks into a repeat row, the "current" index should update. Relevant for accessibility and keyboard navigation. | Low |
| XCM-034 | Alternative repeat syntax (attribute-based) | Repeat | `repeat-nodeset` and `repeat-bind` attributes can be placed on host-language elements for integration where `<xforms:repeat>` is not allowed in the content model. | Not directly relevant to Formspec since JSON definitions don't have content model constraints. However, the principle of flexible binding syntax (items vs. groups) is already in Formspec. | Low |
| XCM-035 | Multiple named instances | Data Model | A single model can contain multiple `<instance id="...">` elements, each holding a separate data document. Cross-instance references are supported. | Formspec could support auxiliary data sources (lookup tables, reference data) accessible via FEL. Currently Formspec has a single data model. Multiple named instances would enable patterns like `lookup('prices', itemId)`. | High |
| XCM-036 | Default instance convention | Data Model | The first `<instance>` child (in document order) is the default instance. Expressions without explicit instance references resolve against it. | Formspec's primary data model is the implicit default. If multiple data sources are added, the primary response data should be the default, with auxiliary data requiring explicit naming. | Medium |
| XCM-037 | `instance()` function for cross-instance access | Expression | `instance('id')` returns the root element of a named instance within the same model. Enables cross-instance references like `instance('prices')/catalog/item[@id = ...]`. | If Formspec adds auxiliary data, a FEL function like `lookup()` or `data()` would serve the same purpose. The function-based access pattern is cleaner than implicit scoping. | High |
| XCM-038 | Cross-instance dependency tracking | Reactivity | `instance()` calls in computed expressions create dynamic dependencies. May require manual rebuild if the referenced instance structure changes. | If Formspec supports auxiliary data, the engine must track dependencies on external data. Changes to lookup data should trigger recalculation of expressions that reference it. | Medium |
| XCM-039 | Model-View-Controller separation | Composition | The model (data + binds + submission config) is fully functional without any UI. You can rebuild, recalculate, revalidate, and submit purely from the model. Multiple UIs can bind to the same model. | Formspec's core architecture already embodies this: FormEngine operates independently of any renderer. The definition JSON is the model; the web component is one possible view. | Critical |
| XCM-040 | Intent-based controls | UI Hint | Form controls are intent-based ("select one", not "radio button"). Rendering is device-dependent. The spec describes purpose, not presentation. | Formspec's component spec defines semantic component types (e.g., `single-select`) that renderers map to concrete widgets. This is a direct adoption of the intent-based pattern. | Critical |
| XCM-041 | Co-located label metadata | UI Hint | Controls always carry their own `<label>` — metadata is co-located with the control, not separate. | Formspec items include `label` as a required property on each item. Labels are co-located with the field definition. Already adopted. | High |
| XCM-042 | Declarative event-driven actions | Lifecycle | XForms actions (setvalue, insert, delete, send, toggle) are declarative and event-driven, attached to UI or model elements. No imperative scripting required. | Formspec uses FEL expressions for reactive behavior rather than explicit action declarations. The `calculate` mechanism covers `setvalue` use cases. Insert/delete for repeats are engine API methods rather than declarative actions. Worth evaluating whether declarative action triggers would add value. | Medium |
| XCM-043 | Multiple models per document | Composition | Multiple independent `<model>` elements can coexist in a single document, each with its own instances, binds, and submissions. | Formspec currently assumes one definition per form. Multi-model composition could enable modular form assembly (e.g., embedding a reusable address sub-form). Worth exploring for composition/modularity. | Medium |
| XCM-044 | Default expression context — root of default instance | Expression | When no ancestor binding element exists, the expression context defaults to the root element of the default instance of the default model, with position 1 and size 1. | Formspec's FEL evaluates unqualified paths against the root of the form's data model. Same concept, simpler (no model selection needed with single-model forms). | High |
| XCM-045 | Scoped resolution via nearest ancestor binding element | Expression | Non-outermost binding elements inherit context from their nearest ancestor binding element. Single-node bindings provide a fixed context; node-set bindings provide per-item contexts. | Formspec's path resolution within groups and repeating groups follows this pattern. Paths in nested items resolve relative to their containing group. Core to how FEL works within repeats. | Critical |
| XCM-046 | Context narrowing through nesting depth | Expression | Expression context narrows progressively as elements are nested deeper in the binding tree. Each level of nesting constrains the evaluation scope further. | Formspec's nested group/repeat path resolution implements progressive narrowing. Deeply nested fields resolve against their immediate container's context. | High |
| XCM-047 | Model switching via explicit attribute | Expression | If an element specifies `model="otherModel"`, the context resets to the root element of that model's default instance, breaking the ancestor chain. | Not directly applicable to single-model Formspec. If multi-model composition is added, an explicit data-source attribute on items or groups would serve the same purpose. | Low |
| XCM-048 | Per-node MIP expression evaluation | Expression | Computed MIP expressions on a bind are evaluated once per node in the bind's node-set, with context node/position/size set per-node. Same mechanism as repeat dynamic context. | Formspec's wildcard binds (e.g., `items[*].price`) must evaluate the bind expression once per matching instance. The engine must set the correct context for each evaluation. | High |
| XCM-049 | Special per-element context rules | Expression | Some attributes (e.g., `value` on `setvalue`, `at` on `insert`/`delete`) have custom context rules documented per-element, overriding the default scoping. | Formspec should document any cases where FEL evaluation context differs from the default scoping rules. Currently, `calculate` and `constraint` expressions have an implicit "current node" context that should be explicitly specified. | Medium |
| XCM-050 | Submission pipeline — flush deferred updates | Submission | Before submission, any pending rebuild and recalculate operations are flushed to ensure data is current. | Formspec's `getResponse()` should guarantee all pending calculations are settled before extracting data. Signal-based systems handle this naturally, but it should be documented. | High |
| XCM-051 | Submission data selection via ref/bind | Submission | The `<submission>` element can select a subtree of the instance for submission, rather than always submitting everything. Default is the entire instance. | Formspec could allow response extraction to target specific paths or groups, enabling partial form submission. `getResponse()` could accept a path filter. | Medium |
| XCM-052 | Submission validation gate | Submission | If `validate="true"` (default for serialized submissions), all selected nodes are validated before submission. Any invalid node blocks submission. | Formspec's `getValidationReport()` serves this role. The form consumer should check the report before accepting the response. Could be made more explicit with a `canSubmit()` convenience method. | High |
| XCM-053 | Multiple serialization formats | Submission | XForms supports multiple serialization formats (XML, URL-encoded, multipart). Each format has different fidelity for preserving data structure. | Formspec's Mapping spec addresses serialization to multiple formats (JSON, XML, CSV). The multi-format approach is already adopted, with JSON as the primary format. | High |
| XCM-054 | Relevance pruning before validation | Submission | In the submission pipeline, relevance pruning happens before validation. Non-relevant data is removed first, then remaining data is validated. | The ordering matters: validating after pruning means non-relevant fields with invalid values do not block submission. Formspec should specify this ordering explicitly. | High |
| XCM-055 | No-data error on complete pruning | Submission | If relevance pruning removes ALL data from the submission, a `no-data` error occurs rather than silently submitting empty data. | Formspec should consider whether a completely empty response (all fields non-relevant) is an error condition. Depends on use case — progressive disclosure forms might legitimately have all-hidden states. | Medium |
| XCM-056 | Custom serialization hook | Submission | The `xforms-submit-serialize` event allows handlers to replace the serialized body with arbitrary content, enabling custom formats. | Formspec's extensible adapter pattern in the Mapping spec serves a similar purpose. Custom adapters can produce any output format. | Medium |
| XCM-057 | Response handling modes (replace attribute) | Submission | Submission responses can replace the entire document, replace instance data, replace text content, or be discarded. `replace="instance"` enables server round-trips that update form data. | Formspec could support a "reload from response" pattern where server responses update the form's data model. Useful for server-side validation feedback or wizard flows. | Medium |
| XCM-058 | Sync vs async submission | Submission | `mode="synchronous"` blocks the UI; `mode="asynchronous"` (default) allows background submission with deferred response handling. | Formspec delegates submission to the consuming application (the engine produces data, not HTTP requests). However, documenting async patterns for form consumers would be valuable. | Low |
| XCM-059 | `targetref` for partial instance replacement | Submission | When `replace="instance"`, `targetref` can specify a subtree to replace, enabling surgical updates rather than full instance replacement. | Useful pattern for Formspec if "load from server" is supported: update only a specific part of the form data (e.g., populate an address from a ZIP code lookup). | Medium |
| XCM-060 | Readonly inheritance via OR semantics | Reactivity | If any ancestor in the data tree is readonly, all descendants are effectively readonly via OR-inheritance. This cascades automatically. | Formspec should specify whether readonly cascades. A readonly group should make all contained fields readonly. This is important for disabling entire form sections. | High |
| XCM-061 | Calculated nodes default to readonly | Reactivity | When a `calculate` MIP is present, `readonly` defaults to `true()` rather than `false()`. This prevents user mutation of computed values. | Formspec should adopt this convention: fields with a `calculate` bind should be implicitly readonly unless explicitly overridden. Prevents user confusion from editing computed values that get overwritten. | High |
| XCM-062 | Validity = constraint AND required AND type | Validation | A node is valid if and only if: constraint expression is true, AND (required is false OR value is non-empty), AND the value satisfies the declared type. All three must hold. | Formspec's validation model should explicitly define the composite validity formula. The three-part check (type + required + constraint) should be documented as the normative validity definition. | Critical |
| XCM-063 | Topological sort for evaluation ordering | Reactivity | The dependency graph is topologically sorted to determine a safe evaluation order where each expression is computed after all its dependencies. | Formspec engine must guarantee evaluation order respects dependencies. Signal-based reactivity handles this, but it is worth specifying as a normative requirement for any conforming engine. | Critical |
| XCM-064 | Change list propagation optimization | Reactivity | Only vertices reachable from the change list are recalculated, not the entire graph. This is the "pertinent dependency subgraph" optimization. | Critical for performance on large forms. Preact signals implement this naturally. Should be documented as an expected optimization in the Formspec processing model. | High |
| XCM-065 | First-run full evaluation | Lifecycle | On form load (first run), ALL vertices are in the pertinent subgraph — every expression is evaluated once to establish initial state. | Formspec engine must perform a complete initial evaluation when `setDefinition()` is called. All binds, calculations, validations, and relevance expressions must be evaluated to establish the initial form state. | Critical |
| XCM-066 | Rebuild triggered by insert/delete | Lifecycle | Structural changes (insert/delete of repeat instances) set a rebuild flag that triggers full dependency graph reconstruction on the next update cycle. | Formspec engine must handle adding/removing repeat instances by updating the dependency graph. New instances introduce new paths that need new signal subscriptions. | High |
| XCM-067 | Node-set binding vs single-node binding | Expression | Two kinds of binding: single-node binding (`ref`) gives one context node; node-set binding (`nodeset`) gives a collection where each node gets its own evaluation context. | Formspec distinguishes between items (single field) and groups (containers/repeats). The binding semantics differ accordingly — items resolve to single values, repeating groups resolve to collections. | High |
| XCM-068 | Notification events for state changes | Lifecycle | Refresh dispatches specific notification events: value-changed, valid/invalid, enabled/disabled, required/optional, readonly/readwrite. These enable fine-grained UI reactions. | Formspec's web component could dispatch custom events for each state change type. Useful for external integrations, analytics, and accessibility announcements. Currently the web component re-renders but may not emit granular events. | Medium |
| XCM-069 | UI binding expression re-evaluation on refresh | Lifecycle | During refresh, all UI binding expressions are re-evaluated, not just those affected by changes. This ensures consistency between model and view. | In signal-based reactivity, only affected UI bindings re-render. Full re-evaluation is unnecessary and would harm performance. Formspec's approach (selective re-render via signals) is an improvement. | Medium |
| XCM-070 | Repeat objects as implicit groups | Repeat | Each repeat item is wrapped in an implicit group (a "repeat object") that provides scoping boundaries for its contained UI elements. | Formspec's repeating group instances implicitly scope their contents. The group boundary is defined by the repeat definition, not by an explicit wrapper element. Same concept, simpler representation. | Medium |
| XCM-071 | Data structure independent of UI structure | Composition | The data instance (model) has its own tree structure independent of the UI layout. Binds bridge the two structures. | Formspec's definition JSON items define data structure. The component/theme specs define UI layout. Binds connect them. The three-tier architecture directly implements this separation. | Critical |
| XCM-072 | Same model, different UIs | Composition | The same model can be presented by completely different UIs (desktop, voice, mobile). The model is presentation-agnostic. | Formspec explicitly supports this: the engine produces data and state, different renderers (web component, mobile, PDF) can consume it. The component spec is one possible rendering layer. | Critical |
| XCM-073 | Same UI pattern, different models | Composition | A UI template can bind to different models, enabling reusable UI patterns. | Formspec's component registry enables reusable rendering components that work with any definition. A custom component implementation can render any conforming form data. | High |
| XCM-074 | Validation separate from submission | Validation | Validation can be performed independently of submission (via revalidate). The submission pipeline optionally includes validation as a gate. | Formspec's `getValidationReport()` is independent of response extraction. Validation can run continuously, on demand, or as a submission gate. Already adopted. | High |
| XCM-075 | Shape rules / cross-field constraints | Validation | `constraint` expressions can reference other nodes (e.g., `../endDate > ../startDate`), enabling cross-field validation that goes beyond single-field type checking. | Formspec supports both bind-level constraints (per-field) and shape rules (cross-field). Shape rules are explicitly a form-level validation mechanism. Direct adoption. | Critical |
| XCM-076 | Submission with no serialization | Submission | `method="none"` skips serialization entirely. Useful for validity-check-only submissions or GET requests with no body. Disables relevance pruning and validation by default. | Formspec could support a "validate only" mode that runs the full validation pipeline without extracting a response. Useful for progressive validation UX. | Low |
| XCM-077 | Namespace handling in XML serialization | Submission | XML serialization includes `includenamespaceprefixes` for controlling namespace output. Follows XSLT output-method rules. | Not relevant to Formspec's JSON-native model. JSON has no namespace concept. The Mapping spec's XML adapter would need to handle namespaces if XML output is required. | Low |
| XCM-078 | Binary attachment handling in multipart submission | Submission | Upload controls and `xsd:anyURI` values are serialized as separate MIME parts in multipart submissions, with the instance referencing them. | Formspec should define how file/binary data is represented in responses and how it integrates with the Mapping spec's serialization. Currently underspecified for file upload scenarios. | Medium |
| XCM-079 | Lossy serialization awareness | Submission | URL-encoded and form-data formats are explicitly documented as lossy (leaf elements only, no hierarchy). The spec acknowledges format limitations. | Formspec's Mapping spec should document fidelity characteristics of each output format. Some formats cannot represent nested data or arrays. | Medium |
| XCM-080 | Bind as bridge between data and behavior | Data Model | The `<bind>` element is the sole mechanism connecting data nodes to computational behavior (type, validation, calculation, visibility). It is not a UI concept. | Formspec's bind concept directly mirrors this. Binds are defined at the data/logic level, not in the component or theme tiers. The three-tier architecture preserves this separation. | Critical |
| XCM-081 | Expression-based MIPs vs literal MIPs | Data Model | Some MIPs are computed (readonly, required, relevant, calculate, constraint — evaluated as expressions) while others are literal (type, p3ptype — static values). This distinction determines which participate in the reactive cycle. | Formspec should clearly distinguish static properties (datatype, label) from reactive properties (binds with FEL expressions). Only reactive properties participate in the dependency graph. | High |
| XCM-082 | Schema references in model | Data Model | The model can reference external XML Schemas (`<schema>`) for type definitions beyond built-in types. | Formspec uses JSON Schema for structural validation of definitions. Custom type extensions could be supported via the extension registry. Lower priority than core features. | Low |
| XCM-083 | Position and size in evaluation context | Expression | Within node-set bindings and repeats, expressions have access to `position()` (1-based index) and `last()` (collection size) in the evaluation context. | Formspec FEL could benefit from `index()` and `count()` functions or equivalent context variables within repeating groups. Currently, repeat context provides the instance index implicitly in path resolution. | Medium |
| XCM-084 | Implicit context vs explicit path references | Expression | Expressions can use relative paths (resolved against context) or absolute paths (from instance root). The choice affects portability and clarity. | FEL supports both relative paths (within repeat context) and absolute paths (from form root). The dual-mode approach matches XForms. | High |
| XCM-085 | Event handlers disabled on non-relevant controls | Reactivity | Non-relevant controls are disabled, which indirectly prevents their event handlers from firing. The controls are removed from interaction. | Formspec's web component should ensure non-relevant fields do not respond to user interaction — no events, no focus, no mutation. Important for form integrity. | High |

### Key Observations

1. **Already adopted (Critical/High)**: Formspec has already absorbed the most important XForms patterns — MIP-on-data-nodes, reactive dependency graphs, relevance as first-class concept, repeat with context scoping, and MVC separation. These are the conceptual backbone shared by both systems.

2. **Gaps worth addressing (High priority, not yet adopted)**:
   - Readonly OR-inheritance down ancestor chains (XCM-003, XCM-060)
   - Implicit readonly for calculated fields (XCM-061)
   - Multiple named data instances / auxiliary lookup data (XCM-035, XCM-037)
   - Explicit four-phase processing model documentation (XCM-019)
   - Duplicate bind conflict detection (XCM-009)

3. **Consciously divergent (Low priority)**:
   - P3P metadata (XCM-008) — not relevant to JSON-native forms
   - XML namespace handling (XCM-077) — not applicable
   - Repeat attribute syntax (XCM-034) — unnecessary in JSON
   - Schema references (XCM-082) — JSON Schema serves this role differently

4. **Submission model is the biggest divergence**: XForms defines a complete submission pipeline (serialize, transmit, handle response). Formspec delegates transmission to the consuming application, focusing on data extraction and validation. This is a deliberate architectural choice — Formspec produces structured data, not HTTP requests.
