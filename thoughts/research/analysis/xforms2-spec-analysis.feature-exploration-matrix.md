# XForms 2.0 Changes --- Feature Exploration Matrix

This matrix is a comprehensive inventory of every distinct feature, mechanism, and design pattern introduced or changed in XForms 2.0 (relative to XForms 1.1), extracted from the research analysis document. Each entry is assessed for its relevance to a JSON-native form specification (Formspec) and prioritized for later cross-referencing against the synthesized Formspec proposal. The source material is the W3C Community Group living document edited by Bruchez, Couthures, and Pemberton.

| ID | Feature/Pattern | Category | Description | Formspec Relevance | Priority |
|----|-----------------|----------|-------------|-------------------|----------|
| XF2-001 | XPath 2.0+ Expression Upgrade | Expression | Moves from XPath 1.0 to XPath 2.0+, enabling typed values, sequences, quantified expressions (`some`/`every`), `for` iteration, inline `if/then/else`, and richer built-in functions. | FEL already provides typed comparisons and a function library. Quantified expressions (`some`/`every`) and inline conditionals are patterns FEL should cover. Sequences are less relevant given JSON arrays as a native concept. | High |
| XF2-002 | Typed Values and Atomics | Expression | Expressions can return and operate on typed atomic values (integers, dates, decimals) rather than just strings and node-sets. | FEL supports typed comparisons already. Ensuring consistent type coercion rules (especially for date arithmetic and numeric precision) is the actionable takeaway. | Medium |
| XF2-003 | Quantified Expressions | Expression | `some $x in seq satisfies condition` and `every $x in seq satisfies condition` allow existential and universal quantification inline in expressions. | Directly useful for cross-field validation (e.g., "every item in cart satisfies price > 0"). FEL should have equivalent functions or syntax. | High |
| XF2-004 | For Expressions (Iteration) | Expression | `for $x in seq return expr` enables mapping/iteration within expressions, producing sequences. | Less critical in a JSON-native context where array functions (`map`, `filter`) serve the same purpose. Could be provided as FEL stdlib functions rather than syntax. | Medium |
| XF2-005 | Regex String Functions | Expression | `matches()`, `replace()`, `tokenize()` provide regex-based string manipulation directly in expressions. | FEL already has `$matches()`. Should ensure `$replace()` and `$split()` (tokenize equivalent) are available. | Medium |
| XF2-006 | Date/Time Arithmetic | Expression | Date subtraction, duration addition, and epoch conversion are built into the expression language. | FEL has date functions. Verify that duration arithmetic (e.g., `$addDays()`, `$dateDiff()`) is complete and consistent. | Medium |
| XF2-007 | Type Constructors | Expression | Explicit type casting via `xs:date(...)`, `xs:decimal(...)`, etc. allows safe type conversion within expressions. | FEL could benefit from explicit cast functions (`$toDate()`, `$toNumber()`, `$toString()`) if not already present. | Low |
| XF2-008 | Attribute Value Templates (AVTs) | Expression | `{expression}` syntax allows dynamic values in most attributes, enabling computed CSS classes, computed URLs, and computed labels. | Analogous to Formspec's template string / expression interpolation support. Ensure FEL expressions can be embedded in label, description, cssClass, and URL properties. | High |
| XF2-009 | `initial` MIP | Data Model | A new Model Item Property that evaluates an expression once during model initialization to set a field's starting value, before any `calculate` runs. | Maps directly to Formspec's `default` property on fields. Ensure `default` accepts FEL expressions (e.g., `$now()`) and that evaluation order is: defaults first, then calculates. | High |
| XF2-010 | `whitespace` MIP | Data Model | Controls whitespace treatment on value assignment: `preserve`, `remove`, `trim`, `collapse`, `normalize`. Applied declaratively per field. | Formspec lacks this. Adding a `whitespace` property (at minimum `trim` and `normalize`) to field bindings would address the extremely common need to clean user input. | High |
| XF2-011 | Declarative Sort MIP | Data Model | `sort`, `direction`, and `collation` MIPs keep node-sets in declared sort order automatically, with support for multi-level sorting. | Could be useful on repeat/array definitions to maintain sorted display. Lower priority because sorting is often handled at the UI layer or via explicit FEL expressions. | Low |
| XF2-012 | Labels/Help/Hint/Alert as MIPs | UI Hint | UI text properties (`label`, `help`, `hint`, `alert`) can be declared on bind elements in the model, not just on controls, with a defined precedence cascade. | Formspec already puts label/description on field definitions. The precedence concept (theme layer can override definition layer) is relevant to the theme cascade design. | Medium |
| XF2-013 | Multiple Constraints per Field | Validation | Multiple bind elements can target the same node, each with its own `constraint` expression and its own `alert` message. Each failing constraint produces a distinct error. | CRITICAL for Formspec. Real-world forms need per-constraint error messages (e.g., "must be a number" vs. "must be positive" vs. "must be less than 100"). The validation model must support an array of constraint objects. | Critical |
| XF2-014 | Per-Constraint Alert Messages | Validation | Each `constraint` on a bind can carry its own `alert` attribute, so the user sees the specific reason for failure rather than a generic message. | Direct corollary of XF2-013. Formspec's constraint shape should pair each expression with a message and optionally a severity level. | Critical |
| XF2-015 | JSON Instance Data | Data Model | XForms 2.0 supports loading JSON as instance data by converting it to an internal XML representation with `type` attributes for round-tripping. | Formspec is JSON-native; this problem does not exist. This validates the architectural decision to avoid XML as an intermediary. No action needed. | Low |
| XF2-016 | CSV Instance Data | Data Model | CSV data can be loaded as instance data via a mapping to XML rows and columns. | Formspec's Mapping spec already covers CSV via adapters. Already addressed. | Low |
| XF2-017 | Instance Fallback | Data Model | The `src` attribute on `<instance>` supports fallback to inline content if the external source fails to load. | Relevant for data prefill. Formspec could support a `defaultData` object in the definition that acts as fallback when external data loading fails. | Medium |
| XF2-018 | Variables (`var` Element) | Expression | Declares named variables with an expression value, scoped within the model, usable in subsequent expressions to reduce repetition. | FEL does not have user-defined variables. A `variables` or `let` section in the definition could reduce expression duplication and improve readability for complex forms. | High |
| XF2-019 | Custom Functions (`function` Element) | Expression | User-defined functions declared in the model, supporting parameters, recursion, and local variables. | Powerful for large, complex forms with repeated logic. A `functions` section in the definition allowing named FEL expressions with parameters would be valuable. Consider carefully whether the complexity is justified. | Medium |
| XF2-020 | `valid()` State Query Function | Expression | Returns the validity state of a node, allowing expressions to reference whether another field is currently valid. | Useful for conditional logic (e.g., "show submit button only when all fields valid"). FEL should have `$valid(path)` or equivalent. | High |
| XF2-021 | `relevant()` State Query Function | Expression | Returns the relevance (visibility) state of a node from within an expression. | Enables meta-logic like "count of visible fields". FEL should have `$relevant(path)` or equivalent. | Medium |
| XF2-022 | `readonly()` / `required()` State Query Functions | Expression | Query the readonly and required MIP states of a node from within expressions. | Completes the set of state introspection functions. Useful for summary displays and conditional logic. | Medium |
| XF2-023 | `serialize()` Function | Expression | Serializes nodes to a string representation, useful for debugging and display. | A `$serialize()` or `$toJSON()` function in FEL would be useful for debugging and for displaying structured data as text. | Low |
| XF2-024 | URI Manipulation Functions | Expression | A family of functions (`uri-scheme()`, `uri-host()`, `uri-path()`, `uri-query()`, `uri-param-names()`, `uri-param-values()`) for parsing and manipulating URIs. | Niche but useful for forms that construct URLs dynamically (e.g., API endpoints, redirect URLs). Lower priority for a form spec. | Low |
| XF2-025 | `location-uri()` / `location-param()` | Expression | Access the current page URL and its query parameters from within expressions. | Useful for pre-filling form fields from URL parameters, a very common real-world pattern. FEL should support `$urlParam(name)` or similar. | High |
| XF2-026 | Dynamic Expression Evaluation | Expression | `eval()` and `eval-in-context()` allow evaluating dynamically constructed expression strings at runtime. | Powerful but dangerous. Formspec should avoid this pattern for security and analyzability reasons. Static expressions are preferable. | Low |
| XF2-027 | `is-card-number()` (Luhn Validation) | Validation | Built-in function that validates credit card numbers using the Luhn algorithm. | A useful stdlib addition. FEL could include `$isCardNumber()` or more generally `$luhn()` as a validation helper. | Low |
| XF2-028 | Cryptographic Functions | Expression | Cryptographic hash and related functions added to the built-in function library. | Generally handled server-side. Not appropriate for a client-side form expression language. | Low |
| XF2-029 | `seconds-from-epoch()` | Expression | Converts a date/time to Unix epoch seconds, enabling numeric date comparisons and arithmetic. | Useful utility function. FEL should have `$epoch()` or `$toTimestamp()`. | Low |
| XF2-030 | Form Embedding (`control` Element) | Composition | A new `<control resource="subform.xhtml">` element allows embedding one XForm inside another, creating composite forms from reusable sub-forms. | Form composition is important for large, modular form systems. Formspec should define how one definition can reference/embed another, with clear data flow semantics. | High |
| XF2-031 | Shared Data (`shared` Element) | Composition | The `<shared>` element specifies which data is shared between parent and embedded child forms, with directionality control (`initial="external"` vs. local-first). | Critical companion to form embedding. Data sharing semantics (parent-to-child, child-to-parent, bidirectional) must be explicitly defined if composition is supported. | High |
| XF2-032 | Inter-Form Signals (`signal` Action) | Composition | The `<signal>` action sends events between embedded forms, enabling inter-form communication beyond shared data. | Event/signal communication between composed forms enables coordination patterns. If Formspec supports composition, a signaling mechanism is needed. | Medium |
| XF2-033 | Dialog/Modal Element | UI Hint | A `<dialog>` element with `level="modal"` or `level="modeless"` for declarative modal dialogs, controlled by `show` and `hide` actions. | Dialogs are a common form UI pattern (confirmation prompts, detail views). Could be a component-layer container type. Lower priority because it can be handled at the UI/component layer without spec support. | Low |
| XF2-034 | Collapsible Groups | UI Hint | The `collapse` attribute on `<group>` (`open`, `closed`) makes sections collapsible, a common progressive disclosure pattern. | Very common in real-world forms. Formspec's group/section component should support a `collapse` or `collapsible` presentation hint with initial state. | Medium |
| XF2-035 | Data-Driven Switch/Case | Reactivity | `<switch ref="path">` with `<case name="value">` children selects the visible case based on the bound data value, automatically updating when the value changes. | Formspec achieves this via `relevant` expressions on groups, but a dedicated switch/case pattern is more ergonomic for mutually exclusive sections driven by a single selector field. Worth considering as a component pattern. | Medium |
| XF2-036 | MIPs Directly on Controls | Data Model | Model Item Properties (relevant, required, readonly, constraint, calculate) can appear directly on control elements, not just on bind elements. | Formspec already co-locates constraints with field definitions. This validates the approach of keeping field logic close to field declaration rather than in a separate binding section. | Low |
| XF2-037 | Labels/Hints as Attributes | UI Hint | `label`, `hint`, `help`, `alert` can be simple attributes on controls (not just child elements), reducing verbosity for simple static text. | Formspec already uses properties on field definitions for labels/descriptions. Confirms the approach. | Low |
| XF2-038 | Rich Output (mediatype) | UI Hint | `<output>` can render HTML fragments, SVG, MathML, and images via a `mediatype` attribute, not just plain text. | Relevant for display-only components. Formspec's `display` component category could support a `mediatype` or `format` property for rich content rendering. | Medium |
| XF2-039 | `select1` Deselection | UI Hint | A single-select control can now be deselected, returning to no selection (null/empty state). | Formspec should ensure that single-select fields can have a null/unset state, not just a default first option. This affects validation (required vs. optional selects). | Medium |
| XF2-040 | Repeat Index Synchronization (`indexref`) | Repeat | `indexref` binds the current repeat index to a data node, enabling two-way sync between repeat position and data values. | Useful for wizard-style step navigation where the current step number is stored in form data. Could be a property on Formspec's repeat definition. | Medium |
| XF2-041 | Inline Repeat Rendering | Repeat | `appearance="minimal"` renders repeat items inline (e.g., comma-separated tags) rather than as block-level sections. | A presentation hint on repeating groups. Formspec's theme layer could support repeat display modes (block, inline, table). | Low |
| XF2-042 | Repeat Over Computed Sequences | Repeat | Repeat can iterate over any sequence (e.g., `1 to 10`), not just data nodes, enabling template-driven iteration. | Interesting for generating fixed-count layouts or iterating over computed ranges. Less critical in JSON-native context where array manipulation is more natural. | Low |
| XF2-043 | Repeat CSS Pseudo-Elements | Repeat | `::repeat-item` and `::repeat-index` CSS pseudo-elements for styling repeat instances and the current index. | Relevant to the CSS integration design (ADR-0020). Formspec could define CSS custom properties or class conventions for repeat item styling. | Low |
| XF2-044 | JSON Submission Serialization | Submission | Form data can be serialized as `application/json` for submission, using the reverse of the JSON-to-XML mapping. | Formspec is JSON-native, so submission serialization is trivially JSON. Already solved by architecture. | Low |
| XF2-045 | CSV Submission Serialization | Submission | Form data can be serialized as `text/csv` for submission. | Covered by Formspec's Mapping spec adapters which handle CSV transforms. | Low |
| XF2-046 | Non-Relevant Field Handling on Submit | Submission | Three modes for handling non-relevant (hidden) fields during submission: `keep` (include all), `remove` (exclude non-relevant, default), `empty` (include but blank out). | CRITICAL design decision. Formspec must define what happens to conditionally-hidden field values in `getResponse()`. All three modes serve real use cases. Should be configurable per-form or per-submission. | Critical |
| XF2-047 | Computed Submission Value | Submission | A `value` attribute on submission allows an expression to compute the exact payload to submit, rather than submitting the instance directly. | Relevant for cases where the submission payload differs from the internal data model (transformations, subsetting). Formspec's Mapping spec partially addresses this. | Medium |
| XF2-048 | `xforms-submit-ready` Event | Lifecycle | Fires before submission actually sends, allowing pre-submission logic (last-minute calculations, confirmation prompts). | Formspec should define lifecycle hooks: pre-submit, post-submit, pre-validate. These enable integration points without requiring imperative code in the spec. | High |
| XF2-049 | `xforms-refresh-done` Event | Lifecycle | Fires after UI refresh completes, signaling that the DOM is in sync with the model. | Relevant for integration testing and external system coordination. FormEngine could emit lifecycle events that component layers can observe. | Medium |
| XF2-050 | `xforms-action-warning` Event | Lifecycle | Non-fatal warning during action processing, allowing graceful degradation rather than hard errors. | A warning severity level in the processing model. Formspec's validation already supports warning severity. Extend to processing/lifecycle events. | Low |
| XF2-051 | Event Handlers on Disabled Controls | Reactivity | Event handlers continue to fire on disabled/non-relevant controls, enabling background processing and state management for hidden fields. | Important behavioral detail. Formspec's `calculate` expressions should continue to evaluate for non-relevant fields so that values are ready if the field becomes relevant again. | High |
| XF2-052 | Conditional Event Cancellation (`cancel`) | Lifecycle | A `<cancel>` action conditionally cancels an in-progress event, providing a declarative guard/veto mechanism. | Useful for pre-submission validation gates. Formspec could support a `guard` expression on lifecycle transitions (e.g., prevent page navigation if validation fails). | Medium |
| XF2-053 | Action Iteration (`@iterate`) | Reactivity | Actions can iterate over node-sets via an `iterate` attribute, applying the action to each node in sequence. | In a JSON-native spec, this maps to operations over arrays. Less relevant given that FEL functions already operate on arrays. | Low |
| XF2-054 | Script Calling from Actions | Extension | Actions can invoke JavaScript functions with parameter passing, bridging declarative form logic and imperative code. | Relevant for extension/escape-hatch scenarios. Formspec's component registry already allows custom components with imperative code. A formal `onAction` callback hook could be valuable. | Medium |
| XF2-055 | Custom Event Properties | Lifecycle | Dispatched events can carry custom properties/payload, enabling richer inter-component communication. | If Formspec supports lifecycle events, custom event data enables form-to-host-application communication patterns. | Medium |
| XF2-056 | Empty Non-Required = Valid | Validation | A field is valid if it is empty and not required, regardless of type constraints. Type checking only applies to non-empty values. | CRITICAL validation semantic. Formspec must match this behavior: an optional number field left blank should not fail "must be a number" validation. Verify current implementation. | Critical |
| XF2-057 | Absolute IRI Datatype | Data Model | `xf:absoluteIRI` type for validating absolute IRIs (internationalized URIs). | Could be a built-in format/type for URL fields. FEL could include `$isURL()` or a `url` format constraint. | Low |
| XF2-058 | HTTP IRI Datatype | Data Model | `xf:httpIRI` type specifically for HTTP/HTTPS URIs. | Narrower than XF2-057. A format constraint `httpUrl` could be useful but is low priority. | Low |
| XF2-059 | Internationalized Email Datatype | Data Model | `xf:iemail` type supporting internationalized email addresses (IDN, Unicode local parts). | Relevant. A `email` format that supports internationalized addresses is important for global forms. | Medium |
| XF2-060 | Telephone Datatype | Data Model | `xf:telephone` type for phone numbers. | Useful built-in format. Phone validation is a common need. Could be a FEL function `$isPhone()` or a format constraint. | Medium |
| XF2-061 | HTML Fragment Datatype | Data Model | `xf:HTMLFragment` type for fields containing rich text / HTML content. | Relevant for rich text editor fields. Formspec should have a `richtext` or `html` field type that the component layer renders with appropriate editors. | Medium |
| XF2-062 | Card Number Datatype | Data Model | `xf:card-number` type for credit card numbers (14-18 digits), validated with Luhn algorithm. | Covered by XF2-027 (function). A dedicated type plus validation function is a clean pattern. Low priority as a built-in type. | Low |
| XF2-063 | Dependency Tracking Clarification | Reactivity | The spec clarifies that an expression "references" a node if that node is selected during evaluation, even if subsequently filtered out. This defines the recalculation trigger boundary. | Formspec's DependencyVisitor performs static analysis of expressions. The clarification suggests that runtime dependency tracking (dynamic dependencies) may be more correct than static analysis alone. Important for correctness. | High |
| XF2-064 | Lazy/Incremental Evaluation | Reactivity | The spec explicitly permits lazy/incremental evaluation as an optimization, as long as the observable result matches eager evaluation. | Formspec already uses Preact Signals for reactive computation, which provides lazy evaluation. Validates the architectural choice. | Low |
| XF2-065 | Batched Updates During Actions | Reactivity | Updates are deferred during action handler processing; recalculation and UI refresh happen only after the outermost action handler completes. | CRITICAL for performance and correctness. Formspec's FormEngine should batch `setValue()` calls and defer recalculation until the batch completes. Preact Signals' `batch()` enables this. | Critical |
| XF2-066 | Standalone `form` Root Element | Composition | XForms 2.0 adds a `<form>` root element so forms can exist standalone, not just embedded in XHTML host documents. | Formspec definitions are already standalone JSON documents. Validates the approach. | Low |
| XF2-067 | `nodeset` Deprecated in Favor of `ref` | Data Model | The `nodeset` attribute is deprecated; `ref` is used uniformly for all binding references. | Formspec uses `bind` uniformly. Confirms the value of a single, consistent binding attribute name. | Low |
| XF2-068 | `get` as Default Submission Method | Submission | Changed from `post` to `get` as the default HTTP method for submission. | Not directly relevant to Formspec, which does not prescribe HTTP methods. The submission/transport layer is external to the spec. | Low |
| XF2-069 | Response Mediatype Override | Submission | `response-mediatype` attribute overrides the server's Content-Type for the submission response, allowing client-side interpretation control. | Niche. Not relevant for a JSON-native spec where responses are expected to be JSON. | Low |
| XF2-070 | Validate Attribute Default Change | Submission | Validation defaults to `false` when `serialization="none"`, `true` otherwise, making non-submitting triggers skip validation by default. | Relevant design pattern: distinguish between "save draft" (no validation) and "submit" (full validation) actions. Formspec could support a `validateOnSubmit` flag. | Medium |
| XF2-071 | Deprecated Processing Events | Lifecycle | `xforms-rebuild`, `xforms-recalculate`, `xforms-revalidate`, `xforms-refresh`, `xforms-reset` events are deprecated in favor of direct action invocations. | Confirms the pattern of using method calls rather than events for internal processing. Formspec's `FormEngine.setValue()` / `getResponse()` API follows this approach. | Low |
| XF2-072 | `reset` and `retain` Actions | Lifecycle | `<reset instance="id"/>` resets an instance to initial state; `<retain/>` marks data to survive a reset, enabling partial reset. | Formspec should support `FormEngine.reset()` and potentially per-field or per-section reset. The `retain` concept is interesting for wizard forms where some sections should survive a reset. | Medium |
| XF2-073 | `bind()` Function | Expression | References a bind element's selected nodes from within an expression, allowing one bind to reference another's node selection. | In Formspec, binds and fields are unified. Less relevant, but the concept of referencing a named field's computed state from another expression is already supported via path references. | Low |
| XF2-074 | `case()` Function | Expression | Queries which case is currently selected in a switch control, enabling expressions that depend on UI navigation state. | If Formspec adds a switch/case component (XF2-035), a corresponding state query function would be needed. Dependent on XF2-035. | Low |

### Critical Items (Immediate Attention)

1. **XF2-013 / XF2-014** --- Per-constraint error messages with individual alert text. The validation model must support multiple constraints per field, each paired with its own message.
2. **XF2-046** --- Non-relevant field handling on submit (`keep`/`remove`/`empty`). A fundamental design decision for `getResponse()` behavior.
3. **XF2-056** --- Empty non-required fields must be valid regardless of type constraints. Core validation semantic that must be verified.
4. **XF2-065** --- Batched updates during action processing. Multiple `setValue()` calls must defer recalculation until the batch completes.

### High Priority Items (Near-Term Design)

1. **XF2-001 / XF2-003** --- Quantified expressions and rich expression capabilities in FEL.
2. **XF2-008** --- Attribute Value Templates / expression interpolation in labels, classes, and URLs.
3. **XF2-009** --- Initial value expressions (dynamic defaults like `$now()`).
4. **XF2-010** --- Whitespace handling (`trim`/`normalize`) on text fields.
5. **XF2-018** --- User-defined variables to reduce expression duplication.
6. **XF2-020** --- Validity state query function (`$valid(path)`).
7. **XF2-025** --- URL parameter access function (`$urlParam(name)`).
8. **XF2-030 / XF2-031** --- Form composition with data sharing semantics.
9. **XF2-048** --- Pre-submit lifecycle hook.
10. **XF2-051** --- Calculate expressions continue evaluating for non-relevant fields.
11. **XF2-063** --- Dependency tracking correctness (static vs. dynamic analysis).
