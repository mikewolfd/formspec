# Secondary Influences — Feature Exploration Matrix

This matrix inventories every distinct feature, pattern, and design decision extracted from the secondary-influence analysis covering ODK XLSForm, SurveyJS, JSON Forms, CommonGrants, and React JSON Schema Form (RJSF). Each entry captures the mechanism as it exists in the source system and assesses its relevance to a JSON-native declarative form standard. The goal is a comprehensive catalog for cross-referencing against Formspec's synthesized design.

| ID | Feature/Pattern | Category | Source | Description | Formspec Relevance | Priority |
|----|-----------------|----------|--------|-------------|-------------------|----------|
| SI-001 | `${field_name}` reference shorthand | Expression | ODK | A template syntax (`${field_name}`) that compiles down to full XPath paths during XLSForm-to-XForm conversion, hiding XPath complexity behind a human-friendly surface. | Validates the approach of having a simple field-reference syntax that compiles to a rigorous underlying model. Formspec's FEL path references serve the same role without XPath baggage. | High |
| SI-002 | `.` (dot) self-reference in constraints | Expression | ODK | In constraint expressions, `.` refers to the current question's value, allowing rules like `. >= 18` without naming the field. | Elegant shorthand for field-level validation rules. Reduces redundancy when the constraint applies to the field it is declared on. Worth adopting in FEL constraint contexts. | High |
| SI-003 | `..` parent reference for repeat context | Expression | ODK | `position(..)` navigates to the parent repeat instance's index using standard XPath relative navigation. | Formspec's indexed-path model (`group[0].field`) already provides explicit navigation, but a relative-parent syntax could simplify nested repeat expressions. | Medium |
| SI-004 | Mixed notation (shorthand + raw XPath) | Expression | ODK | `${field}` shorthand and raw XPath can be freely intermixed, e.g., `${people}[age < 18]/pet_count`. | Demonstrates the value of allowing both simple references and richer query predicates in the same expression. FEL should support predicate filtering on repeat paths. | Medium |
| SI-005 | `calculate` row type with named output | Data Model | ODK | Calculated values are defined as a `calculate` row type with the expression in a `calculation` column; the result is referenced by the row's `name`. | Directly maps to Formspec's `calculate` bind attribute. Confirms the pattern of named calculated fields with declarative expressions. | Critical |
| SI-006 | `constraint` with `.` and `constraint_message` | Validation | ODK | The `constraint` column holds a boolean expression evaluated on advance; `.` represents the entered value. Failed constraints display a `constraint_message`. | Direct analog to Formspec's `constraint` bind with error messages. The `.` self-reference pattern is the specific innovation worth considering. | High |
| SI-007 | `relevant` for conditional visibility | Reactivity | ODK | The `relevant` column holds a boolean expression controlling question/group visibility. | 1:1 mapping to Formspec's `relevant` bind attribute. Confirms the pattern across multiple systems. | Critical |
| SI-008 | `trigger` column for manual dependency | Reactivity | ODK | Controls when a calculation re-evaluates — only when the referenced field changes, acting as a manual dependency declaration. | An anti-pattern that Formspec avoids via automatic dependency graph extraction. Useful as a cautionary example of what happens without proper dependency tracking. | Low |
| SI-009 | Dynamic defaults (expression evaluated once) | Data Model | ODK | The `default` column can hold expressions evaluated once on record/repeat creation, not continuously re-evaluated. | Distinguishes one-shot initialization from continuous calculation. Formspec could benefit from a `default` expression that evaluates once vs. `calculate` that continuously tracks. | High |
| SI-010 | `begin_repeat` / `end_repeat` groups | Repeat | ODK | Block-delimited syntax wrapping repeating question sets with `repeat_count` for fixed or dynamic counts. | Confirms the repeating-group pattern. Formspec uses `repeat` with `minOccurs`/`maxOccurs` and FEL-driven counts. | Critical |
| SI-011 | `position()` function (1-based index) | Repeat | ODK | Returns the current repeat instance index (1-based) for use in expressions within repeating groups. | Formspec needs an equivalent function in FEL for repeat-aware expressions. The 0-based vs. 1-based choice is a design decision. | High |
| SI-012 | Cross-instance repeat references | Repeat | ODK | `${tree}[position() = position(current()/..) - 1]/species` references a field from a previous repeat instance using XPath predicates. | Complex but essential for patterns like running totals or "compare to previous row." FEL needs a clean syntax for cross-instance references. | High |
| SI-013 | Nodeset filtering with predicates | Expression | ODK | `${people}[age < 18]/pet_count` filters repeat instances by predicate; results pass to aggregate functions like `sum()`, `count()`. | Critical for aggregate calculations over filtered subsets of repeat data. FEL should support predicate-based filtering on repeat paths. | High |
| SI-014 | `last-saved` references | Data Model | ODK | `${last-saved#street}` pulls values from the last saved record of the same form definition, enabling pre-population from prior submissions. | A submission-lifecycle feature. Relevant if Formspec supports multi-session or progressive-save workflows. Lower priority for initial spec. | Low |
| SI-015 | Global reactive re-evaluation model | Reactivity | ODK | All expressions re-evaluate on every change (form open, value change, repeat add/delete, save/finalize) — no dependency graph optimization. | An anti-pattern that Formspec explicitly improves upon with its dependency-graph-driven reactive model using Preact Signals. | Low |
| SI-016 | `{fieldName}` curly brace references | Expression | SurveyJS | Clean `{fieldName}` syntax for referencing question values in expressions. | Similar to FEL's path reference syntax. Confirms that simple, clean field references are a universal need. | High |
| SI-017 | Dot notation for nested values | Expression | SurveyJS | `{matrixid.rowid.columnid}` and `{dpanelid[index].qid}` access nested and indexed values within complex structures. | Confirms Formspec's dot-notation path resolution for nested items and repeat instances. | High |
| SI-018 | Array indexing with negative indexes | Expression | SurveyJS | `{qid[0]}` for first element, `{dpanelid[-1].qid}` for last element using negative index syntax. | Negative indexing is a usability improvement for accessing the last repeat instance. Worth considering for FEL. | Medium |
| SI-019 | Relative context prefixes (`row.`, `panel.`, `prevRow.`) | Repeat | SurveyJS | Context-aware references within matrices and dynamic panels: `{row.columnid}`, `{prevRow.columnid}`, `{panel.qid}`, `{parentPanel.qid}`. | Elegant alternative to XPath navigation for repeat-relative references. Could simplify FEL expressions within repeating sections vs. explicit indexed paths. | High |
| SI-020 | Element property access (`{$elname.propname}`) | Expression | SurveyJS | `{$elname.propname}` accesses metadata (visibility, custom properties) of elements, not just values. `{$self.propname}` for current element. | Extends expression scope beyond data values to element metadata. Useful for advanced conditional logic based on UI state (e.g., "if this other field is visible"). | Medium |
| SI-021 | `{$item.propname}` for choice filtering | Expression | SurveyJS | In `choicesVisibleIf`, `{$item}` references each choice item, enabling declarative choice filtering expressions. | Relevant for dynamic choice lists where options are filtered based on other field values. A common real-world need. | Medium |
| SI-022 | `#` prefix disables type coercion | Expression | SurveyJS | `{#q1}` prevents auto-conversion of "true"/"123" strings to boolean/number, preserving raw string value. | Type coercion control is important for edge cases. FEL's type system should have clear coercion rules with an opt-out mechanism. | Medium |
| SI-023 | PEG.js-based expression parser | Expression | SurveyJS | A proper grammar-based parser (PEG.js) rather than regex, supporting rich operator set and extensibility. | Validates Formspec's use of Chevrotain (also a proper parser generator) for FEL. Grammar-based parsing is the right approach. | Critical |
| SI-024 | Rich comparison operators (`empty`, `notempty`, `contains`, `anyof`, `allof`) | Expression | SurveyJS | Word-form operators alongside symbolic ones: `empty`, `notempty`, `contains`, `anyof`, `allof`, `noneof`. Both `>=` and `greaterorequal` supported. | More readable than function calls for common operations. FEL could benefit from `empty`/`notempty` operators or equivalent built-in functions. | High |
| SI-025 | Dual symbolic and word-form operators | Expression | SurveyJS | Both `>=` and `greaterorequal`, `!=` and `notequal` supported for inclusive authoring. | Accessibility for non-developer form authors. Lower priority if Formspec targets developer audiences, but valuable for broader adoption. | Low |
| SI-026 | Built-in date/time functions (`age()`, `today()`, `dateDiff()`, `dateAdd()`) | Expression | SurveyJS | First-class date manipulation in the expression language: `age()`, `today()`, `dateDiff()`, `dateAdd()`. | FEL already includes date functions. Confirms these are essential for real-world form logic (age calculations, date ranges, deadlines). | Critical |
| SI-027 | Array aggregate functions (`sumInArray()`, `countInArray()`) | Expression | SurveyJS | Purpose-built functions for aggregating over arrays/repeat data: `sumInArray(expression, field)`, `countInArray(expression, field)`. | Maps to FEL's `sum()`, `count()` over repeat paths. The "InArray" naming pattern suggests operating on a specific array, which is clearer than generic aggregation. | High |
| SI-028 | `displayValue()` function | Expression | SurveyJS | Returns the display text (not the stored value) of a question — e.g., the label of a selected option rather than its code. | Useful for summary pages and conditional logic based on human-readable values. Formspec could expose display values through a similar function. | Medium |
| SI-029 | Custom function registration with async support | Extension | SurveyJS | `registerFunction({ name, func, isAsync?, useCache? })` allows custom and async functions with optional caching. | Critical for extensibility: server-validated fields, API lookups, custom business logic. Formspec's extension registry should support function registration. | High |
| SI-030 | Expression validation API (`validateExpressions()`) | Validation | SurveyJS | Detects unknown variables, unknown functions, semantic errors (always-true/false), and syntax errors at design time. | Essential for form authoring tooling. Formspec's Python validator/linter already does some of this; the TypeScript side should match. | Critical |
| SI-031 | Parsed-on-instantiation dependency tracking | Reactivity | SurveyJS | Expressions are parsed when the survey model is created; the system tracks which values each expression references for targeted re-evaluation. | Exactly what Formspec's DependencyVisitor does. Confirms this as the correct reactive architecture. | Critical |
| SI-032 | `calculatedValues` array with `includeIntoResult` | Data Model | SurveyJS | Calculated values defined as `{ name, expression, includeIntoResult }` — the boolean controls whether computed values appear in form output. | Useful distinction: some calculations are intermediate (for display/logic only) vs. final (included in submission). Formspec could add an `includeInResponse` flag. | High |
| SI-033 | Variables vs. Calculated Values distinction | Data Model | SurveyJS | Variables are set imperatively via `setVariable()` and evaluated once. Calculated values are declarative and continuously re-evaluated. | Parallels the ODK default-vs-calculate distinction. Confirms the need for both one-shot initialization and continuous computation. | Medium |
| SI-034 | `visibleIf` / `enableIf` / `requiredIf` | Reactivity | SurveyJS | Three separate expression-driven properties controlling visibility, enabled state, and required state dynamically. | Direct analogs to Formspec's `relevant`, `readonly` (inverted), and `required` bind attributes. Validates the tri-property approach. | Critical |
| SI-035 | `choicesVisibleIf` / `rowsVisibleIf` / `columnsVisibleIf` | Reactivity | SurveyJS | Filter individual items within a question (choices, matrix rows, matrix columns) using expressions with `{item}` placeholder. | Enables dynamic choice filtering without rebuilding option lists. A common UX need that Formspec's component spec should address. | High |
| SI-036 | Trigger actions (`complete`, `setvalue`, `copyvalue`, `runexpression`, `skip`) | Lifecycle | SurveyJS | Action-oriented rules that fire when an expression becomes true: complete survey, set a value, copy a value, run an expression, skip to a question. | Side-effect-producing actions complement pure expressions. Formspec could adopt declarative triggers for navigation, value-setting, and form completion. | Medium |
| SI-037 | `checkErrorsMode` (immediate/onNextPage/onComplete) | Validation | SurveyJS | Three validation timing modes: `onValueChanged` (immediate), `onNextPage` (default), `onComplete` (deferred). | Maps closely to JSON Forms' three validation modes. Formspec should specify when validation fires as a first-class configuration. | High |
| SI-038 | Built-in validator types (`numeric`, `text`, `email`, `expression`, `regex`) | Validation | SurveyJS | Pre-built validator objects with type-specific configuration: `numeric` (min/max), `text` (minLength/maxLength), `email`, `expression` (arbitrary), `regex`. | Formspec handles this through JSON Schema constraints plus FEL `constraint` expressions. The typed-validator pattern is an alternative worth noting. | Medium |
| SI-039 | Expression validator type | Validation | SurveyJS | `{ type: "expression", expression: "...", text: "error msg" }` allows any boolean expression as a field-level validator. | Directly parallels Formspec's `constraint` bind with `constraintMsg`. Confirms the pattern. | High |
| SI-040 | Server-side validation event (`onServerValidateQuestions`) | Validation | SurveyJS | An async event with `complete()` callback for server-round-trip validation during form progression. | Critical for real-world forms needing server validation (uniqueness checks, business rule validation). Formspec should define a server-validation hook pattern. | High |
| SI-041 | Async validator functions with caching | Validation | SurveyJS | Custom functions registered with `isAsync: true, useCache: true` for server-validated expressions that cache results. | Performance optimization for expensive server validations. The caching pattern prevents redundant server calls. | High |
| SI-042 | `validationAllowSwitchPages` | Validation | SurveyJS | Allows users to navigate away from pages with validation errors, enabling non-blocking progressive completion. | UX consideration for multi-page forms. Formspec's page-navigation model should specify whether validation blocks page transitions. | Medium |
| SI-043 | Data schema / UI schema separation | Composition | JSON Forms | Complete separation into JSON Schema (data structure + constraints) and UI Schema (layout + presentation). The UI schema is optional — a default layout can be auto-generated. | Core architectural principle already adopted by Formspec's three-tier spec (Core/Theme/Component). JSON Forms validates this as the correct approach. | Critical |
| SI-044 | JSON Pointer `scope` references (`#/properties/name`) | Data Model | JSON Forms | UI schema elements reference data schema paths using JSON Pointer syntax for precise, standardized path resolution. | Formspec uses dot-notation paths instead. JSON Pointer is more verbose but standards-compliant (RFC 6901). A design trade-off to document. | Medium |
| SI-045 | Layout types (`VerticalLayout`, `HorizontalLayout`, `Group`, `Categorization`) | UI Hint | JSON Forms | Structural layout elements in the UI schema control visual arrangement independent of data structure. | Maps to Formspec's component spec layout components (stack, columns, page, section). Confirms the need for layout-as-data. | High |
| SI-046 | Auto-generated UI from data schema | UI Hint | JSON Forms | Given only a JSON Schema, JSON Forms generates a complete default form layout without requiring a UI schema. | "Zero-config" form generation. Formspec could define a default rendering algorithm from the definition spec alone, without a theme or component spec. | High |
| SI-047 | Rule conditions as JSON Schema validation | Validation | JSON Forms | Conditions are expressed as JSON Schema validation against scoped data: if data at `scope` matches `schema`, the condition is true. Uses `const`, `enum`, range, `not`, etc. | Elegant reuse of JSON Schema for conditions. Limited to single-scope checks — no cross-field expressions. Formspec's FEL is strictly more powerful but less standards-aligned. | Medium |
| SI-048 | Rule effects (HIDE/SHOW/ENABLE/DISABLE) | Reactivity | JSON Forms | Rules produce one of four effects: `HIDE`, `SHOW`, `ENABLE`, `DISABLE`, controlling element visibility and interactivity. | Maps to Formspec's `relevant` (HIDE/SHOW) and `readonly` (ENABLE/DISABLE) binds. JSON Forms uses explicit effect names while Formspec uses boolean expressions. | High |
| SI-049 | Three validation modes (ValidateAndShow/ValidateAndHide/NoValidation) | Validation | JSON Forms | `ValidateAndShow` (default, inline errors), `ValidateAndHide` (errors available programmatically but hidden), `NoValidation` (skip entirely). | A practical tri-state for progressive disclosure and draft-save scenarios. Formspec should specify equivalent validation presentation modes. | High |
| SI-050 | External error injection (`additionalErrors`) | Validation | JSON Forms | Server-side errors injected via `additionalErrors` prop using AJV's `ErrorObject` format; mixed with client-side errors and displayed regardless of validation mode. | Critical for server-side validation integration. Formspec's ValidationReport should support external/injected errors alongside engine-computed errors. | High |
| SI-051 | `failWhenUndefined` option for rule conditions | Validation | JSON Forms | When set to `true`, a rule condition fails (rather than being vacuously true) when the scoped value is undefined. | Important edge case handling. Formspec should define how `relevant`/`required`/`constraint` expressions behave when referenced fields have no value. | Medium |
| SI-052 | Renderer architecture with renderer sets | Extension | JSON Forms | Custom renderers replace how any data type or UI element is rendered. Renderer sets swap entire UI frameworks (Material, Vanilla, Angular, Vue). | Directly maps to Formspec's component registry pattern with swappable component implementations. Confirms the renderer-registry architecture. | High |
| SI-053 | Middleware for form behavior interception | Extension | JSON Forms | Middleware layer intercepts and modifies form behavior between the engine and renderers. | An extension point Formspec could adopt: middleware hooks between FormEngine and the web component layer for logging, analytics, or custom behavior. | Medium |
| SI-054 | `field` mapping operation | Submission | CommonGrants | Direct field reference using dot-notation path: `{ "field": "summary.opportunity_amount" }`. Maps a source path to a target structure position. | Already adopted in Formspec's mapping spec. The simplest and most common mapping operation. | Critical |
| SI-055 | `const` mapping operation | Submission | CommonGrants | Injects a constant value: `{ "const": "USD" }`. Essential for injecting fixed values (currency codes, version numbers, type discriminators). | Critical |
| SI-056 | `switch` mapping operation (enum translation) | Submission | CommonGrants | Value mapping with case/default: `{ "switch": { "field": "...", "case": {...}, "default": "..." } }`. | SI-058 | Four-layer form architecture (data + UI + mapping + extensions) | Composition | CommonGrants | `FormBase` combines `jsonSchema`, `uiSchema`, `mappingTo/From`, and `customFields` in a single coherent model. | Formspec's three-tier spec (Core/Theme/Component) plus the mapping spec equals four layers. | SI-059 | ADR-driven design evaluation criteria | Composition | CommonGrants | Evaluated mapping options against: human-readable, serializable, easy to validate, supports transformations, multi-runtime. | Formspec follows the same ADR-driven process (see `thoughts/adr/`). | SI-060 | Planned future mapping transformations | Submission | CommonGrants | Roadmap includes type conversion, string manipulation, date/time manipulation, conditional logic, and array manipulation (map, filter, reduce). | Formspec's mapping spec should consider these transformation types. | SI-061 | Schema-first zero-config form generation | UI Hint | RJSF | Given just a JSON Schema, RJSF generates a fully rendered form with validation. | SI-062 | JSON Schema type-to-widget mapping | UI Hint | RJSF | Automatic mapping: `string` → text input, `boolean` → checkbox, `integer` → number input, `array` → dynamic list, `object` → fieldset. | Formspec's component spec should define default component mappings for each data type, enabling zero-config rendering. | High |
| SI-063 | `oneOf`/`anyOf`/`allOf` as selection widgets | UI Hint | RJSF | JSON Schema `oneOf` renders as a dropdown/radio; `anyOf` as multi-select; `allOf` merges schemas. | SI-064 | JSON Schema `dependencies` for conditional logic | Reactivity | RJSF | Property dependencies make fields required based on others; schema dependencies show/modify fields based on values. | SI-065 | JSON Schema `if/then/else` for conditional schemas | Reactivity | RJSF | Standard JSON Schema conditional: `if` (condition schema), `then` (applied when true), `else` (applied when false). | SI-066 | `$ref` and `$defs` for schema reuse | Composition | RJSF | Standard JSON Schema reference mechanism for reusing schema fragments across the definition. | Formspec's JSON Schema-based definition should fully support `$ref`/`$defs` for DRY form definitions. | High |
