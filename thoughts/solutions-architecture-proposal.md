# Specification-Driven Form Infrastructure

## Thesis

**Declarative form specifications** — not application code — should serve as the single source of truth for data collection. A form defined once as a structured JSON document can be validated at every boundary and rendered on any runtime. This enables forms that are authored visually, generated by LLMs, version-controlled like code, validated statically before deployment, and executed identically across web, mobile, server, and offline environments — without application code per form.

---

## 1. Prior Art

**W3C XForms (2003)** solved model/view separation, reactive dependency graphs, non-relevant pruning, and a four-phase processing cycle (rebuild, recalculate, revalidate, notify). Its five declarative bind properties — `calculate`, `relevant`, `required`, `readonly`, `constraint` — remain the right abstractions. XForms failed on the web because it was XML-everything, but thrives through ODK in humanitarian aid and field research, where offline-capable, specification-driven data collection matters most.

**W3C SHACL (2017)** introduced named, composable validation shapes with structured results (focus node, path, severity, message, source shape). Shapes compose through boolean operators. Its limitation: any result of any severity means non-conformance. Form systems need warnings that do not block submission.

**HL7 FHIR Questionnaire / SDC (2023)** contributes a mature identity model (canonical URL + version + lifecycle), response pinning (responses validate against the definition version that created them), stable `linkId` identifiers, item taxonomy, variable scoping, and modular composition. The concepts are general; the implementation is healthcare-locked.

### Unclaimed Synthesis

No existing system combines all five opportunities:

1. **A purpose-built expression language** — deterministic, side-effect-free, JSON-native, host-language-independent. Not XPath (XML-coupled), FHIRPath (domain-locked), or JavaScript (too powerful).
2. **Composable validation with severity levels** — SHACL's shape architecture adapted so only errors block submission.
3. **Three-layer separation in JSON** — structure, behavior, and presentation as independent documents, with optional sidecars for richer control.
4. **JSON Schema as both machine validator and LLM constraint** — the same schema enforces validity and constrains generation.
5. **Multi-runtime execution from one spec** — client, server, and mobile runtimes coupled only through schemas and expression semantics.

## 2. The Dependency Inversion

Most form systems work backwards: a developer writes components, wires up validation in code, hopes the server enforces the same rules. The specification arrives after the fact.

Invert the chain:

```
Specification (normative prose)  →  what does a valid definition look like?
      ↓
JSON Schemas (contracts)         →  does this document conform?
      ↓
Validators (static + runtime)   →  is this form correct and safe to deploy?
      ↓
Runtimes (renderers, engines)   →  execute it
```

Specifications capture behavioral semantics schemas cannot express. Schemas make those rules machine-checkable. Validators close the gap — dangling references, circular dependencies, expression errors. The result: **you can prove a form definition valid without running it.**

## 3. Architecture

### 3.1 Document Model

Forms are composable document layers. Each adds a concern without modifying the layers below:

- **Definition** — fields, groups, pages, calculated values, conditional visibility, validation. The behavioral core, stable regardless of presentation or runtime.
- **Theme Sidecar** — design tokens, widget preferences, layout grid, selector cascade. Multiple themes per definition; same form renders differently on web, mobile, print.
- **Component Sidecar** — explicit component tree, slot bindings, responsive breakpoints. Fine-grained control when themes are insufficient. Optional.
- **Companion Documents** — mapping rules (bidirectional JSON/XML/CSV transforms), extension registries, changelogs.

**Presentation never affects data collection, validation, or behavioral semantics.** A theme can swap a calendar picker for a text input; it cannot alter what values are valid.

### 3.2 Processing Model

On every data change, the runtime walks a consistent four-phase cycle:

1. **Rebuild** — reconstruct the dependency graph (repeat instances added/removed, DAG acyclicity check)
2. **Recalculate** — evaluate only the affected subgraph in topological order
3. **Revalidate** — run constraints and shapes against updated data
4. **Notify** — signal state changes to the presentation layer

A client runtime triggers this reactively per keystroke. A server runtime triggers it once on submission. Both produce identical final state for identical input.

### 3.3 Expression Language (FEL)

Architectural requirements: deterministic, side-effect-free, JSON-native, host-language-independent. No statements, loops, variable assignment, I/O, or user-defined functions. Covers calculated fields, conditional visibility, cross-field constraints, and repeat aggregation. If client and server evaluate the same expressions with different semantics, the architecture collapses.

### 3.4 Structured Validation

Validation produces a machine-readable report, not a boolean. Each finding carries severity, field path, message, code, and the constraint expression that failed. Only errors block submission.

Two sources: **bind constraints** (per-field: required, pattern, type) and **validation shapes** (cross-field: "line items must sum to budget"). Shapes compose through boolean operators. External systems inject results into the same report, tagged by source.

The specification defines *what* is valid; each runtime decides *when* to evaluate.

## 4. Multi-Runtime Execution

Multiple runtimes implement the specification independently:

- **Client** — compiles expressions into a reactive dependency graph. Keystrokes propagate through signals: calculated fields recompute, visibility re-evaluates, validation updates — synchronously, without server round-trips.
- **Server** — re-evaluates the same definition against submitted data as a batch. Same specification, different interpreter.
- **Mobile** — native controls, local storage, offline validation. Syncs on reconnection; the server re-validates.

Runtimes share no code. They couple only through schemas and expression semantics. A new runtime can be built without touching the existing ones.

### Layout Planner / Renderer Split

Most form systems conflate *deciding what to render* (theme cascade, widget selection) with *rendering it* (DOM, native views, PDF). Every new target reimplements the entire pipeline.

Split them. A **layout planner** resolves definition + theme + component sidecar into a JSON-serializable tree — themes flattened, tokens resolved, widgets selected. A **renderer** translates that tree into framework-specific output. React, Vue, Web Components, SwiftUI, PDF — each is a thin last-mile layer.

Only three concerns defer to the renderer: conditional visibility (subscribe and show/hide), repeat stamping (clone per instance), and field binding (wire engine signals). Integrating a new framework means writing a renderer, not reimplementing the specification.

## 5. Platform Integration

### The Stack

Applications → Orchestration (workflows, approvals, routing) → Form Infrastructure (registry, engine, response store) → Authoring (builder, LLM generation, code-as-config) → Foundation (spec, schemas, validators, FEL).

### Definition Registry

Versioned artifacts managed like packages: semantic versioning, promotion pipeline (draft → review → staging → production), dependency tracking across subforms, access control, structured changelog generation.

### Response Store

Schema-validated, append-only JSON documents. Corrections create new versions. A mapping engine transforms responses into downstream formats (XML, CSV, JSON) without custom ETL. Audit trails are structural — diff two versions to see what changed.

### Workflow Integration

Sequential flows with pre-fill, conditional routing by response values, approval chains, scheduled recurring collection with version pinning.

## 6. Authoring & LLM Integration

The visual builder is one authoring surface among several — not a privileged one. It operates definition-first (no proprietary format), supports round-trip fidelity, embeds the same linter CI uses, and provides live preview. Anything it produces, a developer can write by hand, an LLM can generate, or a script can construct.

LLMs generate against the same JSON Schema that validators enforce — structurally valid by construction, with static validation catching semantic issues. Use cases: form generation from natural language or existing documents, response filling from unstructured data, conversational form filling, validation explanation, and definition modification.

The key advantage over LLM-generated code (React components, HTML): the LLM generates *data*, not code. A JSON document that passes schema validation presents no attack surface — no XSS review, no design system violations, no runtime lock-in.

---

## 7. Governance

Every artifact is structured and schema-validated: definitions prove what rules were enforced, responses prove what was submitted, validation reports prove conformance at submission time, changelogs prove what changed between versions, mapping rules prove how data was transformed.

Definitions are data, so they lint in CI like code. Invalid definitions never reach production. Breaking changes surface before deployment.

---

## 8. Summary

This document argues for a **form infrastructure layer**: specification-first, with structure, behavior, and presentation as independent concerns, every artifact a validated JSON document, and multiple runtimes executing the same behavioral contracts without sharing code.

The architectural bets:

1. **Specification over implementation** — the spec is the product; runtimes are replaceable
2. **Schemas as contracts** — machine-readable truth enables tooling, LLMs, and static analysis
3. **Validators as gates** — catch semantic errors schemas cannot express
4. **Separation of concerns** — presentation never affects behavior
5. **LLMs as first-class authors** — schema-constrained generation makes AI authoring safe
6. **Data over code** — JSON documents carry versioning, diffing, auditing, and portability that code forfeits

The system's value compounds with scale. One form does not justify the infrastructure. A thousand forms make custom code per form untenable.
