# Formspec Integration Gaps -- Design Specification

**Status:** Proposed
**Date:** 2026-04-10
**References:** `wos-spec/TODO.md` (Formspec Coprocessor gap, Formspec dependencies); WOS Kernel Specification v1.0; WOS Runtime Companion v1.0; WOS Workflow Governance Specification v1.0; Formspec Core Specification v1.0; Formspec Mapping DSL v1.0; Respondent Ledger Add-On Specification v0.1

---

## Overview

Three architectural gaps exist at the WOS-Formspec boundary. Each gap is a seam where WOS references Formspec capabilities that are either unspecified (the handoff protocol), nonexistent (missing FEL functions), or incompatible (calling conventions). This document proposes concrete designs for all three.

| # | Gap | Severity | Summary |
|---|-----|----------|---------|
| 1 | Formspec Coprocessor | Critical | No specified protocol for presenting a Formspec form as a WOS task, collecting the Response, validating it, mapping its data into case state, or tracking the respondent. |
| 2 | FEL `every`/`some`/`duration` built-ins | High | WOS specs cite these as FEL built-in functions (Core S3.5), but they do not exist in the Formspec Core function catalog. |
| 3 | FEL array-of-record evaluation | High | WOS uses `every(list, 'field != value')` with a string predicate over records; FEL's `countWhere` uses a boolean expression with `$` referring to the current scalar element, not a record. |

## Gap 1: Formspec Coprocessor

### 1.1 Problem Statement

The WOS Kernel defines `createTask` actions (Kernel S9.2) and `contractRef` properties on actions (Kernel S11, schema `ContractReference`). The Governance spec defines task lifecycle states (Governance S10.1). The Runtime Companion defines the `ContractValidator` host interface (Runtime S12.3) and contract validation pseudocode (Runtime S5.5). But no specification answers five critical questions:

1. **Form presentation.** How does a `createTask` action with a Formspec `contractRef` cause a form to be rendered and presented to the assigned actor?
2. **Response ingestion.** How does a completed Formspec Response flow back into WOS case state?
3. **Data mapping.** How does Response data (which follows a Formspec Definition's item tree) map to case file fields (which follow the kernel's `caseFile.fields` schema)?
4. **Validation gating.** Is the Response validated against the Formspec Definition before the workflow event fires, and what happens when validation fails?
5. **Respondent tracking.** For rights-impacting workflows, how is the Respondent Ledger (Formspec Respondent Ledger Add-On v0.1) integrated into the WOS provenance chain?

Without answers to these questions, every WOS implementation must invent its own submission-to-case bridge. This defeats the standard's purpose: two "conformant" processors could implement the same Kernel Document and produce incompatible submission behavior.

### 1.2 Proposed Solution

Define a **Formspec Coprocessor Protocol** as a new section in the Runtime Companion (proposed S15). The protocol specifies the lifecycle of a Formspec-backed task from creation through Response ingestion.

#### 1.2.1 Task Creation with Formspec Contract

When a `createTask` action references a `contractRef` whose `binding` is `"formspec"`, the processor MUST:

1. Resolve the Formspec Definition from the `contractRef.ref` URI, pinned to the version active at instance creation time (Kernel S9.6, Formspec VP-01).
2. Create a task instance in state `created` (Governance S10.1).
3. Construct a **FormspecTaskContext** -- a structured object that the host uses to present the form.

**FormspecTaskContext:**

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `taskId` | string | REQUIRED | Unique task identifier, generated by the processor. |
| `contractRef` | string (URI) | REQUIRED | Resolved Kernel contract reference (may differ from Formspec Definition URL). |
| `definitionUrl` | string (URI) | REQUIRED | Canonical Formspec Definition **`url`** (Core Definition); MUST match `response.definitionUrl` on submit (VP-01). |
| `definitionVersion` | string | REQUIRED | Pinned Formspec Definition version; MUST match `response.definitionVersion` on submit. |
| `binding` | `"formspec"` | REQUIRED | Always `"formspec"` for this protocol. |
| `assignedActor` | string | REQUIRED | Actor identifier from the `assignTo` action property. |
| `prefillData` | object | OPTIONAL | Data from case state to pre-populate into the form. Keyed by Formspec item path. |
| `prefillMappingRef` | string (URI) | OPTIONAL | Reference to a Formspec Mapping Document that maps case state fields to Definition item paths. When present, the processor MUST use this mapping to generate `prefillData` rather than constructing it manually. |
| `deadline` | string (datetime) | OPTIONAL | ISO 8601 deadline derived from task SLA (Governance S10.3) or `taskTimeout` (Kernel S9.7). |
| `instanceId` | string (URI) | REQUIRED | The WOS instance this task belongs to. Used for routing the completed Response back. |
| `extensions` | object | OPTIONAL | Extension data. All keys MUST be prefixed with `x-`. |

The processor hands the `FormspecTaskContext` to the host via a new host interface operation (see S1.2.5). The host is responsible for rendering the form -- the processor never touches presentation.

#### 1.2.2 Response Completion and Ingestion

When an actor completes the form, the host receives a Formspec Response (Core S2.1.6) with `status: "completed"`. The host MUST submit the Response back to the processor via a new **submitTaskResponse** operation.

**submitTaskResponse input:**

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `taskId` | string | REQUIRED | The task identifier from `FormspecTaskContext.taskId`. |
| `response` | object (Formspec Response) | REQUIRED | The complete Formspec Response document. MUST have `status: "completed"`. |
| `actorId` | string | REQUIRED | The actor who completed the task. |
| `timestamp` | string (datetime) | REQUIRED | ISO 8601 timestamp of submission. |
| `idempotencyToken` | string | OPTIONAL | For exactly-once semantics (Runtime S4.3). |

The processor MUST reject a `submitTaskResponse` where the Response's `definitionUrl` and `definitionVersion` do not match the task's pinned **`definitionUrl`** and **`definitionVersion`** (not the Kernel `contractRef` string, which may use a different URI scheme). This prevents stale or mismatched submissions.

#### 1.2.3 Validation Gate

On receiving a `submitTaskResponse`, the processor MUST execute validation before advancing workflow state. The validation sequence:

1. **Formspec validation.** Validate the **full Response document** (envelope per `response.schema.json`, VP-01 tuple match, then Definition evaluation over **`data`**) via an extended host operation (concrete proposal: **`validateFormspecTaskResponse`** on Runtime S12.3 — see `thoughts/specs/2026-04-11-wos-s15-formspec-coprocessor-proposal.md`). The Formspec-conformant processor (Core S1.4) evaluates Binds and Shapes per Core §5.5. The WOS wrapper result aggregates envelope, pin, and definition validity (**`ValidationOutcome`** — not a single Formspec `ValidationResult`).

2. **Pipeline validation (if configured).** If a data validation pipeline (Governance S5) is attached to this contract via the `contractHook` seam, execute the pipeline stages. Pipeline assertion gates (Governance S5.4) run after Formspec validation, using the validated Response data as input.

3. **Gate decision.**
   - If Formspec validation fails (envelope, pin mismatch, or definition errors), the processor MUST NOT fire the task completion event. The processor MUST apply the rejection policy declared in the Governance Document (Governance S8). The task MUST transition **`claimed` → `failed`** (Governance task lifecycle) so the failure is visible and recoverable — not left **`claimed`** as if work were still in progress.
   - If Formspec validation passes but pipeline validation fails, the rejection policy from the failing pipeline stage applies (Governance S8.1).
   - If all validation passes, the processor advances to data mapping (S1.2.4).

4. **Provenance.** The processor MUST emit a `contractValidation` provenance record (Runtime S5.5) with the validation results, regardless of outcome.

This validation gate ensures that no unvalidated data enters case state. The same Formspec Definition validates human submissions and agent outputs -- the Formspec-as-validator principle (WOS integration rule 5) is structurally enforced.

#### 1.2.4 Response-to-Case-State Mapping

After validation passes, the Response data MUST be mapped to case file fields. This mapping is the bridge between Formspec's item-tree data model and WOS's flat-namespace case state.

**Mapping mechanism:** The mapping MUST use a Formspec Mapping Document (Mapping S3). The Kernel Document's contract declaration is extended with an optional `responseMappingRef`:

```json
{
  "contracts": {
    "intakeApplication": {
      "binding": "formspec",
      "ref": "urn:formspec:agency.gov:intake:2.0.0",
      "responseMappingRef": "urn:formspec:agency.gov:intake-to-case:1.0.0"
    }
  }
}
```

**ContractReference extension:**

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `responseMappingRef` | string (URI) | OPTIONAL | Reference to a Formspec Mapping Document that maps Response data to case file fields. When present, the processor MUST use this mapping. When absent, the processor uses a default identity mapping: Response `data` properties are written to `caseFile` fields with matching names. |

The Mapping Document declares the `direction` as `"forward"` (Response is source, case state is target). The `definitionRef` matches the Formspec Definition; the `targetSchema` describes the case file field structure.

**Mapping execution:**

1. The processor resolves the Mapping Document from `responseMappingRef`.
2. The processor executes the mapping's field rules in priority order (Mapping S3.2), transforming Response `data` into case file mutations.
3. Each mutation is applied via `setData` semantics (Kernel S5.4) -- recorded in mutation history with actor, timestamp, and lifecycle state.
4. The processor emits a `dataMapping` provenance record listing the mapping document used, the fields mapped, and any transform expressions evaluated.

**Default identity mapping:** When no `responseMappingRef` is declared, the processor MUST apply a default identity mapping: for each field in the Response's `data` object, if a case file field with the same name exists and the types are compatible, the value is written. Fields in the Response that have no matching case file field are ignored (not an error). Fields in the case file that have no matching Response field are not modified.

**Why Mapping DSL, not ad hoc:** The Mapping DSL already handles type coercion, value maps, conditional rules, flatten/nest transforms, and bidirectional execution. Using it for Response-to-case-state mapping means WOS inherits all of those capabilities without inventing new transformation semantics. It also means the mapping is a declarative document that can be authored, versioned, and validated independently.

#### 1.2.5 Host Interface Extension

A new host interface operation is added to the Runtime Companion S12:

**TaskPresenter** -- presents Formspec-backed tasks to actors and receives completed Responses.

| Operation | Input | Output | Required | Description |
|-----------|-------|--------|----------|-------------|
| `presentTask` | context: FormspecTaskContext | (none) | REQUIRED | Present a form to the assigned actor. The host renders the Formspec Definition, pre-populates data per `prefillData` or `prefillMappingRef`, and manages the actor's interaction. |
| `onTaskCompleted` | callback(taskId, response, actorId, timestamp) | (none) | REQUIRED | Callback invoked when the actor completes the form. The host calls `submitTaskResponse` on the processor. |
| `onTaskAbandoned` | callback(taskId, actorId, timestamp) | (none) | OPTIONAL | Callback invoked when the actor abandons the form. The host submits a Response with `status: "stopped"`. |

The TaskPresenter is a host interface because form rendering is a host concern, not an engine concern (ADR-0057 boundary test). Two conformant processors present the same form to the same actor -- but one might render it as a web form, another as a mobile app, another as a voice prompt. The rendering is irrelevant; the data contract is what matters.

#### 1.2.6 Event Firing

After successful validation and mapping, the processor:

1. Transitions the task to state `completed` (Governance S10.1).
2. Fires the completion event. The event name is declared on the `createTask` action via a new `completionEvent` property:

```json
{
  "type": "createTask",
  "taskRef": "intake-review",
  "assignTo": "eligibilityWorker",
  "contractRef": "intakeApplication",
  "completionEvent": "applicationSubmitted"
}
```

**Action extension:**

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `completionEvent` | string | OPTIONAL | Event name to fire when the task completes with a valid Response. When absent, the processor fires a synthetic `$task.completed` event with `{ taskId, taskRef }` payload. |
| `failureEvent` | string | OPTIONAL | Event name to fire when validation fails and the rejection policy is `failWithExplanation`. When absent, the processor fires `$task.failed`. |

The completion event carries the mapped case state changes in its `data` payload, making them available in the evaluation context as `event.data` for downstream guard evaluation.

1. Emits provenance: `taskCompleted` record linking the task, the Response, the validation results, the mapping applied, and the case state mutations.

#### 1.2.7 Respondent Ledger Integration

For workflows with `impactLevel` of `rights-impacting` or `safety-impacting`:

1. The processor MUST associate a Respondent Ledger (Respondent Ledger Add-On S2.1) with each Formspec-backed task. The ledger tracks the material history of the respondent's interaction with the form -- drafts, saves, validation attempts, and final submission.

2. The Respondent Ledger's `ledgerId` MUST be recorded in WOS provenance alongside the task completion record. This creates a cross-reference chain: WOS provenance record -> Formspec Response -> Respondent Ledger.

3. When the governance spec's due process policy (Governance S3.2) requires notice delivery confirmation, the Respondent Ledger's event history provides the evidentiary basis. The processor MUST verify that the ledger contains a `submit` event before accepting the Response as a valid task completion.

4. For non-rights-impacting workflows (`operational`, `informational`), Respondent Ledger integration is OPTIONAL.

**Cross-reference structure in provenance:**

```json
{
  "id": "prov-00123",
  "action": "taskCompleted",
  "taskId": "task-abc",
  "responseRef": {
    "responseId": "resp-456",
    "definitionUrl": "urn:formspec:agency.gov:intake:2.0.0",
    "definitionVersion": "2.0.0"
  },
  "respondentLedgerRef": {
    "ledgerId": "ledger-789",
    "href": "https://agency.gov/audit/respondent-ledgers/ledger-789"
  },
  "validationResult": { "valid": true, "errors": [] },
  "mappingRef": "urn:formspec:agency.gov:intake-to-case:1.0.0",
  "caseStateMutations": [
    { "path": "caseFile.applicantName", "value": "Ada Lovelace" },
    { "path": "caseFile.income", "value": 52000 }
  ]
}
```


### 1.3 Delivery Vehicle

New section **S15: Formspec Coprocessor Protocol** in the Runtime Companion (`wos-spec/specs/companions/runtime.md`). This is the right home because:

- The protocol defines runtime behavioral contracts (what the processor does), not document structure.
- It depends on host interfaces already defined in Runtime S12.
- It composes with governance enforcement already defined in Runtime S8.

Schema changes:

- `wos-kernel.schema.json`: Add `responseMappingRef` to `ContractReference`. Add `completionEvent` and `failureEvent` to `Action`.
- `wos-case-instance.schema.json`: Add `activeTasks` array with `ActiveTask` definition (tracking `FormspecTaskContext` for in-flight tasks).
- New `FormspecTaskContext` definition in either the kernel schema or a dedicated runtime schema.

### 1.4 Dependencies

- Formspec Core v1.0 (stable): Response, ValidationResult, processing model.
- Formspec Mapping DSL v1.0 (stable): Forward mapping execution.
- Formspec Respondent Ledger Add-On v0.1 (draft): Ledger event model.
- WOS Runtime Companion S12 (current): Host interfaces.
- WOS Governance S5, S8, S10 (current): Pipelines, rejection policies, task lifecycle.

### 1.5 Open Questions

1. **Prefill direction.** Should `prefillMappingRef` use the same Mapping Document in reverse direction (case-state-to-Response), or a separate mapping? Using the same document in `"both"` direction is elegant but means the mapping must be bidirectional, which not all transforms support.

2. **Partial save semantics.** Should the protocol support `submitTaskResponse` with `status: "in-progress"`? This would allow the processor to persist draft Response data in case state without firing the completion event. Useful for long-running forms, but adds complexity (case state contains unvalidated data).

3. **Multi-form tasks.** Can a single task reference multiple `contractRef` values (e.g., a task that requires completing both an application form and a supporting evidence form)? The current design assumes one task = one contract. Multi-form tasks could be modeled as multiple sequential tasks in the lifecycle.

4. **Agent submissions.** When an AI agent (Layer 2) submits a Response through the same coprocessor protocol, does the `FormspecTaskContext` need additional properties for agent context? The Formspec-as-validator principle says the same contract validates both -- but the `assignedActor` field needs to accept agent identifiers.

5. **Response amendment.** After a task completes, can the Response be amended (Formspec `status: "amended"`)? If so, does this create a new task or reopen the existing one? Amendment has implications for provenance immutability and Respondent Ledger continuity.

## Gap 2: FEL `every`/`some`/`duration` Built-ins

### 2.1 Problem Statement

The WOS Kernel Specification (S7.4) states:

> The same capabilities are achieved without grammar changes: `every` and `some` as built-in functions (Core S3.5), duration as a built-in function [...]

This asserts that `every`, `some`, and `duration` are FEL built-in functions defined in Formspec Core S3.5. They are not. The Formspec Core S3.5 built-in function catalog contains approximately 40+ functions across aggregates (`sum`, `count`, `countWhere`, `sumWhere`, `avgWhere`, `minWhere`, `maxWhere`, `avg`, `min`, `max`), string, numeric, date, logical, type-checking, money, MIP-state query, and repeat navigation categories. None of these are `every`, `some`, or `duration`.

WOS specs reference these functions in normative positions:

- **`every`**: AI Integration S4.8 obligation example: `every(output.extractedFields, 'sourceReference != null')`.
- **`every`/`some`**: Advanced Governance S8.2 restriction 2: "some and every expressions MUST quantify over finite, enumerable domains."
- **`duration`**: Kernel S7.4 (cited as built-in, never shown in an example).

The fallback noted in TODO.md is extension functions (Core S3.12). This is viable but changes the conformance profile: extension functions must be registered with the processor, declared in the Definition's `extensions` array, and are explicitly NOT part of the core FEL specification.

### 2.2 Proposed Solution

Two paths, evaluated in order of preference.

#### Path A: Add to Formspec Core (preferred)

Propose `every`, `some`, and `duration` as additions to Formspec Core S3.5 in the next specification revision. These are general-purpose functions with broad applicability beyond WOS.

**`every(array, predicate) -> boolean`:**
Returns `true` if `predicate` evaluates to `true` for every element in `array`. Returns `true` for an empty array (vacuous truth). Within the predicate, `$` refers to the current element. Null elements are skipped (consistent with existing aggregate null-skip behavior).

Signature: `every(array<T>, boolean) -> boolean`

```
every($lineItems[*].quantity, $ > 0)                          -- scalar array
every(output.extractedFields, $.sourceReference != null)       -- record array (see Gap 3)
```

**`some(array, predicate) -> boolean`:**
Returns `true` if `predicate` evaluates to `true` for at least one element in `array`. Returns `false` for an empty array. Within the predicate, `$` refers to the current element. Null elements are skipped.

Signature: `some(array<T>, boolean) -> boolean`

```
some($reviews[*].score, $ < 60)
some(caseFile.documents, $.type = 'medical-evidence')
```

**`duration(string) -> number`:**
Parses an ISO 8601 duration string and returns the total number of milliseconds. Returns `null` if the input is not a valid ISO 8601 duration. Calendar durations (`P1M`, `P1Y`) MUST return `null` because their millisecond equivalent depends on a reference date (a month is not a fixed number of milliseconds).

Signature: `duration(string) -> number`

```
duration('PT4H')         -- 14400000
duration('P30D')         -- 2592000000
duration('P1M')          -- null (calendar duration)
```

**Justification for Core over extension:** These functions are not domain-specific. `every`/`some` are universal quantifiers present in every programming language and every expression language of comparable power (XPath, FEEL, SQL). `duration` is the natural companion to FEL's existing date/time functions (`dateDiff`, `dateAdd`, `today`, `now`). Making them extension functions would mean every WOS deployment must register them separately -- an unnecessary friction for universally useful operations.

#### Path B: WOS Extension Function Registry (fallback)

If Formspec Core does not accept these additions, WOS MUST define them as a registered extension function set under the `wos` namespace:

```json
{
  "extensions": [
    {
      "namespace": "https://wos-spec.org/extensions/fel/1.0",
      "functions": [
        { "name": "every", "params": [{"name": "array", "type": "array"}, {"name": "predicate", "type": "boolean"}], "returns": "boolean" },
        { "name": "some", "params": [{"name": "array", "type": "array"}, {"name": "predicate", "type": "boolean"}], "returns": "boolean" },
        { "name": "duration", "params": [{"name": "value", "type": "string"}], "returns": "number" }
      ]
    }
  ]
}
```

Under this path:

- Every WOS Kernel Document that uses these functions MUST declare the WOS extension function set.
- Every WOS processor MUST register these extension functions before evaluating FEL expressions.
- The Kernel S7.4 text MUST be amended to reference "WOS extension functions (registered per Core S3.12)" instead of "built-in functions (Core S3.5)."

#### Path B consequences

1. **Formspec Definitions used as WOS contracts** cannot use `every`/`some` in their Binds unless the Formspec processor also registers the WOS extensions. This creates a coupling between the Formspec processor and the WOS processor that the integration rules (rule 1: additive only) seek to avoid.
2. **The verifiable subset** (Advanced Governance S8.2) must exclude extension functions (restriction 5 already does this). Under Path B, `every`/`some` cannot appear in verifiable constraints. This is a significant capability loss -- bounded quantification is the primary tool for expressing safety invariants over collections.

### 2.3 Delivery Vehicle

- **Path A:** Formspec Core specification revision. The functions would be added to Core S3.5 (built-in functions), in a new "Quantifier Functions" subsection alongside the existing "Aggregate Functions" subsection.
- **Path B:** New section in the WOS Kernel Specification (proposed S7.5: WOS Extension Functions) plus a standalone extension function registration document in `wos-spec/specs/extensions/`.

In either path, the WOS Kernel S7.4 rejected-alternative paragraph MUST be updated. Currently it says "the same capabilities are achieved without grammar changes: `every` and `some` as built-in functions (Core S3.5)." Under Path A, this becomes accurate. Under Path B, it must be rewritten to reference extension functions.

### 2.4 Dependencies

- **Path A:** Formspec Core spec revision cycle. The `spec-expert` agent should evaluate whether `every`/`some`/`duration` are consistent with FEL's design constraints (pure, total, deterministic).
- **Path B:** No external dependencies. Requires only WOS spec changes.

### 2.5 Open Questions

1. **`duration` return type.** Should `duration` return milliseconds (number), or a structured object `{ days, hours, minutes, seconds }`? Milliseconds is simpler for arithmetic (`duration('PT4H') > duration('PT2H')`) but loses precision for display. A structured return would require new comparison semantics. Recommendation: milliseconds, consistent with the "primitive return types" pattern of all existing FEL functions.

2. **Calendar duration handling.** The proposal has `duration('P1M')` return `null`. An alternative: `duration` takes an optional second argument (a reference date) and resolves calendar durations relative to it: `duration('P1M', caseFile.applicationDate)`. This is more powerful but adds complexity. For WOS's primary use case (timer durations, SLA comparisons), fixed durations suffice.

3. **`every`/`some` short-circuit.** Should `some` stop evaluating after finding the first `true` element? Should `every` stop after the first `false`? FEL built-in functions are pure and total, so short-circuiting is an optimization, not a semantic difference. Recommendation: permit but do not require short-circuiting (consistent with FEL's no-side-effects guarantee).

4. **Interaction with `countWhere`.** `every(arr, predicate)` is semantically equivalent to `countWhere(arr, predicate) = count(arr)`. `some(arr, predicate)` is equivalent to `countWhere(arr, predicate) > 0`. Should we add `every`/`some` when they are derivable? Yes -- readability and intent. `every(output.extractedFields, $.sourceReference != null)` is clearer than `countWhere(output.extractedFields, $.sourceReference != null) = count(output.extractedFields)`, especially in governance expressions read by non-technical reviewers.

## Gap 3: FEL Array-of-Record Evaluation

### 3.1 Problem Statement

WOS uses `every` with a string predicate over arrays of records:

```
every(output.extractedFields, 'sourceReference != null')
```

Here `output.extractedFields` is an array of objects (records), each with properties like `sourceReference`, `value`, `fieldName`. The predicate `'sourceReference != null'` refers to a property of the current record element.

FEL's existing aggregate functions (`countWhere`, `sumWhere`, etc.) use `$` as the context variable for the current element:

```
countWhere($lineItems[*].amount, $ > 10000)
```

Here `$lineItems[*].amount` produces a flat array of numbers, and `$` refers to the current number. This works for scalar arrays but is inadequate for record arrays because `$` would refer to the entire record object, and FEL has no syntax for `$.sourceReference` (accessing a property of `$`).

The calling conventions are incompatible:

| Convention | Array type | Predicate syntax | Context variable |
|------------|-----------|------------------|------------------|
| WOS (AI Integration S4.8) | array of records | String: `'sourceReference != null'` | Implicit: bare field names resolve against current record |
| FEL (Core S3.5.1) | array of scalars | Expression: `$ > 10000` | `$` = current element |

The WOS convention is borrowed from FEEL/DMN's `every item in list satisfies condition` pattern. The FEL convention is borrowed from XPath/XForms. They are fundamentally different.

### 3.2 Proposed Solution

Extend `$` to support property access on record elements. This requires no grammar change -- FEL already has the path expression `$.field` syntax for repeat-row context (Core S3.4). The extension is semantic, not syntactic: when `$` refers to a record (object) rather than a scalar, `$.propertyName` resolves to the value of that property on the current record.

**Revised `every`/`some` semantics:**

When the input array contains records (objects), `$` refers to the current record, and `$.propertyName` accesses properties of that record:

```
every(output.extractedFields, $.sourceReference != null)
some(caseFile.documents, $.type = 'medical-evidence' and $.verified = true)
```

When the input array contains scalars, `$` refers to the current scalar value (existing behavior for `countWhere` etc.):

```
every($lineItems[*].quantity, $ > 0)
some($scores[*], $ >= 60)
```

**No string predicates.** The WOS AI Integration S4.8 example uses a string predicate: `'sourceReference != null'`. This is incompatible with FEL's expression model. FEL predicates are expressions, not strings. The WOS example MUST be rewritten to use the `$` context variable:

Before (incompatible):

```
requirement: "every(output.extractedFields, 'sourceReference != null')"
```

After (compatible):

```
requirement: "every(output.extractedFields, $.sourceReference != null)"
```

**Why `$` over implicit field names:** Implicit field name resolution (bare `sourceReference` resolving against the current record) would require the FEL evaluator to implement dynamic scope -- looking up unqualified names against the record context before the evaluation context. This conflicts with FEL's flat-namespace evaluation model (Kernel S7.1) and creates ambiguity when a record property name collides with a context variable name (e.g., a record with a `caseFile` property). The explicit `$` prefix eliminates ambiguity at no readability cost.

**Impact on existing `countWhere`/`sumWhere`/etc.:** The `$`-as-record extension applies to all aggregate predicates uniformly. `countWhere(output.extractedFields, $.verified = true)` works identically to `every(output.extractedFields, $.verified = true)` in terms of `$` resolution. This is a backward-compatible extension: existing scalar-array usage is unchanged because `$` already refers to the current scalar element.

### 3.3 Delivery Vehicle

This change goes to **Formspec Core specification** because it modifies the semantics of `$` in predicate expressions -- a core FEL concern, not a WOS-specific concern. The change is:

1. **Core S3.4 (Path References):** Add a paragraph defining `$` behavior in aggregate/quantifier predicate context. When the current element is a scalar, `$` evaluates to the scalar. When the current element is a record, `$` evaluates to the record, and `$.propertyName` resolves to the named property.

2. **Core S3.5.1 (Aggregate Functions):** Update the `countWhere`, `sumWhere`, `avgWhere`, `minWhere`, `maxWhere` descriptions to note that `$` supports property access when the array contains records.

3. **New subsection for `every`/`some`:** If Gap 2 Path A is accepted, the `every`/`some` function descriptions naturally incorporate record support.

Additionally, the WOS AI Integration S4.8 example MUST be rewritten from string predicates to `$`-based expressions. This is a WOS spec change regardless of whether the Formspec Core change is accepted.

### 3.4 Dependencies

- Gap 2 resolution (the `every`/`some` functions must exist before their calling convention can be specified).
- Formspec Core spec revision (the `$` semantic extension is a Core change).
- FEL parser/evaluator implementation in both Rust (`fel-core`) and Python (`src/formspec/fel/`) to handle `$` as a record reference in predicate context.

### 3.5 Open Questions

1. **Nested records.** Should `$.nested.property` work for records containing nested objects? Recommendation: yes, consistent with FEL's existing dotted-path semantics. `$.address.zip` on a record with `{ "address": { "zip": "90210" } }` resolves to `"90210"`.

2. **Array-of-array.** What happens when `$` refers to an array element that is itself an array? Recommendation: `$` evaluates to the inner array, and array operations (`sum($)`, `count($)`) work on it. This is a degenerate case unlikely to arise in WOS but should be well-defined.

3. **Type inference.** Can the FEL evaluator statically determine whether `$` will refer to a scalar or record at parse time? In general, no -- the array's element type depends on runtime data. The evaluator MUST handle both cases at runtime. Static analysis tools (linters) MAY use schema information to warn about type mismatches.

4. **`$` collision with repeat context.** Inside a repeatable group, `$` already has a meaning (the current repeat row). If `every` is called inside a repeat's calculate expression, does `$` inside the predicate refer to the `every` element or the repeat row? Recommendation: the innermost `$` binding wins. `every` creates a new `$` scope that shadows the repeat row's `$`. This is consistent with nested scope rules in every programming language. The repeat row remains accessible via `parent()` (Core S3.5, repeat navigation functions).
