# WOS ProvenanceRecord Schema Extension ‚Äî Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add eight new optional fields to `ProvenanceRecord` (in `wos-spec/crates/wos-core/src/provenance.rs`) to close the gap between the internal provenance struct and the fields that the Semantic Profile ¬ß¬ß5.3, ¬ß5.4, ¬ß5.5, ¬ß6.3, ¬ß6.5 mandate in the PROV-O, XES, and OCEL mappings. **Spec anchor:** `wos-spec/specs/profiles/semantic.md` ¬ß5.3 (Facts tier PROV-O mapping), ¬ß5.4 (higher-tier bundles), ¬ß5.5 (actor type mapping), ¬ß6.3 (XES mapping), ¬ß6.5 (export scope). ## Task 1: Add 8 Fields to `ProvenanceRecord`

**Files:**
- Modify: `wos-spec/crates/wos-core/src/provenance.rs`

All fields are optional strings or `Vec<String>`. All carry `#[serde(default, skip_serializing_if = "Option::is_none")]` (or the Vec equivalent) so any existing fixture that omits them continues to deserialize without error.

- [x] **Step 1.1:** Add the following fields to the `ProvenanceRecord` struct, after the existing `data` field:


## Task 4: Populate `lifecycle_state`, `definition_version`, `inputs`, `outputs`, Digests

**Files:**
- Modify: `wos-spec/crates/wos-runtime/src/eval.rs` (state transition records, case mutation records)
- Modify: `wos-spec/crates/wos-runtime/src/runtime.rs` (integration handler call sites)

**Spec anchor:** ¬ß5.3 (`lifecycleState`, `definitionVersion`, `inputs`, `outputs`, `inputDigest`, `outputDigest`), ¬ß6.3 (XES mapping table).

These fields require access to runtime context beyond the record constructor's parameters:

| Field | Source |
|-------|--------|
| `lifecycle_state` | The current active lifecycle state at the time the record is emitted. For `StateTransition` records this is `from_state`; for `CaseStateMutation` records the lifecycle state is already in `data["lifecycleState"]` ‚Äî promote it to the top-level field. For other kinds, pass the current state from the evaluator context. |
| `definition_version` | The `version` field from the kernel definition (`WosKernel::version` or the document's top-level `version`). Available at runtime initialization; store it in the evaluator context and read it during stamp. |
| `inputs` / `outputs` | For `StateTransition`: `inputs = [event_id]`; `outputs = [to_state]`. For `CaseStateMutation`: `inputs = [path]`; `outputs = [new_value.to_string()]`. For other records, default to empty ‚Äî the spec does not mandate these for every kind, only for those that have identifiable entity relationships. |
| `input_digest` / `output_digest` | SHA-256 hex of the JSON-serialized `inputs` / `outputs` vecs. Computed at stamp time immediately after `inputs`/`outputs` are set. Use `sha2` crate (already a likely transitive dependency; add if absent). If `inputs` / `outputs` are empty, leave digests `None`. |

  ```rust
  fn digest_of(items: &[String]) -> Option<String> {
      if items.is_empty() {

      use sha2::{Digest, Sha256};
      let payload = serde_json::to_string(items).unwrap_or_default();
      Some(format!("{:x}", Sha256::digest(payload.as_bytes())))
  }
  ```

## Out of Scope / Deferred

The following are explicitly excluded from this plan. Each has its own architectural prerequisite:

- **PROV-O ¬ß5.4 `prov:Bundle` wrapping for higher-tier records.** The `audit_layer` field lands here and the tier filter works. But actually emitting Reasoning/Counterfactual/Narrative bundles requires the export API to accept an `audit_layer`-discriminated output path ‚Äî the current `export(log, config)` signature returns a single flat `ProvODocument`. Changing the API shape and deciding whether to produce separate bundle files vs. a multi-bundle JSON-LD document is a separate follow-up.

- **OCEL ¬ß6.4 Case File Items as Objects.** The spec's full OCEL mapping models each case file item as a first-class OCEL object with E2O links per mutation event. Implementing this requires the exporter to receive the case file schema alongside the `ProvenanceLog` so it can derive `objectTypes` from item schemas and `objects` from case state snapshots. Architectural: requires a new `ExportConfig` field and a case-state snapshot protocol. Tracked separately.

- **SHACL validation of PROV-O output.** Validating the emitted JSON-LD against SHACL shapes (SP ¬ß4) requires an RDF library dependency (`oxigraph` or similar). Out of scope; the shapes library is specified in Appendix A and can be applied externally.

- **`"reasoning"` and `"counterfactual"` audit layers.** The `audit_layer_for_kind` helper returns `"facts"` for all current variants except `NarrativeTierRecorded`. Routing Reasoning and Counterfactual records requires new `ProvenanceKind` variants or a Layer 1 injection mechanism not yet implemented.
